# Plan: `chat-inbox-scope-force-archive`

## Objective

Ship a scoped OpenSpec change and implementation plan that keeps inbox behavior chat-native, removes archive bucket UX, and adds per-user manual force archive/unarchive for reservation and support threads without deleting chat history.

## Confirmed Constraints

- Keep notification delivery concerns in NotificationBell/web push; chat unread and inbox state stay in chat surfaces.
- Remove inline archive bucket/collapsible from reservation and support inbox lists.
- Archived threads are hidden from inbox by default.
- Force archive is per-user and applies to reservation + support threads.
- Unarchive is manual only (no auto-unarchive on new messages in this phase).
- Build gate for this migration remains `pnpm lint` + manual smoke matrix (no `pnpm build` unless requested).

## Recommended Implementation Shape

1. Add a dedicated `chatInbox` tRPC router for archive overlay state.
2. Persist per-user archive records in Postgres (`chat_inbox_archive`) keyed by user + thread scope + thread id.
3. Treat `threadId` as provider channel id (`res-*`, `cr-*`, `vr-*`) to align directly with Stream list filtering.
4. Keep existing reservation/support chat transport routers for session/send; use `chatInbox` only for inbox visibility overlay.
5. Move archived recovery into a separate management view/action (manual unarchive), not inline archive bucket.
6. Decouple manual refresh spinner state from passive/background sync state.

### Phase A - Data Model + Backend Contracts

Add archive persistence and APIs.

Files to add/update:
- `src/lib/shared/infra/db/schema/chat-inbox-archive.ts` (new)

- `src/lib/modules/chat/chat-inbox.router.ts` (new)
- `src/lib/shared/infra/trpc/root.ts` (register `chatInbox`)

Contract details:
- Table: `chat_inbox_archive`
  - `id uuid pk default gen_random_uuid()`
  - `user_id uuid not null` FK `auth.users(id)`
  - `thread_kind text not null` check in `('reservation','support')`
  - `thread_id text not null` (provider channel id)
  - `archived_at timestamptz not null default now()`
  - `created_at timestamptz not null default now()`
  - `updated_at timestamptz not null default now()`
  - unique index `(user_id, thread_kind, thread_id)`
  - helper index `(user_id, thread_kind)`
- New procedures:
  - `chatInbox.archiveThread({ threadKind, threadId })` (idempotent upsert)
  - `chatInbox.unarchiveThread({ threadKind, threadId })` (idempotent delete)
  - `chatInbox.listArchivedThreadIds({ threadKind })`

Authorization model:
- Reservation: caller must be participant for `res-*` reservation thread.
- Support: caller must be authorized owner/admin participant for `cr-*`/`vr-*` thread.
- Validate thread prefix/scope mismatch as bad input.

### Phase B - Reservation Metadata Filtering

Extend reservation thread metas to support inbox-only filtering with archive overlay.

Files to update:
- `src/lib/modules/chat/reservation-chat.router.ts`
- `src/lib/modules/chat/services/reservation-chat.service.ts`

Behavior:
- Extend `getThreadMetas` input with `includeArchived?: boolean` (default `false`).
- Default response excludes rows that are archived by current user.
- Default response also excludes inactive/past rows per agreed rules:
  - keep `CREATED`, `AWAITING_PAYMENT`, `PAYMENT_MARKED_BY_USER`
  - keep `CONFIRMED` only when `end_time >= now`
  - exclude `CANCELLED`, `EXPIRED`, past `CONFIRMED`
- Allow controlled compatibility via `includeArchived: true` where explicitly needed.

## Validation Plan

Automated:
- `pnpm lint`

Manual smoke matrix:
- Owner reservation inbox: no archive bucket; archive/unarchive works; per-user isolation holds.
- Player reservation inbox: same behavior and per-user isolation.
- Admin support inbox: no archive bucket; archive/unarchive works.
- Support thread sheets from detail pages remain accessible (inbox-only hiding).
- New messages do not auto-unarchive archived threads.
- Refresh buttons stop spinning after manual operation settles and are not locked by background sync.
- NotificationBell still reflects delivery channel settings/diagnostics, not inbox state.

## Risks and Mitigations

- Scope/prefix mismatch for thread ids -> strict server validation by `threadKind` + prefix.
- Unauthorized archive toggles -> re-check membership/role before any read/write.
- UX dead-end for unarchive after removing bucket -> provide explicit archived management entry point.
- Spinner regressions due mixed states -> isolate `isManualRefreshing` from passive sync flags.
