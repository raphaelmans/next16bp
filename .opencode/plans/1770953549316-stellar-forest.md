# Plan: Unify All Chat Interfaces Into One Component

## What I confirmed first
- `src/app/(public)/owners/get-started/page.tsx` does not render the Claim Support chat widget; it only renders `OwnersGetStartedClient`.
- The actual Claim Support widget is rendered in authenticated owner onboarding at `src/app/(owner-onboarding)/owner/get-started/page.tsx` via `SupportChatSheet`.
- Chat UI is currently split across three implementations:
  - `src/features/chat/components/chat-widget/reservation-inbox-widget.tsx`
  - `src/features/support-chat/components/support-inbox-widget.tsx`
  - `src/features/support-chat/components/support-chat-sheet.tsx`
- All three already share `src/features/chat/components/chat-thread/stream-chat-thread.tsx`, but list/sync/surface logic is duplicated and drifting.

## Final goal (per your requirement)
Create one single canonical chat UI component used everywhere (player, owner, admin, claim support, verification support), with behavior controlled by typed props/adapters instead of separate UI implementations.

## Recommended implementation
1. **Create one canonical component**
   - Add `src/features/chat/components/unified-chat/unified-chat-interface.tsx`.
   - This is the only chat UI component that renders either:
     - floating inbox surface (global widgets), or
     - right-side single-thread sheet surface (contextual support chat).
   - Use a strict discriminated prop model, e.g.:
     - `surface`: `"floating" | "sheet"`
     - `domain`: `"reservation" | "support"`
     - `mode`: `"inbox" | "singleThread"`
     - role/config labels/storage keys/trigger options.

2. **Move domain-specific behavior behind adapters (inside chat feature)**
   - Add `src/features/chat/components/unified-chat/adapters/reservation-adapter.ts`.
   - Add `src/features/chat/components/unified-chat/adapters/support-adapter.ts`.
   - Each adapter owns domain-specific fetch/sync/send behavior (reservation metas vs support sessions, mutation endpoints, archive rules), while UI rendering remains in `UnifiedChatInterface`.

3. **Refactor existing chat components into thin wrappers (compatibility step)**
   - Convert these files to simple delegates to the canonical component:
     - `src/features/chat/components/chat-widget/reservation-inbox-widget.tsx`
     - `src/features/support-chat/components/support-inbox-widget.tsx`
     - `src/features/support-chat/components/support-chat-sheet.tsx`
   - Keep public props stable initially so route files do not all break at once.

4. **Migrate all call sites to canonical component import**
   - Update mounts/usages to import and use `UnifiedChatInterface` directly:
     - `src/components/layout/player-shell.tsx`
     - `src/app/(owner)/layout.tsx`
     - `src/app/(admin)/layout.tsx`
     - `src/app/(owner-onboarding)/owner/get-started/page.tsx`
     - `src/features/owner/components/place-verification-panel.tsx`
     - `src/app/(admin)/admin/claims/[id]/page.tsx`
     - `src/app/(admin)/admin/verification/[requestId]/page.tsx`
   - Keep behavior parity (trigger labels, role labels, storage keys, unread badge, reservation open event).

5. **Remove duplicated UI logic after migration**
   - Delete now-redundant chat UI branches from old files.
   - Keep only minimal alias exports (if needed) for short-term compatibility, then remove.

## Critical behavior to preserve

## Follow-up plan: Fix long-message stretching in unified chat (mobile + desktop)

### Goal
Prevent long messages from stretching chat sheets/dialog-like surfaces. Keep fixed header + fixed composer/footer, and make only the middle message area scrollable across all unified chat surfaces.

### Root cause summary
- The reported public file `src/app/(public)/owners/get-started/page.tsx` does not render chat.
- The issue is in shared chat containers and message rendering used by authenticated support/reservation surfaces.
- Missing `min-h-0`/`min-w-0`/`overflow-hidden` constraints in some wrappers plus weak long-token wrapping in message markdown allow layout blowout.

### Files to modify
1. `src/features/chat/components/unified-chat/unified-chat-interface.tsx`
   - **Support sheet mode (`UnifiedSupportThreadSheet`)**
   - Make `SheetContent` a constrained flex column with `overflow-hidden`.
   - Wrap thread area in `min-h-0 flex-1 overflow-hidden`.
   - Pass `minHeightClassName="min-h-0 flex-1"` to `StreamChatThread`.
   - Add `min-w-0 overflow-hidden` to support inbox `threadPane` container.

2. `src/features/chat/components/inbox-shell/inbox-floating-sheet.tsx`
   - Add `min-h-0 overflow-hidden` to `SheetContent`.
   - Add `min-w-0 overflow-hidden` to the split-pane content wrapper (`listPane` + `threadPane`).

3. `src/features/chat/components/chat-widget/reservation-inbox-widget.tsx`
   - Add `min-w-0 overflow-hidden` to reservation `threadPane` container.

4. `src/features/chat/components/chat-thread/stream-chat-thread.tsx`
   - Add `min-w-0 overflow-hidden` on the root thread container.
   - Add `min-w-0`/overflow guards on nested message-area wrappers.
   - Ensure header text containers remain shrinkable (`min-w-0`) so long titles/subtitles do not force width growth.

5. `src/components/ai-elements/message.tsx`
   - Add robust long-text wrapping to message markdown rendering:
     - `break-words`
     - `[overflow-wrap:anywhere]`
     - keep code blocks horizontally scrollable (e.g. `pre` overflow-x auto).

6. `src/components/ai-elements/conversation.tsx`
   - Add `min-h-0 min-w-0`/overflow guards to conversation wrappers so middle-pane scroll behavior remains stable under long content.

### Execution sequence (minimal-risk)
1. Container constraints first (`unified-chat-interface.tsx`, `inbox-floating-sheet.tsx`, `reservation-inbox-widget.tsx`).
2. Thread wrapper hardening (`stream-chat-thread.tsx`).
3. Message wrapping safeguards (`message.tsx`, `conversation.tsx`).
4. Run lint and then manual QA on both mobile and desktop.
