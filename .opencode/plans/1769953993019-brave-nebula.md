## "Types" Of Notifications In Place (MVP)

If you mean *what kinds of notifications this system currently supports*, the MVP contract is intentionally small:

- Event type: `place_verification.requested` (owner submits a venue verification request)
- Recipients: admins (`user_roles.role = "admin"` joined to `profile`)
- Channels: `EMAIL` (Resend) and `SMS` (Semaphore) when contact details exist
- Delivery system type: async outbox (`notification_delivery_job`) + cron dispatcher (`/api/cron/dispatch-notification-delivery`)
- Job states: `PENDING` -> `SENDING` -> `SENT`, with `FAILED` retries and `SKIPPED` for invalid/unsupported

Why you don't see "all" types in `docs/notification-system/notification-state-machine-ascii.md`:
- That file is a quick reference showing only (1) the MVP delivery pipeline and (2) the outbox job state machine.
- The enumerations (event/recipients/idempotency/payload) live in `docs/notification-system/notification-state-machine-level-2-engineering.md`.

# Plan: Notify Court Owner On Reservation + Verification

User request:
- When a user creates a reservation, notify the court owner.
- When an admin verifies the court owner (approval flows), notify the court owner.

Clarification:
- "admin verifies the court owner" = BOTH flows: claim request approval + place verification approval.

## Current State (not supported)

- Reservation creation (`src/lib/modules/reservation/services/reservation.service.ts`) creates DB rows + logs `reservation.created`, but does not enqueue any `notification_delivery_job`.
- Admin review flows log business events (`place_verification.approved`, `claim_request.approved`, etc.) but do not enqueue any outbox jobs.
- Cron dispatcher (`src/app/api/cron/dispatch-notification-delivery/route.ts`) only delivers `place_verification.requested` and will `SKIP` everything else as `UNSUPPORTED_EVENT_TYPE`.

## Proposed Approach (extend existing outbox)

Add new notification event types (string `eventType`, no enum/migration required):
- `reservation.created` -> notify organization owner/contact.
- `place_verification.approved` (and `place_verification.rejected`) -> notify organization owner/contact.
- `claim_request.approved` (and `claim_request.rejected`) -> notify organization owner/contact.

Recipient model (single owner):
- Organization owner is `organization.ownerUserId`.
- Prefer org business contact: `organization_profile.contactEmail/contactPhone`.
- Fallback to owner user contact: `profile.email/phoneNumber` for `organization.ownerUserId`.

## Implementation Steps

1) Recipient resolution
- Update `src/lib/modules/notification-delivery/repositories/notification-recipient.repository.ts`
  - Add `OwnerRecipient` type.
  - Add methods to resolve owner contact:
    - `findOwnerRecipientByOrganizationId(organizationId, ctx?)`
    - `findOwnerRecipientByReservationId(reservationId, ctx?)` (join: reservation -> court -> place -> organization -> org_profile/profile)
    - `findOwnerRecipientByPlaceVerificationRequestId(requestId, ctx?)` (join: place_verification_request -> organization -> org_profile/profile)
    - (Optional) `findOwnerRecipientByClaimRequestId(requestId, ctx?)` (join claim_request -> organization -> org_profile/profile)

2) Enqueue APIs
- Update `src/lib/modules/notification-delivery/services/notification-delivery.service.ts`
  - Add enqueue methods:
    - `enqueueOwnerReservationCreated(...)`
    - `enqueueOwnerPlaceVerificationReviewed(...)` (approved/rejected)
    - (Optional) `enqueueOwnerClaimReviewed(...)` (approved/rejected)
  - Create EMAIL/SMS jobs only when `email`/`phoneNumber` exist.
  - Use stable idempotency keys, e.g.:
    - `reservation.created:<reservationId>:org:<organizationId>:email|sms`
    - `place_verification.approved:<requestId>:org:<organizationId>:email|sms`
    - `claim_request.approved:<requestId>:org:<organizationId>:email|sms`

3) Hook producers (same transaction)
- Reservation created:
  - Inject notification delivery service into reservation factory and call enqueue inside the transaction after reservation insert.
  - Files:
    - `src/lib/modules/reservation/services/reservation.service.ts`
    - `src/lib/modules/reservation/factories/*.ts` (exact factory file to confirm during build)

- Place verification admin approval/rejection:
  - Inject notification delivery service into:
    - `src/lib/modules/place-verification/services/place-verification-admin.service.ts`
    - `src/lib/modules/place-verification/factories/place-verification.factory.ts`
  - Enqueue after updating request + place_verification/place (still within the tx).

- Claim request approval/rejection (if in scope):
  - Inject notification delivery service into:
    - `src/lib/modules/claim-request/use-cases/approve-claim-request.use-case.ts`
    - `src/lib/modules/claim-request/services/claim-admin.service.ts`
    - and the relevant factories for those classes.

4) Dispatcher support + message templates
- Update `src/app/api/cron/dispatch-notification-delivery/route.ts`
  - Add handlers for new `eventType`s (switch/if chain):
    - Validate job payload with Zod (per event type).
    - Build `subject`, `emailText`, and `smsText`.
    - Add deep links using `appRoutes`:
      - owner reservation detail: `appRoutes.owner.reservationDetail(reservationId)`
      - owner verification: `appRoutes.owner.verification.place(placeId)`
      - (optional) owner venues page: `appRoutes.owner.places.base`

5) Documentation updates
- Update `docs/notification-system/README.md` and `docs/notification-system/notification-state-machine-level-2-engineering.md` to list the new supported event types and recipient rules.
- Update `docs/notification-system/notification-state-machine-ascii.md` contract section to include the new events.

# Plan: Per-Channel Notification Toggles (Email/SMS)

User request:
- Add an existing-like way to toggle/disable notification sending by channel (disable SMS or email).

Current state:
- No explicit toggle system exists; only implicit behavior (no email/phone -> no jobs).
- Cron dispatcher always attempts to send claimed jobs.

Decision:
- When a channel is disabled, existing queued jobs for that channel should be skipped permanently (`SKIPPED` with `DISABLED_CHANNEL:<channel>`).

## Environment variables

Add (server-side):
- `NOTIFICATION_EMAIL_ENABLED` (default: true)
- `NOTIFICATION_SMS_ENABLED` (default: true)

Update `.env.example` to include:
- `SEMAPHORE_API_KEY`, `SEMAPHORE_SENDER_NAME`, `SEMAPHORE_BASE_URL`
- `CRON_SECRET`
- `NOTIFICATION_EMAIL_ENABLED`, `NOTIFICATION_SMS_ENABLED`

## Implementation

1) Typed env
- Update `src/lib/env/index.ts`:
  - Add `NOTIFICATION_EMAIL_ENABLED` and `NOTIFICATION_SMS_ENABLED` as optional booleans.
  - Make `SEMAPHORE_API_KEY` optional so SMS can be disabled without providing credentials.

2) Enqueue gating
- Update `src/lib/modules/notification-delivery/services/notification-delivery.service.ts`:
  - Before pushing an EMAIL job, check `env.NOTIFICATION_EMAIL_ENABLED !== false`.
  - Before pushing an SMS job, check `env.NOTIFICATION_SMS_ENABLED !== false`.

3) Dispatch gating
- Update `src/app/api/cron/dispatch-notification-delivery/route.ts`:
  - If `job.channel === "EMAIL"` and email disabled -> mark `SKIPPED`, `lastError: "DISABLED_CHANNEL:EMAIL"`.
  - If `job.channel === "SMS"` and SMS disabled -> mark `SKIPPED`, `lastError: "DISABLED_CHANNEL:SMS"`.
  - Instantiate `emailService` / `smsService` lazily only when needed and enabled.

4) Provider runtime validation
- Update `src/lib/shared/infra/sms/semaphore-sms.service.ts` to throw a clear error when `SEMAPHORE_API_KEY` is missing (only relevant if SMS enabled and send is attempted).

5) Documentation
- Update `docs/notification-system/notification-state-machine-level-3-ops.md`:
  - Document `NOTIFICATION_EMAIL_ENABLED` and `NOTIFICATION_SMS_ENABLED` behavior.
