# Fix owner setup "Courts configured" false-complete state

## Problem and root cause

- `src/app/(auth)/owner/get-started/page.tsx` marks the courts step as complete when `hasActiveCourt` is true.
- `hasActiveCourt` comes from `ownerSetup.getStatus` and is currently computed from active court existence only.
- The setup flow expects courts to be configured only after schedule hours + pricing rules are present, so the current check is too loose.

## Recommended implementation

### 1) Expand owner setup readiness on the server

Update `ownerSetup` status to distinguish:

- active-court existence (`hasActiveCourt`) and
- booking-ready court configuration (`hasReadyCourt` = active court with both hours and pricing)

Files:

- `src/lib/modules/owner-setup/dtos/owner-setup-status.dto.ts`
- `src/lib/modules/owner-setup/use-cases/get-owner-setup-status.use-case.ts`

Changes:

- Add fields to `OwnerSetupStatus`:
  - `hasReadyCourt: boolean`
  - `readyCourtId: string | null`
  - `hasCourtSchedule: boolean`
  - `hasCourtPricing: boolean`
- In `GetOwnerSetupStatusUseCase`:
  - Load active courts for `primaryPlace`.
  - Fetch hours windows for active courts via `findByCourtIds`.
  - Fetch pricing rules for active courts via `findByCourtIds`.
  - Build readiness by court id (same-court requirement: one active court must have both hours and pricing).
  - Compute:
    - `hasActiveCourt`
    - `hasCourtSchedule`
    - `hasCourtPricing`
    - `hasReadyCourt`
    - `readyCourtId`
- Update completion logic:
  - `isSetupComplete = hasOrganization && hasVenue && isVerificationConfirmed && hasReadyCourt`
- Update next step logic:
  - use `!hasReadyCourt` for `"configure_courts"` gate.

### 2) Wire owner-setup use case dependencies

Files:

- `src/lib/modules/owner-setup/factories/owner-setup.factory.ts`

Changes:

- Inject repositories needed for readiness checks:
  - `makeCourtHoursRepository()`
  - `makeCourtRateRuleRepository()`
- Update `GetOwnerSetupStatusUseCase` constructor parameters accordingly.

### 3) Fix setup hub UI completion criteria

File:

- `src/app/(auth)/owner/get-started/page.tsx`

Changes:

- Read new fields from `setupStatus`:
  - `hasReadyCourt`
  - `readyCourtId`
  - `hasCourtSchedule`
  - `hasCourtPricing`
- Step badge "Go Live" should use `hasReadyCourt` (not `hasActiveCourt`).
- `ConfigureVenueCourtsCard` complete state should use `hasReadyCourt`.
- Keep a partial state when `hasActiveCourt && !hasReadyCourt`:
  - show action-needed copy indicating missing schedule and/or pricing.
  - keep CTA to court setup/schedule.
- Use `readyCourtId` (fallback `primaryCourtId`) for "Manage availability" deep-link.

### 4) Keep downstream behavior aligned

Files to verify (likely no code changes needed):

- `src/app/(owner)/owner/page.tsx`
- `src/features/owner/components/owner-sidebar.tsx`

Because both rely on `isSetupComplete`, they should automatically stop hiding setup CTA too early once server logic is corrected.
