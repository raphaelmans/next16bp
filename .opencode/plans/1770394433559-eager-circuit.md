# Plan: Reservation details + inbox recency + chat auto-links

## Goal

- Show reservation created date in booking details (currently missing).
- Sort reservation inbox by latest updated activity (hybrid rule confirmed).
- Fix owner-confirmed auto-message so users see clickable links (not raw pathnames).

## Confirmed Product Decision

- Inbox sorting uses **Hybrid (Recommended)**:
  - Primary signal: latest chat activity timestamp.
  - Fallback/secondary signal: reservation `updatedAt` timestamp.

## Context7 Notes (ai-elements)

- Source: `/vercel/ai-elements`.
- `MessageResponse` renders markdown and supports clickable links when content contains markdown link syntax (for example, `[View reservation](https://...)`).
- Optional link hardening exists via `allowedLinkPrefixes`, but the minimal fix is to emit proper markdown links in seeded message text.

## Implementation Plan

### 1) Add reservation created date to booking details card

1. Extend booking card input shape to accept reservation creation timestamp.
2. Pass `reservation.createdAt` from reservation detail page into card props.
3. Render a compact `Booked on` field in the booking info section using existing format helpers.

Files:
- `src/app/(auth)/reservations/[id]/page.tsx`
- `src/features/reservation/components/booking-details-card.tsx`

### 2) Reorder inbox by latest updated activity (hybrid)

1. Add `updatedAt` to reservation thread meta payload returned by chat service.
2. Extend client-side `ReservationThreadMeta` type to include `updatedAtIso`.
3. In inbox widget sorting:
   - Derive channel activity timestamp from Stream channel state (latest message/newest channel activity).
   - Derive reservation activity timestamp from `meta.updatedAtIso`.
   - Compute effective activity timestamp as the newest available value.
4. Keep active vs archived grouping behavior unchanged, but sort each group by effective activity descending.
5. Apply deterministic tie-breakers: unread count desc, then slot start time desc, then reservation id asc.
6. Remove premature alphabetical normalization in reservation id extraction to avoid fighting recency ordering.

Files:
- `src/lib/modules/chat/services/reservation-chat.service.ts`
- `src/features/chat/components/chat-widget/reservation-inbox-widget.tsx`

### 3) Fix owner confirmation seeded message links

1. Update seeded message content builder to output markdown links with user-friendly labels.
2. Build links from trusted internal routes (`appRoutes`) and validated app origin (`NEXT_PUBLIC_APP_URL`) using safe URL composition.
3. Keep fallback behavior safe for local/dev if app origin is absent.
4. Keep provider send path unchanged (text message body remains markdown-capable).

Files:
- `src/lib/modules/chat/ops/post-owner-confirmed-message.ts`

### 4) Optional renderer adjustment (only if needed after manual verification)

1. If markdown links still do not become clickable in runtime, pass explicit `allowedLinkPrefixes` to `MessageResponse` in chat thread rendering.
2. Keep scope narrow and avoid broad/wildcard prefixes.

Potential file (conditional):
- `src/features/chat/components/chat-thread/stream-chat-thread.tsx`

## Verification Plan

1. Run static validation:
   - `pnpm lint`
2. Manual E2E checks in dev (`pnpm dev`):
   - Reservation detail page displays `Booked on` with reservation created timestamp.
   - Inbox list order updates correctly when:
     - a new message arrives,
     - reservation status/state updates without a new message.
   - Active and archived sections remain correct.
   - Owner-confirmed auto-message renders clickable links (not plain path text).
   - Links navigate to expected destinations:
     - player reservation detail
     - owner reservations list
   - Link behavior is acceptable with and without `NEXT_PUBLIC_APP_URL` in local/dev.

## Risks / Edge Cases

- Timestamp availability differences from Stream channel state objects.
- Null/undefined `updatedAt` on older records.
- Sorting stability when multiple threads share equal activity timestamps.
- Markdown link rendering differences if runtime markdown policy is stricter than expected.
