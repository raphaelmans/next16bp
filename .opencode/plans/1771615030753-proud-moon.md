# Plan - schedule-pricing-addons-frontend-integration

## Confirmed Scope
- Create a new OpenSpec change named `schedule-pricing-addons-frontend-integration`.
- Implement both owner + player frontend flows.
- UX direction is parity + polished UX (no broad redesign).
- Apply `frontend-design`, `ui-ux-pro-max`, and `copywriting` principles during implementation.

## Recommended Approach

### Phase 1 - Start the new OpenSpec change and scaffold artifacts
1. Run `openspec new change "schedule-pricing-addons-frontend-integration"` (default schema).
2. Create artifacts in order:
   - `proposal.md`: frontend gap bridging for owner/player integration.
   - `specs/*/spec.md`: separate capability specs for owner management, player selection, and warning/breakdown display.
   - `design.md`: data flow and UI architecture (components -> hooks -> featureApi -> tRPC).
   - `tasks.md`: execution waves below.

### Phase 2 - Shared frontend add-on integration layer
1. Extend owner transport wrapper for add-on endpoints:
   - `src/features/owner/api.ts`
2. Add frontend add-on feature module (new):
   - `src/features/court-addons/helpers.ts`
   - `src/features/court-addons/schemas.ts`
   - `src/features/court-addons/hooks.ts`
   - `src/features/court-addons/components/*`
3. Keep all transforms pure in helpers and avoid route-local component sprawl in `src/app/**`.

### Phase 3 - Owner UI integration (court setup)
1. Integrate add-on management panel into existing setup flow:
   - `src/features/owner/components/court-schedule-editor.tsx`
   - `src/features/owner/components/court-schedule-editor/hooks.ts`
   - `src/features/owner/pages/owner-place-court-setup-page.tsx`
2. Add forms for:
   - mode (`OPTIONAL`/`AUTO`)
   - pricing type (`HOURLY`/`FLAT`)
   - rule windows and overlap-safe edits
   - flat fee fields and currency-aligned helper text
3. Apply copywriting style: concise, actionable, non-alarmist validation/warning copy.

### Phase 4 - Player booking integration
1. Add add-on selection state and URL threading:
   - `src/features/discovery/place-detail/state/place-detail-url-state.ts`
   - `src/features/discovery/hooks/place-detail.ts`
   - `src/features/discovery/hooks/search.ts`
2. Add add-on selector and summary in place detail booking surfaces:
   - `src/features/discovery/place-detail/components/sections/place-detail-booking-desktop-section.tsx`
   - `src/features/discovery/place-detail/components/sections/place-detail-booking-mobile-section.tsx`
   - `src/features/discovery/place-detail/components/place-detail-booking-summary-card.tsx`
3. Pass `selectedAddonIds` through reservation creation path:
   - `src/features/reservation/pages/place-booking-page.tsx`

### Phase 5 - Pricing warnings and breakdown UX
1. Surface backend pricing warnings in checkout/review.
2. Show transparent totals (base + add-ons + total) in reservation summary UI:
   - `src/features/reservation/components/order-summary.tsx` (or additive companion component)
3. Ensure mobile responsiveness and accessibility (focus states, readable warning cards, non-blocking alerts).

### Phase 6 - Tests and validation
1. Add unit tests for new helpers/schemas/hooks under mirrored `src/__tests__/` paths.
2. Validate no regressions in current add-on pricing unit/contract coverage.
3. Run quality checks and manual smoke matrix.

- Treat URL add-on IDs as untrusted and revalidate against fetched add-ons.
- If full-repo lint has unrelated pre-existing failures, document and isolate scoped-file validation for this change.
