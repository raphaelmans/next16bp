# Security And Cost-Abuse Audit Plan (Chat-Focused)

## Objective
- Prevent abuse patterns that can spike provider costs (Stream Chat, Supabase Auth email, OpenAI, notification providers), with first priority on chat spam/session churn.

## High-Risk Findings To Address First
- Anonymous tRPC rate limiting is effectively bypassable because limiter identity falls back to `requestId` (`src/lib/shared/infra/trpc/middleware/ratelimit.middleware.ts`, `src/lib/shared/infra/trpc/context.ts`).
- Global limiter currently fails open when Upstash env vars are missing, so throttling can silently become inactive (`src/lib/shared/infra/ratelimit/ratelimit.ts`).
- Public auth endpoints that can trigger provider email are mostly unthrottled (`src/lib/modules/auth/auth.router.ts`, `src/lib/modules/auth/repositories/auth.repository.ts`).
- Chat auth/session endpoints are protected but unthrottled, allowing token/channel churn (`src/lib/modules/chat/chat.router.ts`, `src/lib/modules/chat/reservation-chat.router.ts`, `src/lib/modules/chat/support-chat.router.ts`).
- Chat file uploads happen directly from client via Stream `sendFile` before server mutation checks, increasing spam/billing exposure (`src/features/chat/components/chat-thread/stream-chat-thread.tsx`, `src/features/chat/hooks/useStreamChannel.ts`).

## Recommended Implementation

### P0 - Immediate hardening (first PR)
1. Fix limiter identity and remove request-id bypass
   - Extend tRPC context with a server-derived client identifier (trusted forwarded IP chain + fallback), and do not use caller `x-request-id` for quota identity.
   - Use `userId` for authenticated users; use derived client identifier for anonymous users.
2. Fail closed for rate limiting in production
   - Change limiter bootstrap so missing Upstash config in production does not silently allow all requests.
   - Return explicit error for rate-limited routes if limiter backend is unavailable; allow explicit dev-only bypass.
3. Add throttling to cost-sensitive auth/chat entry points
   - Wrap `auth.loginWithMagicLink`, `auth.register`, `auth.resendSignUpOtp` with rate-limited procedures.
   - Wrap `chat.getAuth`, `reservationChat.getSession`, `supportChat.getClaimSession`, `supportChat.getVerificationSession`, and `chatPoc.getAuth/getOrCreateDm` with protected rate-limited procedures.
4. Add dedicated rate-limit tiers for provider-cost paths
   - Add tiers for `authEmailSend` and `chatSession` in rate-limit config and apply them to routers above.
5. Add audit logs for abuse visibility
   - Log limiter denials and chat token/session issuance counts with route+tier+actor metadata (no secrets/tokens).

### P1 - Chat anti-spam and upload abuse controls (second PR)
1. Add server-side chat spam guards in send flows
   - Add duplicate-content short-window suppression (user+channel+message fingerprint).
   - Add per-channel send ceiling in addition to existing per-user `chatSend` tier.
2. Tighten attachment validation server-side
   - Enforce attachment URL host allowlist (Stream CDN/domain set), strict MIME and size policy, and max attachment count parity.
3. Harden client upload behavior
   - Add pre-upload file type/size checks before `sendFile`.
   - Prevent upload bursts in a single compose action and surface clear user errors.
4. Apply provider-side Stream controls (operational checklist)
   - Restrict channel permissions to members, set upload size/type caps, enable moderation/anti-spam policies, and review token lifetime settings.

### P2 - Broader cost-surface protections (follow-up PRs)
1. Throttle additional expensive endpoints
   - Public availability range queries and other heavy read paths.
   - Bookings import normalize/commit flows (especially AI mode).
   - Owner/admin mutation surfaces that can fan out notifications or chat actions.
2. Make AI usage quota enforcement atomic
   - Remove race between "already used" check and `aiUsedAt` update in bookings import normalize flow.
3. Harden cron and admin amplifiers
   - Require cron secret in all non-local environments.
   - Add limiter coverage to admin test-dispatch routes/backfill triggers.

## Verification (End-To-End)
1. Static checks
   - Run `pnpm lint`.
2. Abuse simulation checks
   - Burst anonymous calls with rotating `x-request-id`; confirm limits still trigger consistently.
   - Burst public auth email-triggering routes; confirm throttling and stable 429 behavior.
   - Burst `chat.getAuth` and `reservationChat.getSession`; confirm throttling prevents token/session storms.
3. Chat spam checks
   - Send rapid duplicate messages in same channel; confirm suppression/limits apply.
   - Attempt attachment spam (too many files, oversized files, disallowed MIME/host); confirm upload/send rejection.
4. Safety checks
   - Validate normal user journeys still work (login, reservation chat messaging, support chat messaging).
   - Verify logs include abuse signals (tier, route, actor class, requestId) and do not include secrets/tokens.
5. Post-deploy monitoring
   - Track 429 rates, auth email send volume, chat token issuance rate, chat upload volume, and provider error spikes for 24-48 hours.
