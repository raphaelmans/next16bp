# Payment Submit Freshness Plan - 2026-02-06

## Goal
Remove stale `AWAITING_PAYMENT` false negatives after payment submit/proof upload by forcing immediate cache convergence before redirecting.

## Root Cause
- Payment/detail pages read `reservation.getDetail`, but `useMarkPayment` currently invalidates `reservation.getById`.
- List queries are invalidated, but with global `staleTime: 60s`, users can briefly see old status while refetch catches up.
- This is client query cache freshness, not server page cache.

## Recommended Implementation

### 1) Fix invalidation target for payment mutation
Primary file:
- `src/features/reservation/hooks.ts`

Changes:
- In `useMarkPayment.onSuccess`, invalidate `reservation.getDetail` (scoped by `reservationId`) instead of/alongside `reservation.getById`.
- Keep existing list/chat invalidations (`getMy`, `getMyWithDetails`, `reservationChat.*`).

### 2) Invalidate detail query on proof upload success
Primary file:
- `src/features/reservation/hooks.ts`

Changes:
- In `useUploadPaymentProof.onSuccess`, also invalidate `reservation.getDetail` for the same reservation.
- Keep existing list invalidations.

### 3) Force deterministic fresh detail before redirect
Primary file:
- `src/app/(auth)/reservations/[id]/payment/page.tsx`

Changes:
- Use `trpc.useUtils()` on payment page.
- After `markPayment.mutateAsync(...)` succeeds and before navigation, run:
  - `await utils.reservation.getDetail.fetch({ reservationId })`
- Keep current redirect target and flow unchanged.

### 4) Do not use `router.refresh` for this fix
Rationale:
- Problem is TanStack/tRPC client cache keys; targeted query fetch is lower-risk and deterministic for the exact stale key.

## Critical Files To Modify

- `src/app/(auth)/reservations/[id]/payment/page.tsx`
