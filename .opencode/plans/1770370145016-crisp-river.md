# Reservation Chat Auto-Message Plan - 2026-02-06

## Goal
Automatically post a default owner chat message when a reservation reaches `CONFIRMED` (created -> confirmed journey), so players get clear in-thread confirmation without waiting for manual owner outreach.

## Confirmed Product Decision
- Trigger auto-message on owner-driven confirmation to `CONFIRMED`.
- Scope includes created->confirmed paths:
  - `acceptReservation` free-booking branch (`CREATED -> CONFIRMED`)
  - `confirmPaidOffline` (`CREATED -> CONFIRMED`)
  - `confirmPayment` (`PAYMENT_MARKED_BY_USER -> CONFIRMED`)

## Root Cause
- Backend has thread ensure operations but no server-side reservation message-posting API.
- Confirmation flows enqueue notifications but do not seed an in-thread owner confirmation message.

## Recommended Implementation

### 1) Add server-side reservation message posting capability
Files:
- `src/lib/modules/chat/providers/chat.provider.ts`
- `src/lib/modules/chat/providers/stream-chat.provider.ts`

Changes:
- Extend provider contract with a reservation-channel send method (owner-authored message).
- Accept deterministic message id/idempotency key and message text.
- In Stream implementation, send message into `messaging` reservation channel and treat duplicate-id errors as idempotent success.

### 2) Add chat op for "owner confirmed" default message
Primary file (new):
- `src/lib/modules/chat/ops/post-owner-confirmed-message.ts`

Changes:
- Resolve reservation channel via existing `ensureReservationThreadForReservation`.
- Post default owner message with deterministic id pattern (e.g. `reservation:{reservationId}:owner-confirmed:v1`).
- Non-blocking behavior: throw only for unexpected provider failures; caller can catch and warn.

### 3) Wire confirmation flows in owner service (post-transaction, best effort)
Primary file:
- `src/lib/modules/reservation/services/reservation-owner.service.ts`

Changes:
- After successful transaction completion, best-effort post default message for each owner-driven path that ends at `CONFIRMED`:
  - `acceptReservation` free branch
  - `confirmPaidOffline`
  - `confirmPayment`
- Use existing pattern from reservation creation chat ensure (outside DB tx + `logger.warn` on failure) so reservation confirmation never fails due to chat outage.

### 4) Add supporting dependency to resolve player user id
Files:
- `src/lib/modules/reservation/services/reservation-owner.service.ts`
- `src/lib/modules/reservation/factories/reservation.factory.ts`

Changes:
- Inject/use profile repository in owner service to map `reservation.playerId` -> `playerUserId` for reservation channel member set.
- Skip posting safely when player user id cannot be resolved (guest/non-chat-eligible cases).

### 5) Document behavior in user stories
Files:
- `agent-plans/user-stories/67-reservation-chat/67-01-player-messages-venue-from-reservation.md`
- `agent-plans/user-stories/67-reservation-chat/67-02-owner-inbox-messages-across-reservations.md`
- `agent-plans/user-stories/67-reservation-chat/67-04-player-chat-widget-on-reservations-switches-threads.md`
- `agent-plans/user-stories/67-reservation-chat/67-10-chat-support-diagram.md`

Changes:
- Add AC that confirmation to `CONFIRMED` auto-seeds one default owner message in thread history.
- Add duplicate-prevention expectation (message appears once).
- Add cross-surface visibility expectation (both owner inbox + player widget see it).

## Default Message (recommended)
- `Your reservation is confirmed. If you need anything before your schedule, message us here.`

### Command
- `pnpm lint`

### Manual scenarios
1. Paid path: `CREATED -> AWAITING_PAYMENT -> PAYMENT_MARKED_BY_USER -> CONFIRMED`; verify one auto owner message appears at confirmation.
2. Free path: owner accepts free reservation (`CREATED -> CONFIRMED`); verify one auto owner message appears.
3. Offline paid path: owner confirms paid offline (`CREATED -> CONFIRMED`); verify one auto owner message appears.
4. Retry/idempotency: repeated confirm attempts should not create duplicate default messages.
5. Failure safety: if chat provider send fails, reservation still becomes `CONFIRMED` and failure is logged.
6. Visibility: message appears in both player and owner chat surfaces for that reservation thread.
