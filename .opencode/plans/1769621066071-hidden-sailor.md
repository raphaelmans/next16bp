# Public Court Page Under Venue (Detail + Availability)

## Context

- Current public canonical "venue/place" routes are plural: `appRoutes.places.base = "/venues"` (`src/shared/lib/app-routes.ts`).
- `src/app/(public)/courts/[id]/page.tsx` is a legacy alias for the *place/venue* detail page (it re-exports `src/app/(public)/places/[placeId]/page.tsx`).
- The public schedule page (`src/app/(public)/courts/[id]/schedule/page.tsx`) is a dense month/day browser for a place and is `noindex` via its layout.
- There is already a client booking picker embedded in the place detail page (`src/app/(public)/places/[placeId]/place-detail-client.tsx`).

## Goal

Create a new public route:

- `/venues/<venueSlug>/courts/<courtId>`

That page should:

- Show court-focused details (court label/tier/sport) within the venue context (name, location, photos/contact as needed).
- Show availability for that specific court (and allow booking flow) on the same page.
- Reuse existing data/availability primitives and UI patterns where possible.
- Respect `place.timeZone` as canonical for all date math.

## Confirmed Decisions

- URL: `/venues/<venueSlug>/courts/<courtId>` (plural, consistent with existing canonicals)
- Availability UI v1: reuse existing month/day schedule patterns (no week-grid)
- SEO: treat as an indexable detail page, but set `alternates.canonical` to the clean path (exclude query params)

- We want the first screen (mobile + web) to clearly communicate: you can browse availability immediately.

### Reality Check (Routing)

## Follow-up: Adjacent Multi-Slot Selection (Two-Click Range) in Month/Day Views

### Why

- Backend booking is already a single continuous interval: `startTime + durationMinutes` (hour-multiples, max 24h).
- Today the public month/day UI is single-select (one slot id). Users can book 9:00–12:00 by setting `duration=180` and selecting start time 9:00, but the UI doesn’t support “select a range” interaction.

### Recommendation (Consistency With Owner Studio)

Adopt an owner-studio-like “time block” mental model, but keep it lightweight for player booking:

- Selection becomes a contiguous range on a single day (adjacent 60-min blocks).
- Range selection produces the existing booking payload:
  - `startTime = first selected block startTime`
  - `durationMinutes = numberOfBlocks * 60`

This is not a true cart (multi-day / disjoint selections). It’s a contiguous range only.

### UX Proposal (Studio-Like)

- Primary interaction: two-click range selection (tap-start / tap-end):
  - First click sets the start.
  - Second click sets the end (inclusive) and highlights the full range.
  - Third click resets and becomes the new start.
- Drag is optional:
  - Pointer drag preview can exist, but selection commit must not interfere with taps.
- Constraints:
  - Selection must stay within a single day.
  - Selection cannot include blocked hours (reservation / maintenance / walk-in).
  - Snap strictly to 60 minutes (matches availability + reservation validation).
- Show endpoint labels + derived summary (eg `9:00 AM – 12:00 PM · 3 hours · ₱1500`).
- Motion:
  - Use `motion` for subtle transitions (opacity/scale), but respect `useReducedMotion()`.

### Component Strategy

- Keep `TimeSlotPicker` as-is for single-start-time flows.
- Introduce a dedicated range picker component that owns two-click selection state.
- For week view, use a grid-based picker (admin-style) to match the studio mental model.

### Component/API Changes

- Add new shared component (suggested path):
  - `src/shared/components/kudos/time-range-picker.tsx`
  - Inputs:
    - `dayKey`, `timeZone`
    - `slots: TimeSlot[]` for that day, **queried at 60 minutes**
    - `selectedStartTime?: string`, `selectedDurationMinutes?: number`
  - Outputs:
    - `onSelectRange({ startTime, durationMinutes, endTime, totalPriceCents })`

- Update `src/shared/components/kudos/availability-month-view.tsx`:
  - Change the right column from “render TimeSlotPicker for each day in a scroll list” to:
    - “render a single selected day timeline” (calendar on the left, timeline on the right)
  - Keep prop API stable for existing consumers where possible, but expect call-site updates.

- Keep `src/shared/components/kudos/time-slot-picker.tsx` intact for other flows (place detail page etc), unless we explicitly want range selection everywhere.

### Data + Page Wiring

To support drag range selection, fetch availability in **60-minute blocks** and derive duration from selection.

- Month view:
  - Keep the existing month range query, but set `durationMinutes=60`.
  - Build `days[]` as 60-min slots.

- Day view:
  - Set `durationMinutes=60` for the day availability query.
  - Derive the booking `durationMinutes` from the dragged selection (N * 60).

This keeps the backend unchanged while enabling a studio-like “select a block” interaction.

- Update public schedule consumers to maintain range state:
  - `src/app/(public)/courts/[id]/schedule/page.tsx`
  - `src/app/(public)/places/[placeId]/courts/[courtId]/court-detail-client.tsx`
  - (and any other call sites found by TS)

State should stay URL-driven:

- Persist `startTime` + `duration` (minutes) in the URL (existing pattern).
- Derive the highlighted range from these values (do not attempt to persist an array of ids in the URL).

### Validation Rules

- Only allow ranges composed of “available” blocks.
- Only allow selecting within a single day.
- Clamp duration to existing constraints (60–1440 minutes, 60-min steps).
