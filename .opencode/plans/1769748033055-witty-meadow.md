Reintroduce Block Resizing (Walk-in + Maintenance)
Goal
- Restore resizing for `WALK_IN` + `MAINTENANCE` court blocks.
- Keep UX: optimistic UI updates while resizing + debounced persistence.
- Constraint: resizing can only extend into the next adjacent VALID/AVAILABLE slot(s):
  - no overlaps with reservations/other blocks
  - for walk-ins, only within open hours (pricing availability remains server-validated)

Scope
- Owner bookings studio: `src/app/(owner)/owner/bookings/page.tsx`
- Owner court availability page: `src/app/(owner)/owner/places/[placeId]/courts/[courtId]/availability/page.tsx`
- Shared UI pieces:
  - `src/features/owner/components/booking-studio/timeline-block-item.tsx`
  - `src/features/owner/components/booking-studio/resize-handle.tsx`

Design
1) Keep pointer-based resize handles (no dnd-kit dependency).
2) Move “is this resize allowed?” logic out of `ResizeHandle` and into the page-level handlers where we have:
   - place time zone
   - court hours windows
   - active blocks + reservations
3) Implement stepwise clamping: for a drag delta of N hours, only apply as many +/-1 hour steps as remain valid/available.

Implementation steps
1) Add a shared resize validator/helper
   - New helper: `src/features/owner/components/booking-studio/helpers.ts` (or similar) exporting:
     - `getBlockDayKey(startTime, timeZone)`
     - `buildDayAvailability({ courtHoursWindows, timeZone, ignoreBlockId, blocks, reservations })` returning `isHourAvailable(hourIdx)` for 0..23
     - `clampResizeHoursDelta({ block, edge, hoursDelta, isHourAvailable })` returning a safe delta (may be smaller or 0)
   - Availability definition:
     - open hour: within court-hours windows
     - free hour: no overlap with other active blocks/reservations (touching is allowed)
     - pricing: not checked client-side (server remains source of truth)

2) Restore resize preview/commit wiring in owner bookings page
   - Re-introduce `handleResizePreview` + `handleResizeCommit`:
     - Preview: optimistic cache update for `trpc.courtBlock.listForCourtRange`
     - Commit: write to `pendingRangeUpdates` + bump `rangeUpdateVersions` + `scheduleRangeFlushRef.current(blockId)`
   - Before applying a preview/commit, compute and clamp the delta using the helper so the block only extends into valid adjacent slots.
   - Only pass resize props to `TimelineBlockItem` when:
     - `block.type` is `WALK_IN` or `MAINTENANCE`
     - block is not pending and not past-day (if you want to keep past edits disabled)
   - Files:
     - `src/app/(owner)/owner/bookings/page.tsx`
     - `src/features/owner/components/booking-studio/timeline-block-item.tsx`

3) Add the same debounced resize pipeline to owner court availability
   - Mirror the bookings page approach (pending map + versioning + debounce flush) OR extract it into a small hook:
     - `src/features/owner/components/booking-studio/use-debounced-block-range-update.ts`
   - Wire `onResizePreview`/`onResizeCommit` into:
     - Day view `TimelineBlockItem`
     - Week view `WeekDayColumn` (props already plumb through)
   - Reuse the same helper clamp logic.
   - File:
     - `src/app/(owner)/owner/places/[placeId]/courts/[courtId]/availability/page.tsx`

4) UX polish for invalid resize attempts
   - If the user drags into invalid territory, clamp to the last valid step; if clamped delta is 0, no-op.
   - Optional: show a subtle toast only on commit (not during preview) when resize is blocked.

Verification

  - Create a walk-in block; resize end later by 1 hour into a free open hour -> should preview, then persist after debounce.
  - Try to resize into an hour that overlaps an existing reservation/block -> should stop/clamp.
  - Try to resize into closed hours -> should stop/clamp (walk-in).
  - Maintenance resize should clamp for overlaps but may allow closed hours (per server rules).
- Manual (availability page): same checks in day and week views.
