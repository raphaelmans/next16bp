# Bug Review Plan - 2026-02-06

## Goal
Fix the 3 filed bugs with minimal-risk, targeted changes:
1) remove misleading notification bell attention state,
2) make amenities filtering resilient to production data drift,
3) ship a quick courts-image UX fix (no full court photo feature yet).

## Root Cause Review
- Bug 1 (bell dot): `NotificationBell` currently shows an orange dot for browser push setup state, not unread notifications (`src/features/notifications/components/notification-bell.tsx`). The dropdown only contains push toggle/settings UI, so users misread the dot as unread.
- Bug 2 (amenities filter): discovery filtering uses exact equality against `place_amenity.name` in `listBaseRecords` (`src/lib/modules/place/repositories/place.repository.ts`). Case/whitespace variants can fail matching even when venues have the amenity.
- Bug 3 ("No img"): courts list expects `coverImageUrl`, but owner mapping hardcodes `undefined` (`src/features/owner/hooks.ts`), and court edit has no image upload field. User requested quick UX fix first.

## Implementation Plan

### 1) Bug 1 - Notification bell dot should not imply unread
Files:
- `src/features/notifications/components/notification-bell.tsx`

Steps:
- Remove the orange attention dot rendering tied to web-push setup state.
- Keep the browser notification toggle behavior unchanged (enable/disable flow stays intact).
- Update dropdown supporting copy to clearly position this popover as notification settings (not an unread inbox).

Acceptance:
- Bell does not show an unread-style badge when there are no actual notification items.
- Browser notification toggle/toasts continue to work as today.

### 2) Bug 2 - Amenities filter should match place amenities reliably
Files:
- `src/lib/modules/place/repositories/place.repository.ts`
- `src/features/discovery/components/court-filters.tsx`

Steps:
- Normalize incoming amenity filters in repository to canonical values (`trim + lowercase + dedupe`).
- In SQL predicates, compare using canonical DB expression (`lower(btrim(place_amenity.name))`) instead of exact raw-name equality.
- Keep existing AND semantics by counting distinct canonical amenity values in `HAVING`.
- Apply the same canonical matching logic in both sport-constrained and non-sport query branches inside `listBaseRecords`.
- In filter UI, use the source option value from the mapped amenity item in `onSelect` to avoid command-normalized value drift.

Acceptance:
- Selecting `Parking` returns places that have `Parking`, `parking`, or equivalent whitespace/case variants in `place_amenity`.
- Multi-select amenities still behave as AND (place must satisfy all selected amenities).
- No regression for non-amenity filters.

### 3) Bug 3 - Quick UX fix for courts image placeholders
