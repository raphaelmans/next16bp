# Plan: Final PlaceDetail composition pass (contain query/state/mutation by semantic section)

## Goal
Reduce `src/features/discovery/place-detail/place-detail-page-view.tsx` from a root-level orchestrator into a thin composition shell where each semantic section owns its own query/state/mutation domain.

## Current issues to fix
1. `place-detail-page-view.tsx` still owns high-churn domains (selection state, availability queries, derived slot maps, mutations/forms, analytics, actions) and remains ~1000+ lines.
2. Root rerenders are broad because many unrelated subscriptions live in the same component.
3. Section components are mostly presentational but depend on large prop surfaces, which indicates ownership is still in root.

## Recommended target architecture
1. **Thin page shell**
   - `src/features/discovery/place-detail/place-detail-page-view.tsx`
   - Own only route param resolution + place loading/not-found branching + layout composition.

2. **Booking domain boundary (semantic section owns booking concerns)**
   - `src/features/discovery/place-detail/components/sections/place-detail-booking-section.tsx`
   - Own:
     - URL selection state consumption (`usePlaceDetailAvailabilitySelection`)
     - all availability queries (desktop + mobile)
     - mobile week prefetch (`useMobileWeekPrefetch`)
     - derived booking models (slots/maps/diagnostics/loading/error/summary)
     - booking actions (date/view/mode/sport/court handlers, reserve flow, analytics)
   - Render and feed:
     - `PlaceDetail.AvailabilityDesktop`
     - `PlaceDetail.MobileSheet`
     - booking summary in sidebar

3. **Listing help domain boundary (semantic section owns forms + mutations)**
   - `src/features/discovery/place-detail/components/sections/place-detail-listing-help-section.tsx`
   - Own:
     - auth/org query needed for claiming
     - RHF claim/removal forms
     - claim/removal mutations + toasts + invalidation
     - claim/removal dialog open state (scoped here, not root)
   - Render and feed:
     - `PlaceDetailListingHelpCard`

4. **Sidebar composition boundary**
   - `src/features/discovery/place-detail/components/place-detail-sidebar.tsx`
   - Accept semantic models/slots from booking/listing-help sections (not raw low-level primitives).

5. **Contact/Location/Courts remain independent low-churn sections**
   - Keep these components focused and stateless from root perspective.

## Implementation sequence
1. Create booking section controller component (`place-detail-booking-section.tsx`) by moving all booking/availability logic out of `place-detail-page-view.tsx`.
2. Create listing-help section controller component (`place-detail-listing-help-section.tsx`) by moving forms/mutations/org query out of root.
3. Replace root-level prop plumbing with semantic section props (model/action objects) and section slots.
4. Shrink `place-detail-page-view.tsx` to composition-only responsibilities.
5. Update `src/features/discovery/place-detail/components/place-detail.tsx` and `src/features/discovery/place-detail/index.ts` exports for new section components.
6. Remove dead imports/hooks/state from root and any now-unused store fields/selectors.

## Rerender containment rules during implementation
1. `place-detail-page-view.tsx` must not subscribe to availability query states, form states, mutation pending states, or now-timer state.
2. High-churn subscriptions stay inside section controller components.
3. Pass semantic `model`/`actions` objects to presentation components instead of large primitive prop sets.
4. Keep `nuqs` as canonical booking URL state, `zustand` for ephemeral UI state, and RHF for form state.
