# Automated reservation-created chat message + system labeling

## User goal
- Send an automated chat message when a player creates a reservation so owners get immediate context in the reservation thread.
- Keep this auto-message immediate and authored as the player.
- Add `[SYSTEM GENERATED]` label to both automated reservation messages:
  - player reservation-created message (new)
  - owner confirmation message (existing)

## Current behavior (confirmed)
- Reservation creation currently writes DB state/events and enqueues owner notifications, then best-effort ensures chat thread only.
- There is no reservation-created auto chat message yet.
- Existing owner-confirmed auto chat message exists in `src/lib/modules/chat/ops/post-owner-confirmed-message.ts`.

## Scope and files to update
- `src/lib/modules/chat/ops/post-player-created-message.ts` (new)
- `src/lib/modules/chat/ops/post-owner-confirmed-message.ts`
- `src/lib/modules/reservation/services/reservation.service.ts`

## Implementation plan
1. Add a dedicated chat op for player reservation-created automation.
   - Create `post-player-created-message.ts` modeled after existing owner-confirmed op.
   - Implement helpers:
     - `buildPlayerCreatedMessage(reservationId: string)`
     - `makePlayerCreatedMessageId(reservationId: string)` with deterministic ID format `reservation:${reservationId}:player-created:v1`.
   - Message copy: concise text prefixed with `[SYSTEM GENERATED]` and no extra default list items.
   - In op flow, call `ensureReservationThreadForReservation(...)` and then `provider.sendReservationMessage(...)` with:
     - `createdById = playerUserId` (authored as player)
     - deterministic `messageId` for idempotency.

2. Trigger the new op from both reservation creation flows as best-effort post-commit side effect.
   - In `reservation.service.ts`, add import for `postPlayerCreatedMessage`.
   - Add a private helper (best-effort wrapper) to:
     - resolve `playerUserId` from `profileRepository.findById(profileId)`
     - use `ownerUserId` from organization lookup
     - return early when IDs are missing
     - catch/log failures without affecting reservation creation success.
   - Use this helper in both:
     - `createReservationForCourt(...)`
     - `createReservationForAnyCourt(...)`
   - Keep execution outside transaction, same reliability pattern as current chat side effects.

3. Add `[SYSTEM GENERATED]` prefix to owner-confirmed automated message.
   - Update `buildOwnerConfirmedMessage(...)` in `post-owner-confirmed-message.ts` so the opening line starts with `[SYSTEM GENERATED]`.
   - Keep existing owner-confirmed idempotency key (`reservation:${reservationId}:owner-confirmed:v1`) and flow unchanged.

4. Logging and failure handling.
   - Keep chat automation non-blocking (warn-level logs only on failure).
   - Add/keep clear event names in warning payloads for traceability:
     - `reservation.player_created_chat_message_failed`
     - `reservation.owner_confirmed_chat_message_failed` (existing pattern)

## Verification
- Run `pnpm lint`.
- Manual end-to-end (player + owner accounts):
  - Create reservation through standard player flow (court-specific path) and verify one new chat message appears in the reservation thread with `[SYSTEM GENERATED]` prefix and player authorship.
  - Create reservation through any-court flow and verify same behavior.
  - Confirm reservation as owner (any confirm path that posts owner-confirmed message) and verify message now includes `[SYSTEM GENERATED]` prefix.
- Idempotency checks:
  - Ensure repeated execution/retry does not create duplicate reservation-created auto messages for the same reservation.
  - Ensure repeated execution/retry does not create duplicate owner-confirmed auto messages.

## Notes
- This plan intentionally keeps chat posting as best-effort so reservation creation UX is never blocked by chat provider failures.
- If we later want stronger observability, we can add a success event (`reservation.player_created_chat_message_posted`) after rollout.
