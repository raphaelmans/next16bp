# Browser Notification System (Web Push)

## Why / context
- This repo already has an async notification delivery system (outbox jobs + cron dispatcher) documented in `docs/notification-system/*` and implemented via `notification_delivery_job` + `src/app/api/cron/dispatch-notification-delivery/route.ts`.
- Reservation lifecycle states (and which transitions matter) are captured in `docs/reservation-state-machine-diagram.md`.
- Reservation chat is Stream-backed; chat visibility is state-gated for players and always mounted for owners (see `/Users/raphaelm/Documents/Coding/boilerplates/next16bp/agent-plans/user-stories/67-reservation-chat/*.md`). Provider webhook ingestion is deferred there, which impacts “notify on new message”.

Goal: add Facebook-like browser notifications using browser APIs (Service Worker + Push API + Notification API), while reusing the existing outbox + cron pattern so delivery is async and resilient.

## Scope
### MVP (recommended)
- Add a new delivery channel: `WEB_PUSH`.
- Store per-user Web Push subscriptions (multi-device).
- Enqueue + dispatch Web Push notifications for:
  - existing MVP events (`place_verification.*`, `claim_request.*`, `reservation.created`)
  - key reservation state transitions that require action (from the reservation state machine)
- Provide UX to opt-in/out (permission + subscription management).

### Not in MVP (plan for later)
- Full in-app notification center (bell dropdown, read/unread, history).
- Chat “new message” browser notifications (requires Stream webhook ingestion or provider support).
- Per-user notification preferences by event/category.

## Proposed architecture
### Delivery model (keep consistent with current outbox)
- Domain event/state transition happens.
- Inside the same DB transaction, enqueue one `notification_delivery_job` per (recipient x channel x target).
- Cron route claims jobs and delivers asynchronously.

For `WEB_PUSH`, the “target” is a **push subscription id** (one job per subscription/device). This keeps the existing job semantics intact (a single target and a single job state machine) and avoids adding a second attempt table.

- Migration: `ALTER TYPE notification_delivery_channel ADD VALUE 'WEB_PUSH';`

### 2) Add push subscription table
Add `push_subscription` table (new Drizzle schema + migration). Recommended columns:
- `id` uuid pk
- `userId` (the Supabase auth user id; matches `profile.userId` usage patterns)
- `endpoint` text (unique)
- `p256dh` text
- `auth` text
- `expirationTime` text nullable (store raw value as returned; often null)
- `userAgent` text nullable
- `createdAt`, `updatedAt` (tz)
- `revokedAt` (tz, nullable) for cleanup when subscription is gone

Indexes:
- unique(endpoint)
- index(userId, revokedAt)

## Server implementation plan
### 1) Web Push infra adapter
Add a new infra provider similar to email/sms:
- `src/lib/shared/infra/web-push/web-push-service.ts` (strategy interface)
- `src/lib/shared/infra/web-push/web-push.factory.ts`
- `src/lib/shared/infra/web-push/web-push.provider.ts` (wraps `web-push` library)

Behavior:
- `send(subscription, payload, options)` -> throws typed/structured errors with HTTP status (410/404 => subscription gone)
- Uses `webpush.setVapidDetails(WEB_PUSH_VAPID_SUBJECT, publicKey, privateKey)`
- Keep payload small (typical push payload limits are ~4KB)

### 2) Push subscription module
Add module (pattern: repository/service/router):
- `src/lib/modules/push-subscription/repositories/push-subscription.repository.ts`
- `src/lib/modules/push-subscription/services/push-subscription.service.ts`
- `src/lib/modules/push-subscription/push-subscription.router.ts`
- `src/lib/modules/push-subscription/factories/push-subscription.factory.ts`

tRPC endpoints (protected):
- `upsertMySubscription(subscriptionJson, userAgent?)`
  - dedupe by `endpoint` (unique)
  - if endpoint exists for another userId, reassign to current user (shared device safety)
  - set `revokedAt = null`, update keys, update `updatedAt`
- `revokeMySubscription({ endpoint } | { id })`
  - soft revoke (`revokedAt = now`)

### 3) Enqueue rules (reservation + existing notifications)
Extend `NotificationDeliveryService` to create `WEB_PUSH` jobs **per subscription** (query active subscriptions by userId, then create a job per subscription id).

Files to touch:
- `src/lib/modules/notification-delivery/services/notification-delivery.service.ts`
- `src/lib/modules/notification-delivery/repositories/notification-recipient.repository.ts` (ensure you have userIds needed)

Event matrix (aligns with `docs/reservation-state-machine-diagram.md`):
- `reservation.created` -> owner (action: accept/reject)
- `reservation.awaiting_payment` (CREATED -> AWAITING_PAYMENT) -> player (action: pay before expiresAt)
- `reservation.payment_marked` (AWAITING_PAYMENT -> PAYMENT_MARKED_BY_USER) -> owner (action: confirm)
- `reservation.confirmed` -> player (action: show confirmation + chat CTA)
- `reservation.rejected` (owner rejects) -> player
- `reservation.cancelled` (player cancels) -> owner
- `reservation.expired` (system cron) -> both (optional in MVP; include if your product wants explicit expiry notices)

Deep links:
- Owner: `appRoutes.owner.reservationDetail(reservationId)`
- Player: `appRoutes.player.reservationDetail(reservationId)` (or the equivalent in `appRoutes`)

Guest bookings:
- Do not enqueue player Web Push for guest reservations (no player userId / chat unsupported).

### 4) Reservation transition enqueue points
Add enqueue calls at transaction chokepoints that perform the status updates:
- `src/lib/modules/reservation/services/reservation-owner.service.ts`:
  - after accept -> AWAITING_PAYMENT: enqueue `reservation.awaiting_payment` to player
  - after accept free -> CONFIRMED: enqueue `reservation.confirmed` to player
  - after confirmPayment/confirmPaidOffline -> CONFIRMED: enqueue `reservation.confirmed` to player
  - after reject -> CANCELLED: enqueue `reservation.rejected` to player
- `src/lib/modules/reservation/services/reservation.service.ts`:
  - after markPayment -> PAYMENT_MARKED_BY_USER: enqueue `reservation.payment_marked` to owner
  - after cancelReservation -> CANCELLED: enqueue `reservation.cancelled` to owner
- Expiry use case (if emitting `reservation.expired`): `src/lib/modules/reservation/use-cases/expire-stale-reservations.use-case.ts`

## Cron dispatcher updates
Extend `src/app/api/cron/dispatch-notification-delivery/route.ts`:
- Add channel toggle: `env.NOTIFICATION_WEB_PUSH_ENABLED !== false`
- Add eventType handlers for new reservation events (zod payload schemas + message builders)
- Add `WEB_PUSH` channel delivery:
  - Interpret `job.target` as `push_subscription.id`
  - Load subscription record; if missing or revoked -> `SKIPPED` (`MISSING_SUBSCRIPTION` / `SUBSCRIPTION_REVOKED`)
  - Send via `web-push` with JSON payload: `{ title, body, icon, data: { url }, tag }`
  - On 410/404: mark subscription revoked, mark job `SKIPPED` (permanent)
  - On 429/5xx/network: mark job `FAILED` and apply existing backoff schedule

Logging:
- Reuse existing events (`notification_delivery.dispatch_complete`, `notification_delivery.failed`) and add `web_push.subscription_revoked`.

- `src/features/notifications/components/web-push-settings.tsx` (client)
- Mount on an account/settings page route (server page imports the feature component).

Flow:
- Show current `Notification.permission` state + browser support (SW + PushManager)
- On button click:
  - register SW (`navigator.serviceWorker.register("/sw.js")`)
  - request permission (must be user-initiated)
  - subscribe with VAPID public key
  - call `pushSubscription.upsertMySubscription` (tRPC)
- Disable:
  - `subscription.unsubscribe()` + call `revokeMySubscription`

Logout hygiene (recommended):
- On logout mutation success, attempt to unsubscribe + revoke current subscription to avoid shared-device leaks.

## Verification checklist
### Local (Chrome)
1) Start app (`pnpm dev`).
2) Navigate to settings page, grant permission, ensure subscription upserts.
3) Trigger an event that enqueues a job (e.g., create reservation, accept reservation, mark payment).
4) Dispatch via cron route (admin tool `dispatchNow` or hit `/api/cron/dispatch-notification-delivery`).
5) Confirm:
   - Notification appears
   - Clicking notification deep-links correctly
   - Re-dispatch does not duplicate for already `SENT` jobs

### Invalid subscription
1) Unsubscribe in browser.
2) Dispatch a job to that subscription.
3) Confirm dispatcher revokes subscription on 410/404 and marks job `SKIPPED`.

### Repo validation
- `pnpm lint`

### Admin debug flow
1) Log in as admin.
2) Go to `/admin/tools/notification-test`.
3) Register browser for push (grant permission).
4) Enqueue a `test.web_push` job and then click “Dispatch now”.
5) Confirm notification shows and clicking it navigates to the provided URL.

## Risks / notes
- Web Push requires HTTPS in production (localhost is OK).
- Safari/iOS support may require an installable PWA + `src/app/manifest.ts` (treat as follow-up if needed).
- Keep payload small; store large context in DB and deep-link instead.
