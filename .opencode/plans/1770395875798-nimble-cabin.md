# Plan: Investigate + fix owner reservations horizontal overflow (iPad Air portrait)

## What we confirmed
- Repro area: `src/app/(owner)/owner/reservations/page.tsx` inside `AppShell -> DashboardLayout -> SidebarInset`.
- Device target: iPad Air portrait is approximately `820x1180` CSS viewport (Apple reports `2360x1640` physical at 2x DPR, and device viewport references list `820x1180`).
- Overflow is likely not one single cause; it is the combination of:
  1. flex shrink boundary in sidebar content lane,
  2. tabs list sizing (`w-fit` + nowrap triggers),
  3. dense desktop table becoming active at `md` with many nowrap cells/actions.

## Recommended implementation approach
1. **Fix layout shrink boundary first (lowest risk, highest impact).**
   - Add `min-w-0` to `SidebarInset` so content can shrink within the sidebar layout.
   - Add `min-w-0` to the `#main-content` wrapper in `AppShell` to avoid inherited min-content overflow pressure.

2. **Contain tab-strip overflow locally on reservations page.**
   - Wrap `TabsList` in a container that allows local horizontal scrolling (`overflow-x-auto`, `max-w-full`) so tab labels/badges never force page-level overflow.
   - Keep current tab visuals/behavior unchanged.

3. **Harden reservations table behavior for tablet widths.**
   - Keep the existing `md` desktop table first.
   - Ensure overflow is isolated to table container (not document/page).
   - If viewport overflow still reproduces at `820` after steps 1-2, apply targeted local table pressure reductions in `reservations-table` (e.g., allow wrapping for specific columns/actions) before changing global/shared table defaults.

4. **Fallback only if still needed:**
   - Move this specific table switch from `md` to `lg` (`md:hidden`/`hidden md:block` -> `lg:hidden`/`hidden lg:block`) to keep cards on iPad portrait.
   - Treat this as behavior change, not first-line fix.

## Verification checklist
1. Run app locally and reproduce on owner reservations route.
   - `pnpm dev`
   - Check responsive widths: `768`, `820` (iPad Air portrait), `900`, `1024`, plus `1180x820` landscape.

2. Confirm page-level overflow is gone.
   - In browser console at target widths: verify `document.documentElement.scrollWidth <= window.innerWidth`.
   - Confirm horizontal scrolling (if any) happens only inside tabs row or table container.

3. Regression checks.
   - Sidebar open/collapsed states do not reintroduce overflow.
   - Tabs are still keyboard/focus accessible.
   - Mobile card layout (`< md`) unchanged.
   - Desktop table (`>= lg` or current `>= md`, depending on final step used) remains usable.

4. Project validation.
   - `pnpm lint`

## Note on "shadcn default?"
- Shared `Tabs` and `Table` primitives are shadcn-style defaults (`w-fit` tabs list, nowrap table cells), but this bug appears from their interaction with this project's sidebar/flex layout and the reservations table density at tablet widths.
- So this is **not purely a shadcn bug**; it is a composition/layout boundary issue in this screen.
