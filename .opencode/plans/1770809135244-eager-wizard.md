# Auth Confirm Redirect Analysis Plan

## Request
Investigate why clicking an email link like:

`/auth/confirm?redirect=%2Fowner%2Fget-started&token_hash=...&type=signup`

did not land on `/owner/get-started`.

## Findings from code review
- The target page exists and maps correctly: `src/app/(auth)/owner/get-started/page.tsx` -> `/owner/get-started`.
- `/auth/confirm` does read `redirect` (and `next`) and computes a safe redirect path via `getSafeRedirectPath`.
- For `type="signup"`, the handler calls `supabase.auth.verifyOtp({ token_hash, type: "signup" })` and redirects to the safe target on success.
- Any failure in confirm flow currently falls back to `/` (home), including missing params or OTP verification error.
- `/owner/get-started` is a protected route; if no valid session cookie is present on next request, proxy/auth guards redirect to `/login?redirect=/owner/get-started`.
- Navbar auth state is client-driven via `useSession()` (`trpc.auth.me`), so a short delay before the navbar flips to "logged in" is expected after redirect/hydration.

## Most likely root causes (ranked)
1. OTP verify failed (expired/used/invalid token hash) and handler silently fell back to `/`.
2. Confirm link was consumed earlier (email client/link scanner/open-preview) and user click hit a now-invalid token; current fallback still sends to `/`.
3. Session did get established, but UI delay in navbar (`useSession` query) makes the auth state update appear ~1s later.
4. Host/redirect config mismatch (Supabase allowed redirects, site URL, or app URL mismatch) causing callback/session inconsistencies.
5. Redirect sanitization fallback (malformed/unsafe redirect), less likely for `/owner/get-started`.

For the exact observed behavior (land on `/` then navbar flips logged-in), strongest suspects are #1/#2 plus #3.

## Recommended implementation approach

### 1) Harden confirm-route error behavior and preserve intent
Modify `src/app/auth/confirm/route.ts`:
- Keep current success behavior for valid OTP + safe redirect.
- Replace silent fallback to `/` with explicit redirect to login while preserving destination:
  - `/login?redirect=<safeRedirect>&auth_error=<reason>`
- Add structured logs with reason codes:
  - `missing_token_hash`, `missing_type`, `otp_verify_error`, `unknown_type`, `unsafe_redirect_fallback`.
- Do not log secrets (`token_hash`, full link).
- In OTP error paths, check whether a valid user session already exists (`supabase.auth.getUser()`); if session exists, proceed to intended redirect instead of `/`.

### 2) Add targeted auth guard observability
Modify `src/proxy.ts`:
- Log protected-route redirects to login with:
  - attempted `pathname`, `hasUser`, `request host`, and optional correlation metadata.
- This distinguishes "OTP failed" from "OTP passed but session missing".

### 3) Improve auth-state UX consistency (optional)
Modify `src/features/discovery/components/navbar.tsx` (or auth hook usage):
- Avoid rendering hard logged-out controls until `useSession()` has resolved at least once.
- This removes the visible 1s flicker where user appears logged out then logged in.

### 4) Optional UX improvement for diagnosability
Modify `src/features/auth/components/login-form.tsx` (optional):
- Read `auth_error` query param and show a friendly message (expired link, invalid link, etc.).

## Non-code checks (required during rollout)
- Supabase project Auth settings:
  - `Site URL` matches production domain.
  - Redirect allow-list includes all active domains/variants used in real emails.
- Hosted Supabase email templates match repo logic:
  - Use `{{ .RedirectTo }}` and append `token_hash`/`type` correctly.
- Environment parity:
  - `NEXT_PUBLIC_APP_URL` is correct in production and points to the same domain users open from email.

## Verification plan
1. Happy path: register owner, click fresh signup email, land on `/owner/get-started`.
2. Expired/used token: click invalid link, verify redirect to `/login?redirect=/owner/get-started&auth_error=...`.
3. Missing params: hit `/auth/confirm?redirect=/owner/get-started`, verify explicit error redirect (not `/`).
4. Unsafe redirect injection: ensure sanitizer falls back safely and logs reason.
5. Session persistence: immediately request `/owner/get-started` after verify and confirm no unintended login bounce.
6. Reused-token simulation: load same confirm URL twice; ensure second hit does not dead-end on `/` if user session is already valid.
7. Navbar UX check: after confirm redirect, verify no misleading "logged out" flicker.
8. Run validation: `pnpm lint`.
