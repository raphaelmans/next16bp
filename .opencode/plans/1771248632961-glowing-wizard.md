## Key Decisions (Recommended Defaults)

1) API base path + versioning
- Recommended: `src/app/api/mobile/v1/**` -> `/api/mobile/v1/**`.
- Why: isolates mobile contract, allows versioning, avoids collisions with existing `/api/public/**` and `/api/trpc/**`.

2) Mobile authentication mechanism
- Recommended: Bearer token auth using Supabase access token.
  - Client sends `Authorization: Bearer <supabase_access_token>`.
  - Server validates token (initially via Supabase Auth `getUser()` call) and enriches role from `user_roles`.
- Why: avoids cookie/session complexity in React Native, aligns with Expo SecureStore patterns, avoids CSRF.

Chosen for this migration: Supabase on-device auth + Bearer token to the REST API.

3) OpenAPI generation approach
- Recommended: use `zod-openapi` (`createDocument`) to generate OpenAPI 3.1 at build-time and/or via a dev route.
- Why: Context7 has strong docs coverage for it; it supports attaching metadata via `zod` `.meta()`.

4) Client connection strategy (Expo)
- Recommended: `openapi-typescript` -> generate `paths` types, then use `openapi-fetch` (optionally `openapi-react-query`) for typed requests.
- Why: very small runtime surface area and works well with TanStack Query.

## Implementation Plan (Modular, Step-by-Step)

### Phase A - REST/OpenAPI foundation (no domain changes)

1) Add OpenAPI spec generation infrastructure
- Add deps:
  - `zod-openapi` (server) to generate OpenAPI 3.1 from Zod schemas.
- Add files:
  - `src/lib/shared/infra/openapi/mobile-v1.document.ts`
    - builds a `createDocument({ openapi: "3.1.0", info, servers, components, paths })`
    - defines shared components:
      - `ApiErrorResponse` schema
      - `bearerAuth` security scheme
    - composes `paths` from per-module `*.openapi.ts` exports (see Phase B/C).
  - `src/app/api/mobile/v1/openapi.json/route.ts`
    - `GET` returns the generated document JSON (dev-friendly)
    - `runtime = "nodejs"`, `dynamic = "force-dynamic"`
    - uses existing `handleError()` for consistent error envelope

2) Create a standard Route Handler helper set (thin, reused everywhere)
- Add files:
  - `src/lib/shared/infra/http/request-id.ts` (helper to read/generate requestId)
  - `src/lib/shared/infra/http/parse.ts`
    - `parseJson(req)`
    - `parseQuery(req.url)`
    - `parseFormData(req)`
  - `src/lib/shared/infra/http/validate.ts`
    - `validate(schema, data)` that throws `ValidationError` with issue details (per node-architecture `core/error-handling.md`)

3) Add mobile session extraction + auth guard
- Add files:
  - `src/lib/shared/infra/auth/mobile-session.ts`
    - `getMobileSession(req): Promise<Session | null>`
      - parse `Authorization` Bearer token
      - validate via Supabase Auth (`getUser()`)
      - fetch role from `user_roles` (existing `makeUserRoleRepository().findByUserId(user.id)`)
    - `requireMobileSession(req): Promise<Session>` throws `AuthenticationError`

4) Align HTTP rate limiting with authenticated user identifiers
- Update or extend `src/lib/shared/infra/http/http-rate-limit.ts` so callers can supply an identifier override (e.g. `userId`).
  - Keep current IP-based default for anonymous endpoints.
  - For protected endpoints, rate limit by `session.userId` to match tRPC behavior.

5) Reduce proxy overhead for mobile API routes
- Update `src/proxy.ts` to skip Supabase cookie refresh for `/api/mobile/**` paths.
  - Early return `makeResponse()` if `path.startsWith("/api/mobile/")`.
  - Keeps existing behavior for `/api/trpc/**` and UI routes.

## OpenAPI Spec Content (Mobile v1)

Document requirements:
- OpenAPI version: `3.1.0`
- Server base: `/api/mobile/v1`
- Security scheme:
  - `bearerAuth` (HTTP bearer)
- Response envelope:
  - Success: `{ data: T }`
  - Error: `{ code, message, requestId, details? }`

Implementation approach:
- For each module, add a `*.openapi.ts` that exports:
  - `paths` object fragment (zod-openapi compatible)
  - shared Zod response schemas using `createResponseSchema()` and a local `ApiErrorResponseSchema`
- Compose all fragments in `src/lib/shared/infra/openapi/mobile-v1.document.ts`.

References (Context7)
- Next.js route handler config + `request.json()` usage (App Router): `/vercel/next.js` docs.
- OpenAPI generation from Zod `.meta()` and `createDocument`: `/samchungy/zod-openapi` docs.
- Expo client generation and typed fetch client: `/openapi-ts/openapi-typescript` docs.

- Use Supabase on-device auth.
- On each API call, attach access token.
- On 401, refresh via Supabase client and retry.

## Risks / Watchouts

- Large file uploads (venue photos, verification docs, bookings import) can exceed serverless body/time limits.
  - Mitigation: enforce size/type checks (already exists in `ObjectStorageService`), keep mobile upload sizes reasonable, and consider a later phase with direct-to-storage signed uploads.
- Proxy overhead: `src/proxy.ts` currently calls Supabase `getUser()` on every request.
  - Mitigation: skip for `/api/mobile/**` paths.
- Query param coercion: REST `GET` querystrings are strings; existing Zod DTOs may expect numbers.
  - Mitigation: create REST-specific query schemas using `z.coerce.*`.
