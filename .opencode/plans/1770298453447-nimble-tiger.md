## Approach

### 1-2) Create Organization modal (mobile centering + clipping)
Root causes:
- The dialog is vertically centered with fixed positioning; without `max-height` + `overflow`, tall content (keyboard, validation messages, loading state) clips offscreen.
- The modal renders a `Card` inside `DialogContent`, causing “double framing” (dialog padding/border + card padding/border) that makes mobile spacing feel cramped/unbalanced.

Changes:
1. Update `DialogContent` base styles to be viewport-safe on mobile:
   - Add `max-h-[calc(100dvh-2rem)] overflow-y-auto`.
   - Keep existing centering behavior.
   - This should also help other dialogs (including Claim search).

2. Remove the nested Card framing in the Create Organization dialog:
   - Make `OrganizationForm` render “dialog-friendly” content (no outer `Card`, no `max-w-*`), letting the dialog provide padding/border.
   - Ensure the form header still looks intentional (icon + title + description) but uses simple `div` layout.

### 3) Organization slug uniqueness
Reality check in repo:
- DB schema already has unique constraint/index on `organization.slug`.
- Service does a pre-check via `organizationRepository.slugExists(...)`, but it does not use the transaction client and it is not race-safe.

Changes:
1. Make `slugExists` accept an optional `ctx` and use `getClient(ctx)` so checks are consistent with the transaction.
2. Catch Postgres unique-violation on create/update as the final source of truth:
   - In `OrganizationRepository.create` and `OrganizationRepository.update`, catch unique constraint violations (`SQLSTATE 23505`) for slug.
   - Re-throw `SlugAlreadyExistsError` so the router returns a clear `CONFLICT`.

No DB migration planned (constraint already exists); we’re making the API behavior correct + user-friendly even under race conditions.

### 4) Claim existing listing search results overflow (mobile)
Root cause overlaps with (2): the dialog itself can exceed viewport and clip; the results list also needs more robust containment.

Changes:
1. Rely on the updated `DialogContent` (`max-h` + `overflow-y-auto`) to prevent modal clipping.
2. Improve the results list section in `ClaimSearchDialog`:
   - Add bottom padding inside the scrollable list so the last row/button isn’t flush/clipped.
   - Ensure the row’s Claim button is `shrink-0` so the text column truncation always wins.

### 5) Schedule: Apply to all overwrites instead of appending
Root cause:
- `handleApplyToAll` currently maps over existing rows and overwrites `startTime/endTime` (and pricing) on every open row, instead of adding a new block.

Changes:
1. Change `handleApplyToAll` in `src/features/owner/components/court-schedule-editor/hooks.ts` to append a cloned block to each other day:
   - Use a new row id (import `createRowId` from helpers).
   - Copy times + pricing from the source row.
   - Do not modify existing rows.
2. Safety (Confirmed: skip overlaps):
   - Skip adding for a given day if an identical block already exists.
   - Skip adding for a given day if the new block would overlap existing open/pricing intervals (use `buildIntervals` to compare) and toast a “skipped days” message.
3. Update UX text:
   - Change toast/message to reflect “Copied block to other days” (not “Applied to all open blocks”).
   - Optionally adjust the button label from “Apply to all” -> “Copy to all” for clarity.

### 6) Schedule end time blank when revisiting (12:00AM / end-of-day)
Root cause:
- `toTimeString(1440)` currently returns `"24:00"`, which HTML `input[type=time]` typically rejects and renders blank.

Changes:
1. Update `toTimeString` in `src/features/owner/components/court-schedule-editor/helpers.ts`:
   - Map `1440 -> "00:00"` so end-of-day renders as a valid time input value.
2. Apply the same normalization in other editors that format minutes to time strings:
   - `src/features/owner/components/court-hours-editor.tsx`
   - `src/features/owner/components/court-pricing-editor.tsx`
