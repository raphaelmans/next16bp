# Plan: Fix Owner Get-Started Shell Mismatch

## Goal
Ensure `/owner/get-started` always renders the onboarding chrome and never falls back to `PlayerShell`, regardless of how the user arrives there.

## Root Cause
- `src/app/(auth)/layout.tsx` currently chooses shell UI by checking request pathname (`x-pathname`).
- In Next.js App Router, layouts are cached and do not reliably re-render on every client navigation, so pathname-based branching inside a shared layout can become stale.
- The current exact check (`pathname === "/owner/get-started"`) plus default `PlayerShell` fallback makes `/owner/get-started` vulnerable to rendering with the wrong shell.

## Critical Files To Modify
- `src/app/(auth)/layout.tsx`
- `src/app/(auth)/owner/get-started/page.tsx` (move)
- `src/app/(auth)/owner/onboarding/page.tsx` (move for consistency)
- `src/app/(owner-onboarding)/layout.tsx` (new)
- `src/app/(owner-onboarding)/owner/get-started/page.tsx` (new location)
- `src/app/(owner-onboarding)/owner/onboarding/page.tsx` (new location)

## Recommended Approach
1. Create a dedicated route group at `src/app/(owner-onboarding)` for onboarding-only owner routes.
2. Add `src/app/(owner-onboarding)/layout.tsx` that:
   - reads pathname for auth guard only (`requireSession(pathname)`), and
   - always returns `<OnboardingShell>{children}</OnboardingShell>` with no route-based shell switching.
3. Move `src/app/(auth)/owner/get-started/page.tsx` to `src/app/(owner-onboarding)/owner/get-started/page.tsx` so the URL stays `/owner/get-started` but layout ownership is deterministic.
4. Move `src/app/(auth)/owner/onboarding/page.tsx` to `src/app/(owner-onboarding)/owner/onboarding/page.tsx` so the redirect entrypoint shares the same onboarding layout.
5. Simplify `src/app/(auth)/layout.tsx` by removing onboarding-specific branching/imports, leaving it responsible for guest/public/player behavior only.

## Verification
1. Run `pnpm lint` from repo root.
2. In `pnpm dev`, validate end-to-end:
   - Hard refresh on `/owner/get-started` shows onboarding shell (no player sidebar/navbar).
   - Client navigation from `/home` (or portal switch) to `/owner/get-started` still shows onboarding shell.
   - Visiting `/owner/onboarding` redirects to `/owner/get-started` and remains in onboarding shell.
   - Logged-out request to `/owner/get-started` redirects to login and returns correctly after auth.
   - Optional: `/owner/get-started/` (trailing slash) still resolves to onboarding shell.
3. Regression spot-check: `/home` and other `(auth)` protected pages continue to render `PlayerShell` as before.
