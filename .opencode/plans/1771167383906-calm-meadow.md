# Temporary fix: hide system-message attachment cards in reservation inbox

## Goal
- Remove the attachment tile/icon from system-generated reservation messages for now.
- Keep normal user-uploaded attachments working as-is.

## Recommended implementation
1. Update `src/features/chat/components/chat-thread/stream-chat-thread.tsx` (single-file UI fix).
2. Add a small helper in this file:
   - `isSystemReservationMessage(msg: LocalMessage): boolean`
   - Detect system reservation messages via deterministic message ID suffixes:
     - `:player-created:v1`
     - `:player-payment-marked:v1`
     - `:owner-confirmed:v1`
3. In the message render loop, keep existing attachment mapping, but gate rendering:
   - From: `attachments.length > 0`
   - To: `attachments.length > 0 && !isSystemReservationMessage(msg)`
4. Do not change backend message creation or provider schema in this pass.

## Why this approach
- Lowest-risk and fast: presentation-only change in one component.
- Avoids touching message contracts/ops and avoids migration concerns.
- Stable detection via message IDs (safer than text matching).

## Critical files
- `src/features/chat/components/chat-thread/stream-chat-thread.tsx`
- (Referenced only; no edits planned)
  - `src/lib/modules/chat/ops/post-player-created-message.ts`
  - `src/lib/modules/chat/ops/post-player-payment-marked-message.ts`
  - `src/lib/modules/chat/ops/post-owner-confirmed-message.ts`

## Verification
1. Run `pnpm lint`.
2. Manual end-to-end in app (`pnpm dev`):
   - Create reservation (player-created system message appears): no attachment tile.
   - Mark payment (player-payment-marked system message appears): no attachment tile.
   - Confirm reservation/payment (owner-confirmed system message appears): no attachment tile.
   - Send a normal user message with uploaded file/image: attachment still renders.
   - Send a normal text-only message: unchanged behavior.

## Notes
- This is intentionally temporary. If later you want payment messages to show structured attached data, we can implement a dedicated message metadata/attachment model and render path in a follow-up.
