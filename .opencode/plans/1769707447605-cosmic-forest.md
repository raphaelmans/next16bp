## Requirements Capture: Desktop Block Palette Removal + Selection Panel

### Objective
Capture requirements only (no implementation plan). The Block Palette sidebar should be removed for desktop and replaced with the same selection workflow used on mobile, but embedded in the existing sidebar rather than a drawer. Also improve selection affordances on the timeline/grid to make selectable slots more obvious.

### Requirements
1. **Remove Block Palette on desktop (both week + day views)**
   - Desktop left sidebar should no longer show “Block Palette.”
   - Applies to both week and day views.

2. **Repurpose the left sidebar as the selection panel**
   - When no range is selected: show an instructional empty state (e.g., “Select time slots to create a block”).
   - When a range is selected: show the same form fields + behavior as the mobile drawer (block type toggle, time range summary, notes/guest form, save button).
   - Uses the same mutations/validation behavior as the mobile drawer (walk-in, maintenance, guest booking).

3. **Improve selection affordance on slots (desktop)**
   - Provide visible cues that time slots are selectable.
   - Use a hover + cursor + soft highlight treatment (subtle hover background, pointer cursor on available rows/cells).
   - Add a small hint near the timeline (e.g., “Click a start time, then click an end time to select a range”).

4. **Avoid duplicate “Today” button on mobile**
   - Keep only one “Today” button visible on mobile (the one in the mobile date header).
   - Hide the filter card “Today” button on mobile.

### Reference Context (current code locations)
- Block Palette (week/day sidebar): `src/app/(owner)/owner/bookings/page.tsx` around the left sidebar cards.
- Mobile drawer form logic to mirror in desktop sidebar: `src/app/(owner)/owner/bookings/page.tsx` mobile drawer section.
- Selection state: `mobileCommittedRange` + related helpers in `src/app/(owner)/owner/bookings/page.tsx`.

### Non-Goals
- No backend or schema changes.
- No behavior changes for mobile drawer itself (only reuse its UX for desktop sidebar).

## Requirements Capture: Import Walk-in -> Guest Replacement UI

### Context
The booking import pipeline (see `docs/ai-normalization-testing/README.md`) normalizes rows into blocks. You want import to remain deterministic (import always creates walk-in-style blocks), but provide a UI in the import context to replace those imported blocks with guest bookings so the schedule shows guest names.

### Requirements
1. **Import remains deterministic**
   - Import commit continues creating non-guest blocks (walk-in-style) by default.
   - No requirement to infer guest identity during normalization/commit.

2. **Replace an imported block with a guest booking (per-row action)**
   - When viewing the availability studio in import mode (i.e., `jobId` import overlay context), each imported walk-in block should expose an action: “Replace with guest booking”.
   - Action opens a guest booking UI (existing guest selector + create new guest) prefilled with the imported row’s `reason/title` as the suggested guest name.
   - On confirm:
     - The original imported block is removed/cancelled.
     - A guest booking reservation is created for the same court/time range.
     - UI shows the guest name on the schedule after replacement.

3. **Association + audit**
   - The system must retain a durable link between the import row and the created guest booking (e.g., store `reservationId` and `guestProfileId` for that row), so the UI can:
     - Display which rows were replaced.
     - Re-open the guest details.
     - Avoid duplicate replacements on refresh.

4. **Bulk/UX affordance (optional requirement)**
   - In import overlay mode, allow quickly stepping through imported rows/blocks to replace them one-by-one.
   - Clearly label imported items (e.g., “Imported”) until replaced.

5. **Conflict handling**
   - Replacement must be safe when overlaps exist.
   - If the guest booking cannot be created (validation/overlap), the original block should not be lost (i.e., no destructive action without a successful replacement).

### Non-Goals
- No changes to the AI normalization tiers themselves.
- No requirement to automatically match guests from the import file.
