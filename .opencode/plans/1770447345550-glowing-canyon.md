# Plan: Reservation Chat Payment-Submitted System Message

## Goal

- Add a new idempotent chat system message when a player submits payment (`AWAITING_PAYMENT -> PAYMENT_MARKED_BY_USER`).
- Keep current reservation/payment/notification behavior unchanged.
- Update reservation-chat user stories so docs match runtime behavior.

## Implementation Plan

1. Add a new chat op for payment-submitted system messages.
   - Create `src/lib/modules/chat/ops/post-player-payment-marked-message.ts` following the same pattern as:
     - `src/lib/modules/chat/ops/post-player-created-message.ts`
     - `src/lib/modules/chat/ops/post-owner-confirmed-message.ts`
   - Include:
     - `buildPlayerPaymentMarkedMessage(reservationId)`
     - `makePlayerPaymentMarkedMessageId(reservationId)`
     - `postPlayerPaymentMarkedMessage({ reservationId, playerUserId, ownerUserId, messageText? })`
   - Use deterministic message id for idempotency: `reservation:${reservationId}:player-payment-marked:v1`.

2. Wire payment-submitted chat posting into reservation payment flow.
   - Update `src/lib/modules/reservation/services/reservation.service.ts`.
   - Import the new op.
   - Add a best-effort helper similar to `postPlayerCreatedMessageBestEffort`, e.g. `postPlayerPaymentMarkedMessageBestEffort`:
     - Resolve `playerUserId` from `profileRepository.findById(profileId)`.
     - Resolve `ownerUserId` from `organizationRepository.findById(organizationId)`.
     - No-op if IDs are missing.
     - Catch/log failures without affecting payment success.
   - In `markPayment(...)`, keep DB state transition and outbox enqueue logic as-is, then post chat message after transaction commit (same post-transaction pattern used by owner-confirmed chat seeding).
   - Guard chat call with `organizationId` presence.

3. Keep message content aligned with existing system-message tone.
   - Prefix with `[SYSTEM GENERATED]`.
   - Mention that the player submitted payment and owner should verify.
   - Include reservation deep link via `appRoutes.reservations.detail(reservationId)` + existing app URL builder pattern.

4. Update reservation-chat stories/docs to include this behavior.
   - `agent-plans/user-stories/67-reservation-chat/67-00-overview.md`
     - Add payment-submitted system message to the shipped behavior summary.
   - `agent-plans/user-stories/67-reservation-chat/67-01-player-messages-venue-from-reservation.md`
     - Add AC that payment submission seeds one system-generated message.
     - Add AC that retries/reloads do not duplicate it.
   - `agent-plans/user-stories/67-reservation-chat/67-04-player-chat-widget-on-reservations-switches-threads.md`
     - Add AC that player sees payment-submitted system message in thread history.
   - `agent-plans/user-stories/67-reservation-chat/67-10-chat-support-diagram.md`
     - Update nuance/annotation that payment-marked transition also seeds a system message once.

## Verification

1. Static checks
   - Run `pnpm lint`.

2. Functional checks (manual)
   - Create a paid reservation and move it to `AWAITING_PAYMENT`.
   - As player, submit payment (calls `markPayment`).
   - Verify thread now contains exactly one new `[SYSTEM GENERATED]` payment-submitted message.
   - Verify both player and owner see the same message in their reservation thread/inbox.

3. Idempotency and regression
   - Reload/open chat multiple times and verify no duplicate payment-submitted system message appears.
   - Confirm existing seeded messages still behave correctly:
     - player-created message posts once
     - owner-confirmed message posts once
   - If chat posting fails, payment submission should still succeed (best-effort behavior).
