# Fix owner "Get started" visibility across multiple venues

## Problem and current requirement

- `Get started` in both sidebar and dashboard currently depends on `setupStatus.isSetupComplete`.
- `isSetupComplete` is calculated from the **primary (latest created) place** only in `src/lib/modules/owner-setup/use-cases/get-owner-setup-status.use-case.ts`.
- If the latest place is incomplete, `isSetupComplete` becomes false even when another place is already fully configured.
- Expected behavior (confirmed): hide `Get started` when the owner has **at least one configured venue**, and apply this to both sidebar and dashboard CTA.

## Recommended implementation

### 1) Add an aggregate setup flag for nav-level visibility

Add a new additive field to owner setup status:

- `hasAnyConfiguredVenue: boolean`

Definition:

- `true` when **any** place in the current org is `VERIFIED` and has at least one active court with both schedule windows and pricing rules.

Keep existing primary-place fields and `isSetupComplete` semantics unchanged to avoid changing onboarding-step behavior on `/owner/get-started`.

Files:

- `src/lib/modules/owner-setup/dtos/owner-setup-status.dto.ts`

### 2) Compute aggregate readiness across all places in owner setup use case

In `GetOwnerSetupStatusUseCase.execute`:

- Keep existing primary-place computation (`primaryPlace`, `verificationStatus`, `hasReadyCourt`, `isSetupComplete`, `nextStep`).
- Add aggregate calculation for `hasAnyConfiguredVenue`:
  - Load all places with verification (already available).
  - Fetch courts for each place via `courtRepository.findByPlaceId` (parallelized with `Promise.all`).
  - Keep active courts only and build `placeId -> activeCourtIds` map.
  - Query hours and pricing once for all active court ids via existing `findByCourtIds` methods.
  - For each place, check if it is `VERIFIED` and has at least one active court that appears in both sets (`hours` and `pricing`).
  - Set `hasAnyConfiguredVenue = true` if any place satisfies the condition.
- Return `hasAnyConfiguredVenue` in both:
  - early return branch (no org)
  - normal return payload

Files:

- `src/lib/modules/owner-setup/use-cases/get-owner-setup-status.use-case.ts`

### 3) Switch CTA visibility logic to the aggregate flag

Update both UI entry points:

- Sidebar `shouldShowGetStarted`: show only when setup query is loaded and `!setupStatus.hasAnyConfiguredVenue`.
- Dashboard `showSetupCta`: show only when setup query is loaded and `!setupStatus.hasAnyConfiguredVenue`.

Files:

- `src/features/owner/components/owner-sidebar.tsx`
- `src/app/(owner)/owner/page.tsx`

### 4) Keep onboarding page behavior stable

No change to `/owner/get-started` step card logic unless explicitly requested later. It can continue to rely on primary-place progression (`isSetupComplete`, `nextStep`, etc.).

File to validate only:

- `src/app/(auth)/owner/get-started/page.tsx`

## Verification plan

### Manual scenarios

1. One configured venue + one incomplete venue (incomplete is newest):
   - Sidebar hides `Get started`.
   - Dashboard setup CTA hides `Get started`.
2. One incomplete venue only:
   - Sidebar and dashboard both show `Get started`.
3. Venue verified but no ready court (missing schedule or pricing):
   - Sidebar and dashboard show `Get started`.
4. No org or no venue:
   - Sidebar/dashboard show `Get started` as expected.
5. Direct visit to `/owner/get-started`:
   - Existing onboarding step behavior remains consistent with primary-place logic.
