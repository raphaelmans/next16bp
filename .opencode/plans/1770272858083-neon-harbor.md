# Owner Reservations: Show Created-At + TTL Expiry

## Problem
On `http://localhost:3000/owner/reservations`, the reservations list does not show:
- when a reservation was created (`createdAt`)
- whether it is already expired due to TTL (`expiresAt`), especially when backend status has not yet flipped to `EXPIRED`

This leads to confusion in the Inbox/Upcoming/Past/Cancelled tabs (Radix `TabsProvider` shows the tab UI, but the missing indicators are within the tab content rows).

## Goals
- Show `Created <date time>` for each reservation row/card.
- Show a clear TTL indicator using `expiresAt`:
  - `Expires <date time>` (or `Expires in <...>`) when still valid
  - `Expired` when TTL has elapsed
- Ensure the backend is the source of truth for TTL expiry in lists (so UI grouping/counts and actions are consistent without client-side hacks).

## Key Data (already available)
- Owner list query returns ISO strings:
  - `Reservation.createdAt: string`
  - `Reservation.expiresAt?: string | null`
  - `Reservation.reservationStatus: CREATED | AWAITING_PAYMENT | PAYMENT_MARKED_BY_USER | CONFIRMED | EXPIRED | CANCELLED`

Relevant files:
- UI list rows: `src/features/owner/components/reservations-table.tsx`
- Tabs + grouping/counts: `src/app/(owner)/owner/reservations/page.tsx`
- Data mapping: `src/features/owner/hooks.ts` (`useOwnerReservations`)

## Implementation Plan

### 0) Backend: expire stale reservations in a use-case (source of truth)
Add a reservation module use-case that transitions TTL-expired reservations to `EXPIRED` (and writes an audit event), then call it from the owner list endpoint.

Why:
- avoids the frontend needing to “pretend” a reservation is expired
- ensures tabs/counts and actions behave consistently across all owner views
- matches your existing cron logic (same rules) but runs on-demand as well

Proposed:
- Add `src/lib/modules/reservation/use-cases/expire-stale-reservations.use-case.ts`
  - Select reservations where `expiresAt < now` and `status IN (CREATED, AWAITING_PAYMENT, PAYMENT_MARKED_BY_USER)`.
  - Scope by organization by joining `reservation -> court -> place` and filtering `place.organizationId = <orgId>`.
  - In a single transaction:
    - Update those reservations to `status = EXPIRED`, `updatedAt = now`.
    - Insert one `reservationEvent` per expired reservation (`fromStatus`, `toStatus=EXPIRED`, `triggeredByRole=SYSTEM`, `notes=...`).
- Wire the use-case via `src/lib/modules/reservation/factories/reservation.factory.ts` (e.g. `makeExpireStaleReservationsUseCase`).
- Call it from `ReservationOwnerService.getForOrganization(...)` after verifying org ownership and before fetching the list.

Optional follow-up (nice cleanup): refactor `src/app/api/cron/expire-reservations/route.ts` to call the same use-case (global scope) to avoid duplicated logic.

### 1) Add created-at + TTL display in the list UI
Edit `src/features/owner/components/reservations-table.tsx`.

- Add safe ISO parsing helpers inside the module:
  - `parseIsoDate(value?: string | null): Date | null`
  - `getIsTtlStatus(status: Reservation["reservationStatus"]): boolean` (true for `CREATED | AWAITING_PAYMENT | PAYMENT_MARKED_BY_USER`)
- Derive per-reservation timestamps:
  - `createdAtDate = parseIsoDate(reservation.createdAt)`
  - `expiresAtDate = parseIsoDate(reservation.expiresAt)`
- Desktop table UI:
  - In the existing Date/Time cell, add a 3rd muted line:
    - `Created {format(createdAtDate, "MMM d, h:mm a")}` (fallback `--`)
  - In the Stage cell, render the badge plus a small TTL line beneath it (only when `expiresAt` exists and status is TTL-relevant):
    - If `reservation.reservationStatus === "EXPIRED"`: `Expired (TTL)`
    - Else: `Expires {format(expiresAtDate, "MMM d, h:mm a")}`
- Mobile card UI:
  - Add `Created ...` under the date/time line.
  - Add TTL line near/under the stage badge (same rules as desktop).

### 2) Actions + grouping become consistent automatically
Once the backend returns `status = EXPIRED` for TTL-expired records (because it actually expired them), the existing owner UI logic naturally:
- moves them out of `pending` groups
- includes them in `cancelled`
- avoids showing Accept/Confirm actions for them

## Formatting Details
- Use `date-fns/format` (already used) for absolute timestamps.
- If we add relative labels, use `date-fns/formatDistanceToNowStrict` (keeps output short, e.g. `5m`) or reuse the existing `differenceInSeconds` pattern.
- Keep output in muted text (`text-xs text-muted-foreground`) to avoid clutter.

## Edge Cases
- `createdAt` may be empty string; handle gracefully.
- `expiresAt` may be null; show no TTL line.
- Time zone: timestamps are ISO strings; formatting uses browser local time (consistent with existing `formatTime()` behavior in `useOwnerReservations`).
