## Goal
Document where the global top "loading strip" (navigation progress bar) is mounted, and what to change if we want different behavior.

User intent confirmed: route transitions only (no React Query/tRPC fetch tracking).

## Current Implementation (Already Global)
- The strip is implemented as a client component: `src/components/navigation/navigation-progress.tsx`.
- It is mounted globally in `src/common/providers/index.tsx` as `<NavigationProgress />`.
- `src/app/layout.tsx` wraps the whole app with `<Providers>{children}</Providers>`, so the strip appears across all routes.
- The shimmer animation used by the strip is defined in `src/app/globals.css` as `@keyframes courtline-scan`.

## How The Strip Decides When To Show
`src/components/navigation/navigation-progress.tsx`:
- Starts when it detects an in-app navigation attempt:
  - Captures document-level `click` events and looks for an `<a href>`.
  - Ignores modified clicks, external origins, hash/mailto/tel, downloads, and same-path clicks.
  - Also hooks `window.history.pushState` / `replaceState` and listens to `popstate`.
- Finishes when navigation settles by watching `usePathname()` changes.
- Sets `aria-busy="true"` on `#main-content` while active (if such an element exists).

Notes:
- It intentionally waits `START_DELAY_MS` (currently 150ms) before becoming visible; very fast navigations can complete before that, so you might not see the strip even though it worked.
- It is driven by pathname changes; query-only changes are ignored by design.

## Relevant Shells (For aria-busy)
- `src/components/layout/public-shell.tsx` defines `<main id="main-content" ...>`.
- `src/components/layout/app-shell.tsx` defines an element with `id="main-content"`.
- `src/components/layout/onboarding-shell.tsx` does NOT define `id="main-content"` (strip still shows; only aria-busy is missing).

## If You Need To "Add It Globally" (In Another App)
1. Ensure the component exists (or add one): `src/components/navigation/navigation-progress.tsx`.
2. Render it once near the top of your global provider tree: `src/common/providers/index.tsx`.
3. Ensure root layout uses that provider: `src/app/layout.tsx`.
4. Optional accessibility: ensure the main content wrapper includes `id="main-content"` so `aria-busy` toggles during navigation.

## Cross-check vs Next.js Docs (App Router)
- Next.js recommends route-level fallbacks via `loading.tsx` ("Loading UI") for instant navigation feedback and streaming; this is the first-class pattern for navigation loading UI.
  - Ref: `.next-docs/01-app/03-api-reference/03-file-conventions/loading.mdx`
- For immediate feedback on slow networks when prefetch isn't done, Next.js recommends `useLinkStatus` (pending state) for inline indicators and notes you can debounce to avoid flicker.
  - Ref: `.next-docs/01-app/03-api-reference/04-functions/use-link-status.mdx`
  - Ref: `.next-docs/01-app/01-getting-started/04-linking-and-navigating.mdx` ("Slow networks" section mentions progress bar patterns and links to an example)
- App Router does not expose Pages-Router-style `router.events`; Next.js suggests composing `usePathname` / `useSearchParams` in a client component when you need to react to URL changes.
  - Ref: `.next-docs/01-app/03-api-reference/04-functions/use-router.mdx` ("Router events")

Interpretation for this repo:
- A global top strip is not a first-class Next.js API, but implementing it as a single client component mounted in the root provider tree aligns with App Router constraints (no router events) and with the docs' guidance about debouncing to avoid flicker.
- Keep relying on `loading.tsx` for meaningful route-level skeletons; the strip is best treated as supplemental feedback ("click acknowledged") rather than the canonical loading mechanism.
