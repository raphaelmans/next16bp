# Fix forced scroll-to-top on query param updates

## Root Cause
- `src/common/providers/route-scroll-manager.tsx` currently treats `pathname + searchParams` as a route key.
- Query-only URL updates (from `nuqs` state changes) change that key and trigger `window.scrollTo(...)`, so the page jumps to top even when staying on the same pathname.

## Recommended Implementation

### File to modify
- `src/common/providers/route-scroll-manager.tsx`

### Change details
1. Stop using `useSearchParams` in `RouteScrollManager`.
2. Track only `pathname` for route-change detection.
3. Replace `previousRouteRef` with `previousPathnameRef`, initialized from the current pathname.
4. In the scroll effect, run the scroll reset only when pathname actually changes.
5. Keep existing safeguards unchanged:
   - `popstate` skip flag behavior (`skipNextScrollRef`) to avoid fighting browser back/forward restoration.
   - hash guard (`window.location.hash`) so anchor navigation is not overridden.

## Expected Behavior After Fix
- Query param changes on the same page (filters, tabs, pagination URL state): no forced scroll-to-top.
- Pathname changes (actual navigation to another page): still scrolls to top.
- Browser back/forward and hash-anchor behavior remains preserved.

## Verification
1. Run `pnpm dev`.
2. Open a query-state-heavy page (e.g., discovery filters), scroll down, then change filters/pagination; confirm viewport does not jump to top.
3. Navigate to a different pathname; confirm page starts at top.
4. Use browser back/forward; confirm no jarring forced top scroll on popstate.
5. Test a hash link (if available) to confirm anchor behavior still works.
6. Run `pnpm lint`.
