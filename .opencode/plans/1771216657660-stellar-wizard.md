# Desktop first-click adjacent selection bug + light shared core extraction

## Goal
- Fix the desktop-only first-load bug where first slot click auto-selects an adjacent slot.
- Keep existing interaction features intact: drag selection, two-click extension, shift-click extension.
- Do a safe, light extraction of shared range-selection core logic so desktop/week-grid/day-list and mobile/day-list use one core behavior path.

## Scope confirmed
- User-selected scope: **targeted fix + light shared core extraction** (not full architecture migration).

## Root-cause hypothesis (to validate while implementing)
- In `src/components/kudos/range-selection/range-selection-store.ts`, `didDrag` becomes `true` as soon as `pointerEnter` hits a different index.
- On first desktop interaction, UI reflow/animation (summary appearance + layout movement) can cause an adjacent `pointerEnter` during the same pointer sequence.
- That makes `pointerUp` take the drag path and commit a 2-slot range.

## Implementation plan

### 1) Harden drag intent in shared store (bug fix)
- **Modify** `src/components/kudos/range-selection/range-selection-store.ts`
  - Add pointer metadata tracking for active pointer sequence (down position + latest position).
  - Change `didDrag` to require movement threshold (not only index change).
  - Keep click suppression and pointer-up ordering behavior unchanged.
  - Preserve existing two-click and shift-click logic paths.

- **Modify** `src/components/kudos/range-selection/types.ts`
  - Add shared pointer metadata type used by store + adapters.

- **Add** `src/components/kudos/range-selection/core/pointer-intent.ts`
  - Pure helpers:
    - pointer delta calculation
    - drag threshold check (small px threshold, tuned for desktop mouse jitter)

### 2) Pass pointer metadata from both adapters (desktop/mobile UI surfaces)
- **Modify** `src/components/kudos/time-range-picker.tsx`
  - Pass pointer coordinates/type to `pointerDown` and `pointerEnter`.
  - Keep touch-intent hold behavior unchanged.

- **Modify** `src/components/kudos/availability-week-grid.tsx`
  - Pass pointer coordinates/type to `pointerDown` and `pointerEnter`.
  - Keep same-day/contiguous clamping unchanged.

### 3) Light shared-core extraction (facade/adapter direction)
- **Add** `src/components/kudos/range-selection/core/derived-range.ts`
  - Centralize derived range computations currently duplicated across components:
    - active range (anchor/hover + committed fallback)
    - awaiting-end-click flag
    - optional hover preview range derivation

- **Modify** `src/components/kudos/range-selection/use-cell-state.ts`
  - Use shared derived-range helpers (remove inline duplication).

- **Modify** `src/components/kudos/time-range-picker.tsx`
  - Reuse shared derived helpers in summary bar selectors.

- **Modify** `src/components/kudos/availability-week-grid.tsx`
  - Reuse shared derived helpers in summary bar selectors.

- **Modify** `src/components/kudos/range-selection/index.ts`
  - Export new core helpers needed by adapters/facade usage.

## Why this structure
- Bug fix lives in one engine point (`range-selection-store`) used by both desktop and mobile picker UIs.
- Pointer-intent + derived-range pure modules create a reusable core without large component churn.
- Existing adapter surfaces (`TimeRangePicker`, `AvailabilityWeekGrid`) remain intact, minimizing regression risk.

## Regression hotspots to watch
- Tiny drags near threshold should still produce drag selection.
- Click-only selection should not accidentally become drag.
- `pointerup` then `click` ordering should still avoid double commits.
- Week-grid and day-list must remain behaviorally consistent.
- Keyboard interactions (`Enter`/`Space`, shift-click) must remain unchanged.

## Verification plan

### Manual repro + fix checks
- Desktop, fresh page load, no query params:
  - First click on available slot commits exactly one slot.
  - First drag across slots commits contiguous multi-slot range.
- Desktop follow-up behaviors:
  - Two-click extension works.
  - Shift-click extension works.
  - Week view and day view both behave consistently.
- Mobile parity checks:
  - Tap/hold + selection still works.
  - No regression in range selection and clear/reset behavior.
- URL-state checks:
  - Prefilled `startTime` + `duration` still hydrates selected range correctly.
  - Clearing/reselecting keeps expected behavior.
