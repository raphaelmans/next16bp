# Production Email Notification Diagnosis Plan

## Objective
- Confirm the current notification fan-out behavior across `EMAIL`, `SMS`, and `WEB_PUSH`.
- Diagnose why email notifications are not being received in production for unknown/multiple event types.
- Deliver an ASCII diagram of the current end-to-end flow.

## What Is Confirmed In Code
- Fan-out is implemented per event as independent channel branches in `src/lib/modules/notification-delivery/services/notification-delivery.service.ts`.
- For key owner/admin events (`reservation.created`, `place_verification.*`, `claim_request.*`), jobs are conditionally enqueued per channel:
  - `EMAIL` when `NOTIFICATION_EMAIL_ENABLED !== false` and recipient email exists.
  - `SMS` when `NOTIFICATION_SMS_ENABLED !== false` and normalized phone exists.
  - `WEB_PUSH` when `NOTIFICATION_WEB_PUSH_ENABLED !== false` and active push subscriptions exist.
- Important nuance: several reservation lifecycle events (`reservation.awaiting_payment`, `reservation.payment_marked`, `reservation.confirmed`, `reservation.rejected`, `reservation.cancelled`) are currently web-push only in this service.

## Recommended Approach (Single Path)
- Use a production-safe trace-and-compare investigation (no code changes first):
  1) inspect real outbox + logs for a known affected window,
  2) run one controlled admin enqueue+dispatch test in production,
  3) compare channel outcomes (`EMAIL`/`SMS`/`WEB_PUSH`) to isolate failure stage.

## Diagnosis Checklist (Ordered)
1. **Pick a trace window**
   - Select one impacted user and a narrow time window.
   - Use identifiers when available (`reservationId`, verification/claim request id, target email).

2. **Verify dispatch entrypoint health**
   - Confirm cron route path exists and is scheduled (`vercel.json`).
   - Confirm cron auth behavior in production (`CRON_SECRET` required by `verifyCronAuth`).

3. **Inspect outbox records for the window**
   - Review `notification_delivery_job` by `eventType`, `channel`, `status`, `attemptCount`, `lastError`, `nextAttemptAt`, `providerMessageId`.
   - Decision split:
     - No `EMAIL` jobs: enqueue/recipient/channel gating issue.
     - `EMAIL` jobs present but not `SENT`: dispatch/provider issue.

4. **Interpret failure modes quickly**
   - `SKIPPED:DISABLED_CHANNEL:EMAIL` -> email channel toggle disabled.
   - `SKIPPED:MISSING_TARGET` -> recipient email missing at enqueue or invalid target in job.
   - `FAILED:*` -> provider/runtime failure (Resend auth/sender/domain/etc.) with retry/backoff.
   - Stuck `PENDING` -> dispatch not running/reaching route.

5. **Validate recipient resolution source data**
   - Owner path: `organization_profile.contactEmail` fallback to owner `profile.email`.
   - Admin path: `profile.email` for users with role `admin`.

6. **Run controlled production test**
   - Use `/admin/tools/notification-test` to enqueue test jobs and `Dispatch now`.
   - Compare immediate response + resulting outbox statuses.

7. **Conclude with channel matrix**
   - For each channel: expected? enqueued? sent? provider id/error?
   - Distinguish "not expected for this event" vs "expected but failing".

## Most Likely Causes For Missing Email In Production
- `NOTIFICATION_EMAIL_ENABLED` is effectively false.
- Recipient email is absent for the resolved target (owner/admin).
- Dispatch route blocked in prod by cron auth (`CRON_SECRET` missing/mismatch) or cron not executing.
- Email provider failure (Resend credentials/sender/domain/recipient rejection), resulting in `FAILED` with retries.
- Event path expected by user is currently web-push only (not designed to enqueue email).
