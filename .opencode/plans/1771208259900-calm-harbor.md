# Browser Notification Self-Test Plan

## Objective
- Add a simple, high-signal browser notification check so a user can confirm: "this browser/device can receive notifications".
- Prioritize local self-test first (as requested), then keep full server push dispatch as an optional deeper check.

## Current State (What likely fails today)
- Web push enable flow in `src/features/notifications/hooks/use-web-push.ts` requires server VAPID config before proceeding, so users cannot run a basic browser-capability check when backend push is not configured.
- UI in `src/features/notifications/components/web-push-settings.tsx` and `src/features/notifications/components/notification-bell.tsx` shows support/config/permission states, but does not provide a direct local "send test notification now" action.
- Existing end-to-end tool already exists at `src/app/(admin)/admin/tools/notification-test/page.tsx`, but this depends on enqueue + dispatch and can mask whether the issue is client/browser vs backend pipeline.

## Context7 References Used
- Source: `/mdn/content` (MDN Web Docs).
- `Notification.requestPermission()` should be called from a user gesture (e.g. button click), and returns `"granted" | "denied" | "default"`.
- `Notification.permission` should be read to drive UI state and recovery guidance.
- Service Worker and Push APIs are secure-context-only (HTTPS, with `http://localhost` treated as secure for local dev).
- Push handling/display should use service worker notification APIs (`push` event + `showNotification`).

## Recommended Implementation

### 1) Extend web-push hook with explicit readiness diagnostics
- File: `src/features/notifications/hooks/use-web-push.ts`.
- Add derived diagnostics state that separates:
  - unsupported browser
  - insecure context (`window.isSecureContext === false`)
  - denied permission
  - server not configured for push
  - ready for local test
- Keep existing enable/disable behavior for real subscriptions, but make diagnostics clearer for troubleshooting.

### 2) Add a local self-test method (no backend dependency)
- File: `src/features/notifications/hooks/use-web-push.ts`.
- Add `sendLocalTestNotification()` that:
  - runs only from user action,
  - checks support + secure context,
  - requests permission if needed,
  - ensures service worker registration exists,
  - calls `ServiceWorkerRegistration.showNotification(...)` with a test title/body/tag/url.
- Return structured errors/messages for UI to display exact failure reason.

### 3) Surface self-test in settings card
- File: `src/features/notifications/components/web-push-settings.tsx`.
- Add a "Send test notification" button and status hint text.
- Disable the button only when local prerequisites are not met; show the specific reason.
- Keep existing enable/disable buttons for actual push subscription management.

### 4) Improve bell popover troubleshooting text
- File: `src/features/notifications/components/notification-bell.tsx`.
- Reuse readiness diagnostics to show why toggle/action is unavailable (especially insecure context and denied permission reset path).
- Keep current behavior and routing unchanged.

### 5) Optional wording update in admin test page
- File: `src/app/(admin)/admin/tools/notification-test/page.tsx`.
- Clarify that admin test is end-to-end backend delivery, while settings self-test verifies browser capability only.

## Critical Files To Modify
- `src/features/notifications/hooks/use-web-push.ts`
- `src/features/notifications/components/web-push-settings.tsx`
- `src/features/notifications/components/notification-bell.tsx`
- `src/app/(admin)/admin/tools/notification-test/page.tsx` (optional copy update)

## Verification (Local + Optional E2E)
1. Static checks
   - Run `pnpm lint`.
2. Local browser capability check (primary)
   - Run `pnpm dev`.
   - Open settings page containing `WebPushSettingsCard`.
   - Click "Send test notification" and confirm OS/browser notification appears.
   - Click notification and verify focus/navigation behavior from `public/sw.js`.
3. Permission/insecure-context edge checks
   - With blocked permission, verify UI explains manual browser reset requirement.
   - In insecure context (non-HTTPS non-localhost), verify clear "secure context required" message.
4. Optional full pipeline check (secondary)
   - Open `/admin/tools/notification-test`.
   - Ensure subscription exists, enqueue Web Push test, then click dispatch.
   - Confirm notification delivery; if this fails while local self-test passes, issue is server pipeline/config.

## Risks / Edge Cases To Handle
- `denied` permission cannot be re-prompted programmatically.
- Browser/OS notification suppression (Focus/Do Not Disturb) can look like delivery failure.
- Service worker scope/registration mismatch can produce false negatives.
- VAPID misconfiguration should not block local self-test, but should still block real subscription flow with explicit messaging.
