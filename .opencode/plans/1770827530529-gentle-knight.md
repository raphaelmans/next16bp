# Fix plan: Admin inbox should see claim chats immediately

## Goal
Ensure claim-request chats appear in admin inbox without requiring an admin to manually open a specific claim first.

## Confirmed root cause
- `owners/get-started` public page is marketing + redirect only; claim submission is not triggered there.
  - `src/app/(public)/owners/get-started/page.tsx`
- `/venues/[placeId]` is a route alias to `/places/[placeId]`, and both venue and owner-onboarding claim flows call the same backend endpoint: `trpc.claimRequest.submitClaim`.
  - `src/app/(public)/venues/[placeId]/page.tsx`
  - `src/features/discovery/place-detail/components/sections/place-detail-listing-help-section.tsx`
  - `src/app/(owner-onboarding)/owner/get-started/page.tsx`
- Claim support chat provisioning is currently lazy in `supportChat.getClaimSession` (only when a chat sheet is opened).
  - `src/lib/modules/chat/services/support-chat.service.ts`
- Owner-triggered session creates channel members as owner-only; admin inbox filters channels by admin membership, so those channels are invisible until an admin joins.
  - `src/lib/modules/chat/services/support-chat.service.ts`
  - `src/features/support-chat/components/support-inbox-widget.tsx`

## User-approved behavior
- Auto-create claim chat on submit with members: owner + all admins.
- Backfill existing pending claim requests so current missing threads also become visible in admin inbox.

## Recommended implementation

1) Centralize claim-thread provisioning in `SupportChatService`
- Add a reusable claim-thread ensure method that:
  - builds `cr-<claimRequestId>` channel id,
  - resolves admin recipients,
  - dedupes members: `[ownerUserId, ...adminUserIds]`,
  - calls `ensureUsers`, `ensureSupportChannel`, and `upsertClaimThread`.
- Reuse this method from `getClaimSession` so lazy path remains a safe fallback.

2) Add eager provisioning for new claim submissions
- In `claimRequestRouter.submitClaim`, after `submitClaimRequest` succeeds, call support-chat claim provisioning.
- Keep claim submission successful even if chat provisioning fails (best-effort post-commit), and log a warning event with `claimRequestId`.

3) Backfill existing pending claim requests (admin-triggered)
- Add admin-only support-chat endpoint (e.g., `backfillClaimThreads`) that iterates pending claim requests and provisions each eligible claim thread idempotently.
- Skip non-eligible records (`requestedByUserId` null) and return a summary (`processed/skipped/failed`) for observability.
- Trigger this backfill from admin support inbox when opened (once per open/session), then refresh channel list.

4) Keep inbox UX aligned with new behavior
- Update support inbox empty-state copy to no longer imply manual â€śopen request chat to join it,â€ť since threads are now auto-provisioned/backfilled.
