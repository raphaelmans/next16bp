# Fix plan: Stream channel created before client connection

## Goal
Eliminate the runtime error `Call connectUser or connectAnonymousUser before creating a channel` in the owner onboarding support chat flow, while keeping the fix low risk and aligned with existing chat patterns.

## Confirmed root cause
- `SupportChatSheet` mounts `StreamChatThread` with `client={client}` before Stream connection is ready.
  - File: `src/features/support-chat/components/support-chat-sheet.tsx`
- `useStreamChannel` calls `client.channel(...)` as soon as `client` and `channelId` exist, with no connection-state guard.
  - File: `src/features/chat/hooks/useStreamChannel.ts`
- The same repo already uses a safe pattern in other chat entry points (`client={isReady ? client : null}`), but `SupportChatSheet` is an outlier.

## Recommended implementation

1. Align `SupportChatSheet` with existing readiness gating
- Update `StreamChatThread` props in `src/features/support-chat/components/support-chat-sheet.tsx`:
  - from `client={client}`
  - to `client={isReady ? client : null}`
- Keep current loading skeleton and `readOnly`/`readOnlyReason` behavior unchanged.

2. Add defensive connection guard inside shared channel hook
- In `src/features/chat/hooks/useStreamChannel.ts`, gate channel creation on actual Stream client connection, not only object existence.
- Concretely, only create channel when all are true:
  - `client` exists
  - `channelId` exists
  - connected user is present on client (e.g. `client.userID` or equivalent connected signal)
- Ensure memo dependencies include the chosen connection signal so channel is created once connection completes.
- Keep existing watch/query/send logic unchanged; when not connected, return `channel = null` so existing “Connecting” UI path remains active.

## Files to modify
- `src/features/support-chat/components/support-chat-sheet.tsx`
- `src/features/chat/hooks/useStreamChannel.ts`

## Verification plan

- Confirm no runtime error is thrown before chat connection completes.
- Confirm thread shows connecting/read-only state, then loads messages once connected.

3. Regression checks
- Open existing chat surfaces that already gate by readiness:
  - support inbox widget
  - reservation inbox widget
  - player reservation chat widget
- Confirm message load/send still works and no new channel/watch errors appear.

## Notes
- This plan intentionally avoids changing `useStreamClient` readiness semantics to keep scope small.
- If any residual race appears after this patch, next step is to tighten `useStreamClient` readiness criteria, but that is not required for the current reported crash.
