# Fix unread badge while chat is closed

## Goal
Ensure the floating chat button shows unread notifications immediately after refresh, even if the inbox sheet has not been opened yet.

## Confirmed root cause
- `channels` initializes as empty in `src/features/chat/components/chat-widget/reservation-inbox-widget.tsx`, and the floating badge (`unreadCount`) is derived from that in-memory array.
- Initial inbox sync is gated by `open`, so after refresh (closed widget) there is no cold-start `queryChannels` hydration.
- Stream event handling currently updates from future `message.new` events, but does not guarantee backfill for already-unread messages at mount time.

## Recommended implementation
1. Add one-time background channel hydration in `src/features/chat/components/chat-widget/reservation-inbox-widget.tsx`.
   - Introduce a ref guard keyed by authenticated user (e.g. `didBackgroundHydrateRef`) so hydration runs once per user session/mount.
   - Add a new `useEffect` that triggers when `isReady`, `client`, and `auth?.user.id` are available, independent of `open`.
   - In that effect, call `fetchChannels.current?.()` (not full `syncInbox`) so unread state is hydrated without loading reservation metas.

2. Keep the existing `open`-gated behavior for full inbox sync.
   - Preserve the current `syncInbox()` path when `open === true` so reservation metadata refresh remains tied to active inbox usage.
   - Keep `getThreadMetas` query `enabled: open && reservationIds.length > 0` to avoid unnecessary closed-state TRPC traffic.

3. Broaden event-triggered refresh conditions for resiliency.
   - In the existing Stream subscription effect, continue handling `message.new`.
   - Also handle `notification.message_new` and `notification.added_to_channel` to ensure badge/channel list updates when membership/messages change while closed.
   - When closed, call `fetchChannels.current?.()`; when open, keep using `syncInbox()`.

4. Prevent avoidable duplicate refresh bursts.
   - Add lightweight in-flight dedupe for channel refresh calls (ref-backed promise) so startup hydration + event refresh + manual refresh do not stack redundant `queryChannels` calls.
