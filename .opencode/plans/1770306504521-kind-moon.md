# Notification Opt-In UI (Bell + Toggle)

## Goal
Make it easy (and hard to miss) for court owners and users to enable browser notifications (Web Push) with a Facebook-like affordance.

Key constraint: browser permission + subscription must be triggered by a user gesture (click/toggle), never on page load.

## Current state (repo)
- Browser notification enable/disable UI exists only on owner settings: `src/features/notifications/components/web-push-settings.tsx` mounted in `src/app/(owner)/owner/settings/page.tsx`.
- There is no global “bell” entry point in headers yet.
- Design system: warm neutrals; Teal is primary action/focus; Orange is highlight (use for subtle notification emphasis dot/badge). Source: `business-contexts/kudoscourts-design-system.md`.

## Recommended UX
### 1) Header bell as the primary entry point
- Add a bell icon button in the header for:
  - Owner dashboard: `src/features/owner/components/owner-navbar.tsx`
  - Player/protected dashboard: `src/components/layout/player-navbar.tsx`
  - (Optional) Admin dashboard: `src/features/admin/components/admin-navbar.tsx`

Behavior:
- If notifications are enabled (`Notification.permission === "granted"` and the device has an active subscription), bell shows normal.
- If not enabled but supported, show a small Orange dot on the bell (subtle: highlight without “error red”).
- Clicking the bell opens a small Popover (w ~320px) that contains:
  - A short explanation (“Get notified when reservations change.”)
  - A toggle or primary CTA:
    - Switch “Browser notifications” (user gesture) OR a primary `Enable` button
  - Secondary link: “Manage in settings” (deep-link to settings section)
- If permission is denied: show “Blocked in browser settings” + instructions.
- If unsupported: show “Not supported on this device/browser.”

Visual language (match design system):
- Keep control neutral-first (icon button, subtle border/radius-md).
- Teal focus ring / active state.
- Orange dot/badge only for the “needs attention” state.

### 2) Settings remain the detailed control surface
- Owner: keep the existing card on `/owner/settings`, but make it a first-class section with an anchor.
- Player: add the same card to `/account/profile` (or create `/account/notifications` if you prefer IA separation).

## Implementation plan
### A) Refactor shared Web Push logic
Create a reusable hook/utility used by both the settings card and the bell popover, to avoid duplicated SW + subscribe/unsubscribe logic.

Add:
- `src/features/notifications/hooks/use-web-push.ts`
  - `supported` detection
  - reads `Notification.permission`
  - `enable()`
    - registers `navigator.serviceWorker.register("/sw.js")`
    - requests permission
    - subscribes using VAPID public key
    - calls `trpc.pushSubscription.upsertMySubscription`
  - `disable()`
    - unsubscribes current `PushSubscription`
    - calls `trpc.pushSubscription.revokeMySubscription({ endpoint })`

Update:
- `src/features/notifications/components/web-push-settings.tsx` to use the hook.

### B) New bell component
Add:
- `src/features/notifications/components/notification-bell.tsx`

Responsibilities:
- Render icon button + Orange dot state.
- Popover content with:
  - Switch or CTA button wired to `useWebPush()`
  - “Manage in settings” link
  - Error/help states (unsupported / denied / not configured)

Dependencies:
- `Popover` from `src/components/ui/popover.tsx`
- `Button`, `Switch`, `Badge` from existing UI kit
- `lucide-react` icons (`Bell`, optional `BellOff`)

### C) Mount bell in navbars

- `src/features/admin/components/admin-navbar.tsx` (optional)

Placement:

### D) Make settings deep-linkable

  - add `browserNotifications` entry to `SETTINGS_SECTION_IDS` + `SETTINGS_SECTION_HASHES`.

Player:

## Edge cases
- Don’t request permission automatically.
- Handle `Notification.permission === "denied"` gracefully; no repeated prompts.
- Handle “configured=false” when VAPID keys are missing.
- Multiple devices/subscriptions are allowed; the UI should reflect “enabled on this device” (based on `pushManager.getSubscription()`).
