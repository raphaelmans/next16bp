# Notification Opt-In UI (Bell + Toggle)

## Goal
Make it easy (and hard to miss) for court owners and users to enable browser notifications (Web Push) with a Facebook-like affordance.

Key constraint: browser permission + subscription must be triggered by a user gesture (click/toggle), never on page load.

## Current state (repo)
- Browser notification enable/disable UI exists only on owner settings: `src/features/notifications/components/web-push-settings.tsx` mounted in `src/app/(owner)/owner/settings/page.tsx`.
- There is no global “bell” entry point in headers yet.
- Design system: warm neutrals; Teal is primary action/focus; Orange is highlight (use for subtle notification emphasis dot/badge). Source: `business-contexts/kudoscourts-design-system.md`.

## Recommended UX
### 1) Header bell as the primary entry point
- Add a bell icon button in the header for:
  - Owner dashboard: `src/features/owner/components/owner-navbar.tsx`
  - Player/protected dashboard: `src/components/layout/player-navbar.tsx`
  - Admin dashboard: `src/features/admin/components/admin-navbar.tsx`

Behavior:
- If notifications are enabled (`Notification.permission === "granted"` and the device has an active subscription), bell shows normal.
- If not enabled but supported, show a small Orange dot on the bell (subtle: highlight without “error red”).
- Clicking the bell opens a small Popover (w ~320px) that contains:
  - A short explanation (“Get notified when reservations change.”)
  - A toggle or primary CTA:
    - Switch “Browser notifications” (user gesture) OR a primary `Enable` button
  - Secondary link: “Manage in settings” (deep-link to settings section)
- If permission is denied: show “Blocked in browser settings” + instructions.
- If unsupported: show “Not supported on this device/browser.”

Visual language (match design system):
- Keep control neutral-first (icon button, subtle border/radius-md).
- Teal focus ring / active state.
- Orange dot/badge only for the “needs attention” state.

### 2) Settings remain the detailed control surface
- Owner: keep the existing card on `/owner/settings`, but make it a first-class section with an anchor.
- Player: add the same card to `/account/profile` (or create `/account/notifications` if you prefer IA separation).

## Implementation plan
### A) Refactor shared Web Push logic
Create a reusable hook/utility used by both the settings card and the bell popover, to avoid duplicated SW + subscribe/unsubscribe logic.

Add:
- `src/features/notifications/hooks/use-web-push.ts`
  - `supported` detection
  - reads `Notification.permission`
  - `enable()`
    - registers `navigator.serviceWorker.register("/sw.js")`
    - requests permission
    - subscribes using VAPID public key
    - calls `trpc.pushSubscription.upsertMySubscription`
  - `disable()`
    - unsubscribes current `PushSubscription`
    - calls `trpc.pushSubscription.revokeMySubscription({ endpoint })`

Update:
- `src/features/notifications/components/web-push-settings.tsx` to use the hook.

### B) New bell component
Add:
- `src/features/notifications/components/notification-bell.tsx`

Responsibilities:
- Render icon button + Orange dot state.
- Popover content with:
  - Switch or CTA button wired to `useWebPush()`
  - “Manage in settings” link
  - Error/help states (unsupported / denied / not configured)

Dependencies:
- `Popover` from `src/components/ui/popover.tsx`
- `Button`, `Switch`, `Badge` from existing UI kit
- `lucide-react` icons (`Bell`, optional `BellOff`)

### C) Mount bell in navbars
Update:
- `src/features/owner/components/owner-navbar.tsx`
- `src/components/layout/player-navbar.tsx`
- `src/features/admin/components/admin-navbar.tsx`

Placement:
- Right side, before user dropdown.

### D) Make settings deep-linkable
Update:
- `src/common/section-hashes.ts`
  - add `browserNotifications` entry to `SETTINGS_SECTION_IDS` + `SETTINGS_SECTION_HASHES`.
- `src/app/(owner)/owner/settings/page.tsx`
  - wrap the notifications card in a section with the new `id`.

Player:
- Update `src/app/(auth)/account/profile/page.tsx` to include the notifications card (likely in the right column under the owner CTA section, or as its own card).

## Edge cases
- Don’t request permission automatically.
- Handle `Notification.permission === "denied"` gracefully; no repeated prompts.
- Handle “configured=false” when VAPID keys are missing.
- Multiple devices/subscriptions are allowed; the UI should reflect “enabled on this device” (based on `pushManager.getSubscription()`).

# Owner↔Admin Support Chat (Claims + Verification)

## Goal
- Enable a request-scoped chat thread between the requesting owner and admins during:
  - `claim_request` review (claim a curated listing / removal)
  - `place_verification_request` review (verification support)
- Access model: any admin can view/reply in any request thread.

## Recommended approach (MVP)
- Reuse Stream Chat (already integrated) and the existing chat UI primitives (`useStreamClient`, `StreamChatThread`).
- Use deterministic Stream channel IDs (no human input):
  - Claim request: `cr-<claimRequestId>`
  - Verification request: `vr-<placeVerificationRequestId>`
  - Channel type: reuse existing `messaging` (avoid requiring Stream dashboard changes).
- Add a small DB mapping table to persist channel metadata and support future inbox/transcripts.
- Admin access: join-on-demand membership.
  - Owners are always members.
  - When an admin opens a thread, the server ensures that admin user is added as a channel member.
  - This avoids needing Stream permission changes while still allowing any admin to participate.

## Data model + migrations
Add a new schema + migration similar to reservation chat:

- `support_chat_thread`
  - `id` uuid pk
  - `claim_request_id` uuid nullable fk -> `claim_request(id)` (unique)
  - `place_verification_request_id` uuid nullable fk -> `place_verification_request(id)` (unique)
  - `provider_id` varchar(20) ("stream")
  - `provider_channel_type` varchar(32) ("messaging")
  - `provider_channel_id` varchar(128) ("cr-..." / "vr-...")
  - `created_by_user_id` uuid fk -> `auth.users(id)` (set null)
  - `created_at` timestamp tz
  - Constraint: exactly one of `claim_request_id` or `place_verification_request_id` is non-null.

Indexes:
- provider channel uniqueness (provider_id + provider_channel_type + provider_channel_id)
- lookup by `claim_request_id` / `place_verification_request_id`

Files:
- Add schema: `src/lib/shared/infra/db/schema/support-chat.ts`
- Export from: `src/lib/shared/infra/db/schema/index.ts`
- Add migration: `drizzle/00xx_support_chat.sql`

## Server implementation
Add a new support chat service + router following the reservation chat pattern.

### Provider helpers
- Add helpers in `src/lib/modules/chat/helpers/`:
  - `claim-channel-id.ts` -> `cr-${claimRequestId}`
  - `verification-channel-id.ts` -> `vr-${requestId}`

### Repository
- `src/lib/modules/chat/repositories/support-chat-thread.repository.ts`
  - `findByClaimRequestId`
  - `findByVerificationRequestId`
  - `upsertClaimThread`
  - `upsertVerificationThread`

### Service
- `src/lib/modules/chat/services/support-chat.service.ts`
  - `getClaimSession(viewerUserId, claimRequestId, viewer)`
  - `getVerificationSession(viewerUserId, requestId, viewer)`
  - Enforce auth:
    - owner: must match `requested_by_user_id`
    - admin: allowed if `ctx.session.role === "admin"`
  - Claim requests with `requestedByUserId = null` (guest removal requests) are not eligible for chat.
  - Ensure Stream channel exists (idempotent):
    - `type: "messaging"`, `id: channelId`
    - members:
      - always include owner requester userId
      - if viewer is admin, include admin viewer userId (join-on-demand)
  - Return a session payload similar to `reservationChat.getSession`:
    - `auth: { apiKey, user, token }`
    - `channel: { channelType, channelId, memberIds }`
    - `meta: { kind, requestId, placeName? }` (optional)

### Router
- `src/lib/modules/chat/support-chat.router.ts`
  - `getClaimSession({ claimRequestId })` (protected)
  - `getVerificationSession({ requestId })` (protected)

### Factory + wiring
- `src/lib/modules/chat/factories/support-chat.factory.ts`
- Register router in `src/lib/shared/infra/trpc/root.ts`

## Client UI
Reuse `StreamChatThread` and present it as a contextual panel/drawer.

Add feature UI:
- `src/features/support-chat/components/support-chat-panel.tsx`
  - Calls `trpc.supportChat.get*Session`
  - Connects Stream via `useStreamClient`
  - Renders `StreamChatThread` with contextual header.

Owner entry points:
- Claim pending card on `src/app/(auth)/owner/get-started/page.tsx`
  - When there is a pending claim request, show a secondary action: “Message admin” → opens support chat panel.
- Verification panel `src/features/owner/components/place-verification-panel.tsx`
  - For `PENDING` + `REJECTED`, show “Message admin” → opens support chat panel.

Admin entry points:
- Claims review page `src/app/(admin)/admin/claims/[id]/page.tsx` (and/or `src/features/admin/components/claim-review-actions.tsx`)
- Verification review page `src/app/(admin)/admin/verification/[requestId]/page.tsx` (and/or `src/features/admin/components/place-verification-review-actions.tsx`)

MVP UX recommendation:
- Use a `Sheet` (right-side drawer) for the support chat panel to avoid disrupting review flows.

## Notifications for new support messages
MVP:
- No email/SMS/Web Push on chat messages yet (messages are sent client→Stream; server has no event).

Phase 2:
- Add a Stream webhook endpoint (`/api/webhooks/stream-chat`) to ingest `message.new` events.
- Translate into notification outbox jobs (`notification_delivery_job`) to notify:
  - owner when admin replies
  - admin group when owner sends a message
- Add idempotency using Stream message id/event id + recipient + channel.

## Also: Admin notification bell
- Mount the existing bell popover in `src/features/admin/components/admin-navbar.tsx`.
- Use the same “manage in settings” link target as player (`/account/profile#browser-notifications`) since Web Push is per-user, not per-role.

## Verification

- Admin: open the same request in admin portal, open chat, confirm history visible, reply.
- Confirm non-requesting owners cannot access other owners’ threads.
- Confirm any admin can open any request thread.

# Admin Global Support Chat Widget

## Problem
Support chat is currently only reachable from specific request detail pages. Admins want a persistent way to access ongoing owner conversations from anywhere in the admin portal.

## Goal
- Add a floating “messages” widget (consistent with `ReservationInboxWidget`) that appears on all admin routes.
- It should show support chat threads the current admin is a member of (Stream channel membership).

## MVP approach
### 1) Add a dedicated inbox widget for support threads
Add:
- `src/features/support-chat/components/support-inbox-widget.tsx`

Behavior (mirror `src/features/chat/components/chat-widget/reservation-inbox-widget.tsx`):
- Fixed button bottom-right with unread badge.
- On open, query Stream channels for current user (`members: { $in: [myUserId] }`).
- Filter to support channels by id prefix:
  - Claim: `cr-<claimRequestId>`
  - Verification: `vr-<placeVerificationRequestId>`
- List shows:
  - Kind pill (CLAIM / VERIFICATION)
  - Request id short code (e.g. `CR-xxxxxxxx`)
  - last message preview + per-thread unread count
- Selecting a row opens `StreamChatThread` for that channel.
- Refresh on `message.new` events.
- Persist open state + active thread in `localStorage` (separate keys for admin).

Auth:
- Use `trpc.chat.getAuth` + `useStreamClient` (same as reservation inbox).

### 2) Mount it across admin routes
Update:
- `src/app/(admin)/layout.tsx`

Render:
- `{children}`
- `<SupportInboxWidget />` (client component) so it is visible everywhere.

## Notes / limitation
- The inbox lists only threads the admin has joined (membership). Joining happens when the admin opens a request’s support chat at least once.
- Phase 2 (optional): “All-admin inbox” requires webhook ingestion + server-side thread indexing (or auto-adding all admins as members), which has cost/scale implications.

## Decision
- Admin inbox scope: joined threads only.

## Verification
- Login as admin.
- Open any admin route; confirm floating widget is visible.
- Open a claim/verification support chat once; confirm it appears in the inbox widget list.
- Send a message as owner; confirm unread badge increments for admin.
