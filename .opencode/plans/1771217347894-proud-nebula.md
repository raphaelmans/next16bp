# Plan: Thorough Workshop Re-Review of Notification Docs

## Goal
Make `docs/notification-system` workshop-ready for product/sales while staying accurate to current implementation, with complete coverage of all business notification types. Per user direction, exclude `test.web_push` from workshop docs.

## Current gaps found in review
- Event coverage drift: docs still emphasize only the 4 original MVP events, while code also supports reservation lifecycle events (`reservation.awaiting_payment`, `reservation.payment_marked`, `reservation.confirmed`, `reservation.rejected`, `reservation.cancelled`).
- Recipient coverage drift: docs mostly mention admins + owners, but player recipients exist for reservation lifecycle web push paths.
- Guarantee wording drift: some docs imply all enqueue paths are transaction-coupled, but at least some reservation flows enqueue best-effort (warn and continue on failure).
- Channel inconsistency: ASCII doc still states EMAIL/SMS only and omits WEB_PUSH.
- Chat wording narrowness: overview references `notification.message_new`, while client behavior also listens to `message.new` and `notification.added_to_channel` on some surfaces.
- Workshop navigation friction: index docs rely on code-formatted paths instead of clickable markdown links.
- Ops wording drift: docs claim Semaphore `Retry-After` handling, but current adapter does not implement explicit header-based retry behavior.
