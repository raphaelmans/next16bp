# OTP Auth (Email Code) Plan

## Goal
Add an email one-time-password (OTP) sign-in flow (code entry) as a more reliable alternative to magic links, especially to avoid email link prefetching/token consumption on mobile device previews.

## Reference Implementation Review (Ample)
File: `/Users/raphaelm/Documents/Work/ample/src/features/auth/components/auth-otp-form.tsx`

Product/user flow implemented there:
- Single-screen verification step (not the “send code” step).
- Inputs:
  - `email` (optional UI via `showEmail`; default on)
  - `code` (6-digit, via shadcn `InputOTP`)
- Action: `Verify` submits `{ email, code }` to `trpc.auth.verifyOtpEmailCode`.
- Loading: disables submit + changes label to “Verifying...”.
- Error: shows a generic inline message (“Verification failed...”).
- Success: shows a success message, but immediately `router.push(appRoutes.welcome.base)`.
- Analytics: on success, `mixpanel.identify(email)` (production only).

Notable UX gaps (worth fixing in our implementation):
- No “request/send code” step and no resend/cooldown.
- No hint about where the code was sent, spam folder, or expiration.
- No support for redirecting back to an intended destination.
- No “change email” affordance when code entry is shown.

## Recommended OTP Product Flow (This Repo)
Route: add a guest page like `/otp` (or rename `/magic-link` to `/otp`, see Decision).

Two-step UX:
1) Request code
   - Input email
   - CTA: “Send code”
   - Success state: “We emailed you a 6-digit code” + show code input
   - Resend with cooldown (e.g., 60s)
   - “Back to password sign-in”

2) Verify code
   - 6-digit code input (shadcn `InputOTP`)
   - CTA: “Verify & sign in”
   - On success: redirect to safe `redirect` param (same logic as login/magic link) and `router.refresh()`

## Supabase Configuration Requirement
Supabase email OTP uses the same `signInWithOtp()` call but depends on the *email template*.

Action (Dashboard): Auth -> Templates -> “Magic Link” template
- Replace link content with `{{ .Token }}` (OTP code) per Supabase docs.
- This prevents link prefetchers from consuming a confirmation URL.

## Backend Changes (tRPC + Service/Repo)
Add DTOs:
- `src/lib/modules/auth/dtos/email-otp.dto.ts`
  - `RequestEmailOtpSchema`: `{ email, redirect? }`
  - `VerifyEmailOtpSchema`: `{ email, token, redirect? }` (token is the 6-digit code)

Extend repo/service:
- `src/lib/modules/auth/repositories/auth.repository.ts`
  - Add `requestEmailOtp(email, shouldCreateUser)` -> `client.auth.signInWithOtp({ email, options: { shouldCreateUser } })`
  - Add `verifyEmailOtp(email, token)` -> `client.auth.verifyOtp({ email, token, type: "email" })`

- `src/lib/modules/auth/services/auth.service.ts`
  - Add `requestEmailOtpCode(email)` + business event logging (`user.email_otp_requested`).
  - Add `verifyEmailOtpCode(email, token)` + logging (`user.email_otp_verified`).
  - Keep existing magic link flow intact.

Add router procedures:
- `src/lib/modules/auth/auth.router.ts`
  - `requestEmailOtp` (public mutation)
  - `verifyEmailOtp` (public mutation)

Notes:
- Verification should happen on the server (tRPC) so session cookies are written immediately.
- Consider using `rateLimitedProcedure("tier")` (existing middleware) on `requestEmailOtp` to reduce abuse.
