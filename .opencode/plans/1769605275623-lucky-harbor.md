# Multi-File Bookings Import (Up To 3 Files)

## Intent
Enable owners to import/commit roughly ~3 months of existing bookings in one go by allowing **up to 3 uploaded files per import job**, in **any mix** of:
- `.ics` calendar export
- `.csv`
- `.xlsx`
- `.png/.jpg` calendar screenshots

Normalization should merge rows across all uploaded files into a single review/commit flow.

## AI Usage Policy (Confirmed)
You want to keep **AI normalization once per venue** (place) and allow AI mode for **any** import files (CSV/XLSX/ICS/images).

Implication:
- For **images**, AI is still required.
- For **CSV/XLSX/ICS**, AI mode should be meaningfully different than deterministic mode (today it is not), so we will implement an AI-assisted mapping step.

## Recommended Approach

### 1) Data model: represent multiple uploaded files per job
Add a new table to represent each uploaded file as a â€śsourceâ€ť attached to a job.

- New: `src/shared/infra/db/schema/bookings-import-source.ts`
  - `id` (uuid, pk)
  - `jobId` (fk -> `bookings_import_job.id`, cascade)
  - `sourceType` (`ics|csv|xlsx|image`)
  - `fileName`, `fileSize`, `filePath`
  - `sortOrder` (1..3)
  - `metadata` (jsonb, optional)
  - timestamps

Add provenance to rows so review UI can show where a row came from.

- Update: `src/shared/infra/db/schema/bookings-import-row.ts`
  - Add `sourceId` (fk -> `bookings_import_source.id`)
  - Add `sourceLineNumber` (nullable int; per-file line/event index)

Migration/backfill:
- For existing single-file jobs, create one source row using `bookings_import_job.{sourceType,fileName,fileSize,filePath}`.
- Backfill each existing rowâ€™s `sourceId` and set `sourceLineNumber = lineNumber`.

### 2) API contract: accept up to 3 files (mixed)
Change createDraft input from `(sourceType, file)` to `(files[])`.

- Update: `src/modules/bookings-import/dtos/create-bookings-import.dto.ts`
  - Replace `file` with `files: zfd.repeatableOfType(importFileSchema)`
  - Remove `sourceType` from input
  - Add `.refine((files) => files.length >= 1 && files.length <= 3, ...)`
  - Keep `placeId` and optional `selectedCourtId`

Server will infer per-file source type from extension/MIME.

### 3) Backend: upload + normalize multiple sources

Files to touch:
- `src/modules/bookings-import/services/bookings-import.service.ts`
- `src/modules/bookings-import/bookings-import.router.ts`
- `src/modules/bookings-import/factories/bookings-import.factory.ts`
- New repo: `src/modules/bookings-import/repositories/bookings-import-source.repository.ts`

Create draft:
- Accept 1..3 files.
- For each file:
  - Detect `sourceType` (`ics|csv|xlsx|image`) via extension/MIME
  - Upload to Supabase storage under a job folder, e.g. `imports/<placeId>/<jobId>/<sortOrder>.<ext>`
  - Insert a `bookings_import_source` row.
- Keep `bookings_import_job` fields populated using the first file (to avoid making existing DB columns nullable):
  - `sourceType = firstSourceType`
  - `fileName/fileSize/filePath = first file`
- Store `selectedCourtId` in `job.metadata` as today.

Normalization:
- Load sources for the job (after backfill, every job should have sources).
- Determine if the job contains any images:
  - If yes: require `mode === "ai"` and `OPENAI_API_KEY`.
  - If no: allow either deterministic or AI normalization.
- For each source:
  - Download buffer
  - Parse strategy depends on `mode`:
    - **deterministic**:
      - `ics`: current deterministic mapping (location/summary/etc as in current service)
      - `csv/xlsx`: current heuristic column selection
      - `image`: not allowed (must use AI)
    - **ai**:
      - `image`: existing `extractScreenshotBookings` (AI vision)
      - `csv/xlsx`: AI generates a mapping hint/spec from headers + sample rows, then we parse using that spec (similar to `scripts/normalize-data.ts` but implemented in `src/modules/bookings-import/lib/`)
      - `ics`: AI chooses which fields (location/summary/description/constant) represent the court/resource label and which represent the reason
  - Produce `InsertBookingsImportRow` records with:
    - `sourceId`
    - `sourceLineNumber` (dataset row index / occurrence index / extracted event index)
    - `lineNumber` as a job-global incrementing number (keeps uniqueness invariant)
    - Add `sourceFileName` and `sourceType` into `sourceData` for easy debugging/UI fallback.
- Run duplicate detection across the merged rows.
- Insert rows, update counts, set `aiUsedAt` only when mode is `ai`.

AI mapping implementation notes:
- Reuse existing parsing utilities in `src/modules/bookings-import/lib/`:
  - CSV/XLSX dataset: `parseCsv`, `buildTabularDataset`, `parseXlsx`, `buildHeaderLookup`, `resolveHeader`
  - Datetime parsing: `src/modules/bookings-import/lib/datetime-parser.ts` (`parseDateTimeValue`, `buildLocalDateTime`, etc.)
- Port the *minimal* pieces from `scripts/normalize-data.ts` into server code:
  - Zod schemas for mapping hints/specs
  - Prompt builder that includes headers + sample rows/events
  - Deterministic conversion: hint -> spec -> row parsing
- Store the chosen mapping spec(s) in `bookings_import_job.metadata` keyed by sourceId for debug/replay.

Discard:
- Delete all stored source files by iterating sources.
- Mark job status `DISCARDED` as today.

### 4) tRPC read paths: expose sources to UI

## Limits / Validation
- Hard limit: `1..3` files.
- Per-file max size: keep `MAX_IMPORT_FILE_SIZE` (20MB).
- Optional: enforce max total size (e.g. 60MB) in DTO refinement.
- File type validation:
  - Client accept filter for UX
  - Server validates via MIME and extension fallback (already done per single file; replicate per file)

## Risks / Edge Cases
- AI once-per-venue: if the venue already used AI, mixed imports containing screenshots will be blocked (current product behavior). Decide if thatâ€™s acceptable.
- CSV/XLSX timezone parsing: current `new Date(str)` can interpret naive datetimes in runtime TZ; multi-file import doesnâ€™t fix this but may surface it more.
- Duplicate detection when `courtId` is null: current duplicate logic may miss duplicates if court mapping fails; consider using `courtLabel` fallback.
