# Plan: Separate Pending From General Reservation Tabs

## Goal
Update the My Reservations UI so `Pending` is displayed as its own standalone section, while the tabbed reservations area only contains `Upcoming`, `Past`, and `Cancelled`.

## Files To Modify
- `src/features/reservation/hooks.ts`
- `src/features/reservation/components/reservation-tabs.tsx`
- `src/features/reservation/components/reservation-list.tsx`

## Implementation Steps
1. **Split tab navigation from list-view filters in hooks** (`src/features/reservation/hooks.ts`)
   - Change `reservationTabs` to only: `upcoming`, `past`, `cancelled`.
   - Keep `ReservationTab` tied to those three tabs.
   - Introduce a separate list-view type (e.g. `ReservationListView = ReservationTab | "pending"`) for components/hooks that still need pending filtering.
   - Update `useMyReservations` option typing to accept the list-view type and keep existing pending filter logic intact.
   - Keep `useReservationCounts()` returning all four counts (`pending`, `upcoming`, `past`, `cancelled`) so badges continue to work.

2. **Render Pending as a dedicated section above tabs** (`src/features/reservation/components/reservation-tabs.tsx`)
   - Remove `Pending` from tab labels and tab trigger/content mappings.
   - Add a standalone `Pending` heading block before the tab UI.
   - Show the pending count badge next to the standalone heading using `counts.pending`.
   - Render pending items with `ReservationList tab="pending" isActive`.
   - Keep tab UI and `TabsContent` only for `upcoming`, `past`, `cancelled`.

3. **Allow ReservationList to render pending outside tab state** (`src/features/reservation/components/reservation-list.tsx`)
   - Update `tab` prop typing to use the new list-view type.
   - Keep fallback behavior (`tab ?? activeTab`) so tabbed usage remains unchanged.
   - Preserve empty-state copy for pending so the standalone section has a proper empty state.

4. **Handle legacy query params safely**
   - With `useReservationsTabs()` now parsing only `upcoming|past|cancelled`, URLs like `?tab=pending` or `?tab=foo` should resolve to default `upcoming` without runtime errors.

## Verification
1. Run `pnpm lint`.
2. Manual UI check on `/reservations`:
   - Page shows `My Reservations`.
   - `Pending` appears as a standalone section above tabs.
   - Tabs show only `Upcoming`, `Past`, `Cancelled`.
   - Pending cards and empty state render correctly.
   - Tab switching still updates content and URL query state.
   - Visiting `?tab=pending` and `?tab=foo` does not break UI and falls back to `Upcoming` content.

## Notes
- Keep changes minimal and localized to reservations feature components/hooks.
- Do not add new route-local components under `src/app/**`; continue using `src/features/reservation/**`.
