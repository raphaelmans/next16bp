# Admin Courts/Venues: featured filter + delete; add venues dashboard

## Goals
- Add a Featured filter on the admin courts list: `All` / `Featured only` / `Not featured` where featured means `place.featuredRank > 0`.
- Add a DELETE row action on the admin courts list for test-data cleanup.
- Add a new `/admin/venues` page that uses the same dashboard UI but defaults to `type=reservable` (venues).

## Key Findings (current state)
- `/admin/courts` (`src/app/(admin)/admin/courts/page.tsx`) is a dashboard over `place` records (venues) via `trpc.admin.court.list`.
- Featured is stored as `place.featuredRank` (0 = not featured).
- Admin list filter schema lives in `src/lib/modules/court/dtos/admin-update-court.dto.ts` (`AdminCourtFiltersSchema`) and is applied in `src/lib/modules/court/repositories/admin-court.repository.ts` (`findAll`).
- There is no existing `/admin/venues` route.

## Decisions (from you)
- Venues dashboard: add a new `/admin/venues` page (keep `/admin/courts` unchanged).
- Featured filter: `All / Featured only / Not featured`.
- Delete: hard delete the `place` row.

## Implementation Plan

### 1) Backend: featured tri-state filter
1. Update the admin list input schema to accept a tri-state:
   - File: `src/lib/modules/court/dtos/admin-update-court.dto.ts`
   - Add `featured?: boolean` to `AdminCourtFiltersSchema`.
     - `undefined` => all
     - `true` => featured only
     - `false` => not featured
2. Apply it in the repository query:
   - File: `src/lib/modules/court/repositories/admin-court.repository.ts`
   - In `findAll()`, add condition:
     - `featured === true` => `place.featuredRank > 0`
     - `featured === false` => `place.featuredRank <= 0` (rank is non-negative)
   - Import Drizzle comparators as needed (`gt`, `lte`, etc.).

### 2) Backend: admin hard delete mutation
1. Add a DTO for delete:
   - Add file: `src/lib/modules/court/dtos/admin-delete-place.dto.ts`
   - Export in: `src/lib/modules/court/dtos/index.ts`
   - Schema: `{ placeId: S.ids.placeId }`
2. Add repository method to delete a place:
   - File: `src/lib/modules/court/repositories/admin-court.repository.ts`
   - Add `deletePlaceById(placeId, ctx?)` to interface + implementation (Drizzle `client.delete(place)...`).
3. Add service method:
   - File: `src/lib/modules/court/services/admin-court.service.ts`
   - Add `deletePlaceHard(adminUserId: string, placeId: string)`:
     - Transaction: `transactionManager.run`
     - Lock/validate existence: `findByIdForUpdate(placeId, ctx)`; throw `PlaceNotFoundError` if missing.
     - Best-effort photo storage cleanup:
       - `findPhotosByPlaceId(placeId, ctx)`; for each URL, if it matches the public storage marker, delete from `STORAGE_BUCKETS.PLACE_PHOTOS`.
       - Don’t fail the whole delete if a storage delete fails; `logger.warn` and proceed.
     - Delete the place row via `deletePlaceById`.
     - Log event: `place.deleted` with `{ placeId, adminUserId }`.
4. Expose tRPC mutation:
   - File: `src/lib/modules/court/admin/admin-court.router.ts`
   - Add `deletePlace: adminRateLimitedProcedure("mutation")` (admin-only + rate limited)
   - Returns `{ success: true }`.

- Hard delete cascades: place photos/amenities/claim requests/place verification/booking import jobs that reference `place` will be removed by FK cascades.
- Courts (unit rows) will be detached (`court.placeId` becomes null due to FK onDelete set null), preserving downstream records that reference `court` (e.g., reservations) for audit.

## Follow-up: Expandable row details (Courts + Venues admin lists)

### Goal
- In both `/admin/courts` and `/admin/venues`, allow each row to expand (collapsible) to show key venue/court details inline so admins don’t need to open the dedicated detail page.

### Current primitives to reuse
- UI pattern already exists in `src/features/owner/components/reservations-table.tsx`: render a second `<TableRow>` with a single `<TableCell colSpan={N}>` as the expanded panel.
- Admin detail endpoint already exists: `trpc.admin.court.getById` (used by `useAdminCourt`) returns rich `AdminPlaceDetails` including contact details, photos, amenities, and court units.

### Proposed UX (recommended)
- Add a chevron/toggle button per row (single-row expanded at a time).
- On expand: lazy-load details via `trpc.admin.court.getById` and cache it (avoid inflating list payload / avoid 1:N joins in list endpoint).
- Expanded panel content (confirmed):
  - Address block: address, city, province, country, timeZone, extGPlaceId
  - Ownership block: organization (name + slug) or “Unowned”; claim status
  - Featured block: featuredRank
  - Contact block: phone/viber/website/social links (if present)
  - Courts block: list of court units (sport + label + tier)
  - Media: a small photo strip (first 3) + counts (photos/amenities)
  - Quick actions: copy placeId, open public detail, open admin edit

### UI/UX direction (frontend-design)
- Keep the existing admin visual language (shadcn + current typography), but make the expanded panel feel “designed”, not like a raw JSON dump:
  - Use a compact 2-column grid on desktop (single column on mobile)
  - Section cards with subtle borders + muted background stripe to separate the expanded row from the table
  - Clear micro-hierarchy: small uppercase labels, strong values, chips for amenities, and a photo strip.
  - Motion: small expand/collapse transition (height/opacity) and chevron rotation; keep it subtle.

### Potential endpoint updates (only if needed)
- If you want “at-a-glance” admin-only fields visible without expanding, extend `trpc.admin.court.list` output to include:
  - place verification status + reservationsEnabled (join 1:1 `place_verification`)
  - organization slug/id (already a left join; just select more fields)
- Otherwise keep list as-is and rely on lazy-loaded `getById` for expanded details.
