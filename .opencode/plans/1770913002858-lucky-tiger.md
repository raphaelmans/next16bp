# Plan: Strict Chat Rate Limit (5 messages/minute)

## Goal
Enforce a strict server-side chat send limit of **5 messages per minute per authenticated user** so chat spam is blocked on reservation and support chats.

## Recommended Approach
Route all user-composed chat messages through new rate-limited tRPC mutations (instead of direct `channel.sendMessage` from the client), then keep Stream SDK on the client for read/watch and file upload only.

This gives one shared backend enforcement point and uses existing Upstash rate-limit infrastructure.

## Implementation Steps

1. Add a dedicated chat-send rate-limit tier.
   - Update `src/lib/shared/infra/ratelimit/config.ts` with a new tier (for example `chatSend`) set to `{ requests: 5, window: "1 m" }`.

2. Add validated DTO/schema for outbound chat messages.
   - Add a chat DTO file (module-local) with Zod schemas for send payloads:
     - `text` optional (trimmed, max length)
     - `attachments` optional array (bounded count, allowed fields only)
     - rule: at least one of `text` or `attachments`
   - Reuse in both reservation and support routers to prevent injection/shape abuse.

3. Add rate-limited send mutations in chat routers.
   - Update `src/lib/modules/chat/reservation-chat.router.ts`:
     - add `sendMessage` mutation via `protectedRateLimitedProcedure("chatSend")`.
   - Update `src/lib/modules/chat/support-chat.router.ts`:
     - add send mutation(s) for claim/verification threads via `protectedRateLimitedProcedure("chatSend")`.
   - Keep existing error mapping; ensure `TOO_MANY_REQUESTS` is propagated cleanly.

4. Add service-layer send methods with authorization checks.
   - Update `src/lib/modules/chat/services/reservation-chat.service.ts`:
     - add send method that revalidates reservation chat access (participant + allowed status), ensures thread, then sends message.
   - Update `src/lib/modules/chat/services/support-chat.service.ts`:
     - add send method(s) that revalidate access for claim/verification conversations before sending.
   - Do not trust client-provided channel IDs as authorization proof.

5. Extend provider abstraction for generic channel sends.
   - Update `src/lib/modules/chat/providers/chat.provider.ts` and `src/lib/modules/chat/providers/stream-chat.provider.ts`:
     - add a provider method for sending message payloads (text + attachments + optional idempotency id) to a specific channel.
   - Keep existing `sendReservationMessage` behavior intact for current system-generated messages, and have new flow reuse shared send path.

6. Rewire frontend composer flow to backend mutations.
   - Update `src/features/chat/components/chat-thread/stream-chat-thread.tsx`:
     - add optional `onSendMessage` prop (backend callback).
     - keep file upload to Stream (`channel.sendFile`) but submit final message payload via callback when present.
     - fallback to existing direct send only where callback is intentionally not provided (PoC/legacy).
   - Wire callbacks in production chat surfaces:
     - `src/features/chat/components/chat-widget/player-reservation-chat-widget.tsx`
     - `src/features/chat/components/chat-widget/reservation-inbox-widget.tsx`
     - `src/features/support-chat/components/support-chat-sheet.tsx`
     - `src/features/support-chat/components/support-inbox-widget.tsx`

7. Improve user-facing rate-limit feedback.
   - Update `src/common/hooks/toast-errors.ts` with a friendly message for `RATE_LIMIT_EXCEEDED`.
   - Ensure chat send errors surface a clear "5 messages per minute" message in the UI.

## Security / OWASP Execution During Implementation

- Apply **security-review** and **owasp-security** checks while coding each step.
- A01 Broken Access Control: enforce membership/ownership checks in service before send.
- A03 Injection: strict Zod validation for text/attachments (size, count, shape).
- A04 Insecure Design: enforce rate limit on authoritative server mutation path.
- A09 Logging: keep rate-limit denial logs, but avoid logging message content.
- Secrets/config hygiene: rely on existing env-based Upstash config; verify rate limiter is active in target environment.

## Verification

1. Static validation
   - Run `pnpm lint`.

2. Manual end-to-end checks
   - Reservation chat: send 6 messages within 60 seconds as one user; 6th must fail with rate-limit message.
   - Support chat: repeat same burst test; 6th must fail.
   - Cross-thread check: send across different chat threads; global per-user cap should still block after 5/min.
   - Attachment paths: text-only, attachment-only, and text+attachment sends still work and are counted as one message send.

3. Authorization checks
   - Attempt sending to a reservation/support thread the user should not access (via crafted mutation input); expect forbidden/not-found behavior.

4. Regression checks
   - Existing system-generated reservation messages still post as before.
   - Existing chat watch/read flows and unread updates remain functional.

## Notes

- This plan enforces strict limits on all app-sanctioned message sends.
- If you want hard anti-bypass guarantees against direct client-token Stream writes, add a follow-up Stream capability/permission hardening step to disallow direct client `send-message` and keep backend as the only write path.
