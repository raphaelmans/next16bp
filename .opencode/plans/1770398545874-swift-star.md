# Owner chat status sync with payment state

## User goal
- In owner chat widget, ensure reservation status updates from `AWAITING_PAYMENT` to `PAYMENT_MARKED_BY_USER` when player marks payment.
- Make chat refresh behavior align with `docs/polling-state-approach.md` so manual refresh actually reconciles stale state.

## Root causes
- `syncInbox` in `src/features/chat/components/chat-widget/reservation-inbox-widget.tsx` currently uses `utils.reservationChat.getThreadMetas.fetch({ reservationIds })` for manual/context refresh.
- Query defaults use `staleTime: 60_000` in `src/trpc/query-client.ts`, so `fetch` can return cached data during the stale window and skip a network update, which makes chat refresh appear broken.
- Owner manual refresh paths in reservation pages refresh owner list/count but do not consistently invalidate chat metadata queries, so list and chat can drift.

## Scope and files to update
- `src/features/chat/components/chat-widget/reservation-inbox-widget.tsx`
- `src/app/(owner)/owner/reservations/page.tsx`
- `src/app/(owner)/owner/reservations/active/page.tsx`
- `docs/polling-state-approach.md`

## Implementation plan
1. Fix chat widget refresh to force reservation-meta reconciliation.
   - In `syncInbox` (`reservation-inbox-widget.tsx`), replace `getThreadMetas.fetch(...)` with targeted invalidation/refetch behavior so manual refresh bypasses stale cache.
   - Keep the existing sync state handling (`idle`/`partial`/`error`) and preserve current stream-channel refresh flow.
   - Ensure the refresh path still works when stream refresh fails by using existing fallback reservation IDs.

2. Align owner reservations page manual refresh with chat metadata.
   - In `src/app/(owner)/owner/reservations/page.tsx` `handleRefresh`, add chat meta invalidation (`reservationChat.getThreadMetas`).
   - In `confirmPaidOfflineMutation.onSuccess`, also invalidate `reservationChat.getThreadMetas` and `reservationChat.getSession` for the affected reservation.

3. Align owner active reservations refresh contract.
   - In `src/app/(owner)/owner/reservations/active/page.tsx` `handleRefresh`, invalidate:
     - `reservationOwner.getForOrganization`
     - `reservationOwner.getPendingCount`
     - `reservationChat.getThreadMetas`
   - This keeps unresolved list/count/chat refresh behavior consistent with the anti-drift guardrails.

4. Update docs to match implemented behavior.
   - In `docs/polling-state-approach.md`, add a note that chat manual refresh uses targeted invalidation/refetch (not cache-only fetch) to guarantee reconciliation under query stale-time caching.

## Verification
- Run `pnpm lint`.
- Manual repro in two sessions (player + owner):
  - Owner opens chat widget on reservation currently `AWAITING_PAYMENT`.
  - Player marks payment.
  - Owner clicks chat refresh (thread header or inbox refresh) and confirms status updates to `PAYMENT_MARKED_BY_USER` without relying on history-list refresh.
- Owner reservations page check:
  - Use page refresh button and verify list/count and chat widget status reconcile in the same refresh cycle.
- Owner active page check:
  - Use refresh button and verify active list, pending badge/count, and chat widget status are consistent.

## Risk notes
- `getThreadMetas` query keys depend on reservation ID sets; use invalidation/refetch carefully to avoid leaving an old key hot in cache.
- Keep chat event-driven behavior (no fixed polling loop) to stay aligned with current architecture.
