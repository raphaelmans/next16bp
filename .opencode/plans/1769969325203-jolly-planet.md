# Security + OWASP + Supabase/Postgres Audit Plan

## Objective
Audit the entire repository against:
- `security-review` checklist (secrets, input validation, auth, uploads, logging, dependency risk)
- OWASP Top 10 (2021) (`owasp-security`)
- Supabase/Postgres best practices (`supabase-postgres-best-practices`)

Deliverable:
- A written audit report with evidence (file + line refs), impact/likelihood, and a prioritized remediation backlog.

## Scope
### Repo (static) scope (always)
- All Next.js route handlers under `src/app/**/route.ts`
- tRPC (`src/app/api/trpc/[trpc]/route.ts`, `src/lib/shared/infra/trpc/**`, `src/lib/modules/**`)
- Auth/session handling (`src/proxy.ts`, `src/app/auth/**`, `src/lib/shared/infra/supabase/**`)
- Storage/file upload flows (`src/lib/modules/storage/**`, `scripts/seed-storage-buckets.ts`, any consumer services)
- DB schema/migrations (`drizzle/**`, `src/lib/shared/infra/db/schema/**`)
- Config/hardening (`next.config.ts`, `vercel.json`, any middleware/proxy/config)
- Scripts (`scripts/**`) for injection/unsafe patterns (even if not runtime)

### Supabase project (live) scope
Out of scope for this run (repo-only audit). Any Supabase/RLS/storage findings will be inferred from migrations, schema code, and documented assumptions.

## Repo-Specific Risk Areas Already Identified (to prioritize)
1) Cron auth fails open when `CRON_SECRET` is unset
- `src/app/api/cron/expire-reservations/route.ts` (auth check only enforced if env is set)
- `src/app/api/cron/dispatch-notification-delivery/route.ts` (same pattern)

2) Host / forwarded header trust influences redirect/origin construction
- `src/app/auth/callback/route.ts` uses `x-forwarded-host` directly in redirect target host.
- `src/lib/shared/infra/trpc/context.ts` derives `ctx.origin` from `x-forwarded-host` / `host`.
- `src/lib/shared/infra/trpc/server.ts` builds absolute URLs from forwarded headers when `env.NEXT_PUBLIC_APP_URL` is missing.

3) Storage: service-role key + public buckets + public URLs for sensitive documents
- `scripts/seed-storage-buckets.ts` sets all buckets `public: true` (incl `payment-proofs`, `place-verification-docs`).
- `src/lib/modules/storage/services/object-storage.service.ts` always returns `getPublicUrl()` after upload.
- Sensitive flows persist and/or log public URLs:
  - `src/lib/modules/payment-proof/services/payment-proof.service.ts`
  - `src/lib/modules/place-verification/services/place-verification.service.ts`

4) CSRF exposure risk for cookie-authenticated APIs
- `src/app/api/trpc/[trpc]/route.ts` allows both GET and POST and relies on cookies for auth via Supabase.

5) Missing baseline security headers
- `next.config.ts` has no `headers()`; no CSP/HSTS/etc.
- `mdx-components.tsx` uses `target="_blank"` with `rel="noreferrer"` (missing `noopener`).

6) Supabase/Postgres hygiene
- Drizzle journal only lists migrations up to `0003` (`drizzle/meta/_journal.json`) but repo contains later migrations.
- `drizzle/0000_jazzy_galactus.sql` creates `auth.users` (Supabase-managed schema in real projects).
- No visible RLS enablement/policies in migrations; Drizzle snapshots show `isRLSEnabled: false`.
- Trigram GIN indexes declared in `src/lib/shared/infra/db/schema/place.ts` imply `pg_trgm`, but no migration enabling it.
- Some query patterns can be unbounded (e.g. `BookingsImportRowRepository.findByJobId`).

## Audit Workflow
### Phase A: Baseline + Inventory
1) Build a route inventory
- Enumerate all `src/app/**/route.ts` and classify:
  - Public vs authenticated vs admin
  - Side-effecting vs read-only
  - Inputs (query/body), validation, rate limiting

2) Inventory privileged clients and secrets
- Map all Supabase client factories and how keys are used:
  - Publishable/anon vs service role
- Verify no secrets are logged, committed, or passed to client bundles.
- Validate env schema in `src/lib/env/index.ts` matches usage and required variables.

3) Dependency + supply-chain checks
- Look for dependency audit process (lockfile, overrides) and scan results.

### Phase B: OWASP Top 10 Deep Dive (repo)
For each category, collect at least: entry points, exploit narrative, and concrete mitigation.

A01 Broken Access Control
- Cron endpoints: verify auth is mandatory and cannot be bypassed by missing env.
- tRPC procedures: verify admin/owner/player access checks exist at service/repo layer.
- Storage access: ensure sensitive objects are not world-readable and access is authorized.

A02 Cryptographic Failures
- Cookie/session handling: confirm cookie flags and session refresh logic.
- Sensitive data exposure: ensure no public URLs to sensitive docs are leaked; confirm any encryption requirements.

A03 Injection
- Search for raw SQL (`sql`, `unsafe`, string concatenation) and validate parameterization.
- Confirm input validation (Zod) across route handlers and tRPC.

A04 Insecure Design
- Document security invariants (what must always be true) and compare to current implementation.

A05 Security Misconfiguration
- Security headers / CSP
- Proxy/forwarded header trust model
- Debug toggles/comments that could lead to unsafe production changes (e.g. role debug block in `src/lib/shared/infra/trpc/context.ts`).

A06 Vulnerable and Outdated Components
- Dependency scan outputs and remediation recommendations.

A07 Identification and Authentication Failures
- Auth callback/confirm correctness (redirect safety, session fixation, expected cookie behavior).

A08 Software and Data Integrity Failures
- Cron-triggered state changes: ensure idempotency and authorization.
- Any remote content (MDX, external fetch): allowlists, integrity.

A09 Security Logging and Monitoring Failures
- Verify PII isn’t logged (e.g. `src/app/api/public/track/route.ts` allows arbitrary `properties`).
- Ensure server logs use redaction (`src/lib/shared/infra/logger/index.ts`) and avoid `console.*` for sensitive errors.

A10 SSRF
- Identify all server-side `fetch` calls with dynamic URLs.
- Confirm allowlists/timeouts; treat `ctx.origin`-built URLs as risky unless origin is trusted.

### Phase C: Supabase/Postgres Best Practices
1) Migration integrity + Supabase compatibility
- Verify Drizzle migration state (`drizzle/meta/_journal.json` vs `drizzle/*.sql`).
- Identify any migrations that try to own/manage Supabase internal schemas (`auth`, `storage`) and flag.

2) RLS and policy model
- Create an RLS matrix per table (enabled? policies? intended access patterns?)
- Validate that any RBAC table (e.g. `user_roles`) cannot be self-modified to escalate privileges.

3) Extensions, indexes, constraints
- Confirm required extensions exist for schema (`pg_trgm` for trigram indexes, `btree_gist` for exclusions).
- Validate booking correctness constraints (no overlaps for reservations, time range checks).

4) Query patterns
- Identify unbounded list queries and ensure pagination/limits.
- Identify N+1 risks and heavy selects.

## Report Format (what I will produce)

- Findings (each with OWASP mapping, evidence, impact/likelihood, recommendation)
