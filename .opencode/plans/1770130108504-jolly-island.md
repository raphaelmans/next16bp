# Stream Chat PoC (Provider Adapter + Auth + DM + Custom UI)

## Goal
Ship a minimal, gated Stream Chat proof-of-concept that:
- Uses an adapter/strategy provider layer so we can swap to a self-hosted provider later.
- Mints Stream user tokens server-side and ensures users exist.
- Creates/opens a deterministic 1:1 DM channel for two user IDs.
- Exposes a protected dashboard page that uses our own shadcn-based UI (no `stream-chat-react` UI kit / global CSS).

## Key Decisions
- API shape: use `tRPC chatPoc` (selected) to reuse existing auth context + typed hooks.

## Dependencies
- Keep: `stream-chat`
- Remove: `stream-chat-react` (we will not rely on the UI kit or its global CSS)
- Add/keep: AI Elements (already installed via `npx shadcn@latest add @ai-elements/all`)
  - UI building blocks live in `src/components/ai-elements/**`.

Deterministic DM channel id
- Don’t use `dm:${a}:${b}`:
  - `:` is risky/invalid in Stream channel IDs, and UUID-based IDs will exceed common 64-char limits.
- Use a stable hash: `dm-<sha256(min+":"+max).slice(0, 32)>`.

Stream provider behaviors (via `stream-chat` docs)
- Server client: `new StreamChat(apiKey, apiSecret, { disableCache: true })`
- Ensure user(s): `client.upsertUsers([{ id, name?, image? }, ...])`
- Mint token: `client.createToken(userId)`
- Create channel: `client.channel("messaging", channelId, { created_by_id: userId, members })` + `await channel.create()`

## Next: Productize For Reservations (V1)

### Goals
- Enable peer-to-peer chat between the **player** and **court owner** for a given reservation.
- Enable chat from `CREATED` through the active lifecycle (player UI can still auto-open only on `CONFIRMED`).
- Ensure future complaint/dispute workflows can reliably retrieve **auditable message evidence**.
- Keep provider adapter/strategy so we can swap Stream → self-host later.

### V1 Scope (Simple, Extensible)
- Channel model: **1 channel per reservation** (recommended for auditability).
- Channel becomes available when reservation `status ∈ { CREATED, AWAITING_PAYMENT, PAYMENT_MARKED_BY_USER, CONFIRMED }`.
- Persist a small mapping in DB: reservation ↔ provider channel.
- Add admin-only ability to capture a **transcript snapshot** (immutable evidence) stored in DB.

Non-goals for V1 (defer)
- Full dispute/complaint workflow UI.
- Stream webhook ingestion of every message event (v2).
- Guest booking chat (`guestProfileId` reservations) (v2).

### Reservation + Participants Mapping
Source of truth:
- Reservation status contract: `docs/reservation-state-machine-level-2-engineering.md`
- Owner acceptance/confirm flows:
  - `src/lib/modules/reservation/services/reservation-owner.service.ts`
    - `acceptReservation(...)`: paid -> `AWAITING_PAYMENT`, free -> `CONFIRMED`
    - `confirmPayment(...)`: `PAYMENT_MARKED_BY_USER` -> `CONFIRMED`

Participants (auth user ids):
- **Player**: `reservation.playerId -> profile.userId` (if `playerId` is null, reservation is guest booking; v1: no chat)
- **Owner**: `reservation.courtId -> court.placeId -> place.organizationId -> organization.ownerUserId`

### Channel Model
- Provider type: `messaging`
- Deterministic channel id: `res-${reservationId}`
- Channel custom data (provider-specific): `{ reservation_id: reservationId }`

### DB Schema (Minimal)
Add two tables to support linkage + audit snapshots.

1) `reservation_chat_thread`
- Purpose: map reservation → provider channel.
- Columns (suggested):
  - `id` uuid PK
  - `reservation_id` uuid FK (unique) -> `reservation.id`
  - `provider_id` (text/varchar; e.g. `stream`)
  - `provider_channel_type` (text; `messaging`)
  - `provider_channel_id` (text; `res-<uuid>`)
  - `created_by_user_id` uuid FK -> `auth.users.id` (nullable)
  - `created_at` timestamptz default now
- Constraints/indices:
  - `unique(reservation_id)`
  - `unique(provider_id, provider_channel_type, provider_channel_id)`

2) `reservation_chat_transcript`
- Purpose: immutable “evidence capture” snapshots for disputes/complaints.
- Columns (suggested):
  - `id` uuid PK
  - `reservation_id` uuid FK -> `reservation.id`
  - `provider_id`, `provider_channel_type`, `provider_channel_id`
  - `captured_by_user_id` uuid FK -> `auth.users.id` (nullable)
  - `captured_at` timestamptz default now
  - `message_count` int
  - `first_message_at`, `last_message_at` timestamptz nullable
  - `transcript_sha256` varchar(64)
  - `transcript_json` jsonb
- Indices:
  - `(reservation_id, captured_at)`

Files:
- New schema: `src/lib/shared/infra/db/schema/reservation-chat.ts`
- Export: `src/lib/shared/infra/db/schema/index.ts`
- New migration: `drizzle/0020_reservation_chat.sql` (next available number)

### Backend (tRPC)
Add a reservation-aware router separate from `chatPoc`.

Router:
- `src/lib/modules/chat/reservation-chat.router.ts`
- Wire: `src/lib/shared/infra/trpc/root.ts` as `reservationChat`

Procedures:
1) `reservationChat.getSession`
- Input: `{ reservationId }`
- Output: `{ auth: { apiKey, user, token }, channel: { providerId, channelType, channelId, memberIds } }`
- Behavior:
  - Resolve reservation + participants
  - Gate by status (CREATED | AWAITING_PAYMENT | PAYMENT_MARKED_BY_USER | CONFIRMED)
  - Ensure caller is participant (player or owner)
  - Ensure Stream users exist
  - Ensure channel exists (idempotent)
  - Upsert `reservation_chat_thread` mapping (idempotent)

2) `reservationChat.captureTranscriptSnapshot` (admin-only)
- Input: `{ reservationId }`
- Behavior:
  - Look up thread mapping; export all messages from provider
  - Canonicalize JSON + compute sha256
  - Insert row into `reservation_chat_transcript` (append-only)

3) `reservationChat.listTranscriptSnapshots` (admin-only)
- Input: `{ reservationId }`
- Output: snapshot metadata list

Service/repository layering (match conventions like claim/verification):
- `src/lib/modules/chat/services/reservation-chat.service.ts`
- `src/lib/modules/chat/repositories/reservation-chat-thread.repository.ts`
- `src/lib/modules/chat/repositories/reservation-chat-transcript.repository.ts`

Provider abstraction extensions (keep strategy pattern):

- `src/lib/modules/chat/providers/stream-chat.provider.ts` implement
- `src/lib/modules/chat/services/chat.service.ts` can stay for DM; add reservation-focused service separately.

V2: webhook ingestion

## Archive + Refresh + Composable Inbox (Player + Owner)

### Goals

### Archive Rules (Recommended)

## Data Fetching State Machine + Semantic Loading (Inbox)

### Observed Problem
- Inbox list initially renders fallback labels (e.g. `Reservation ABCD1234`) and updates seconds later when reservation metas arrive.
- This is amplified by backend N+1 in `reservationChat.getThreadMetas` (multiple DB queries per reservationId).

### Desired UX
- Two-stage rendering that feels intentional:
  - Stage A (Main items): show list rows quickly using Stream channel data (unread + latest message + stable skeleton title).
  - Stage B (Sub-details): hydrate reservation meta in parallel (status pill + venue/court/time). No layout jump.
- Refresh should not “blank” the UI; it should keep previous data and show a subtle fetching indicator.

### State Machine (ASCII)

Inbox Widget

  [CLOSED]
     |
     | open
     v
  [BOOT]
     |  authQuery pending
     v
  [AUTH_READY] --(stream connect pending)--> [CONNECTING]
     |                                      |
     | connect ok                           | connect error
     v                                      v
  [CONNECTED] --------------------------> [ERROR_CONNECT]
     |
     | fetch channels
     v
  [CHANNELS_LOADING] --ok--> [CHANNELS_READY] --derive ids--> [METAS_LOADING] --ok--> [READY]
        |  err           |                         |  err              |
        v                v                         v                   v
   [ERROR_CHANNELS]  [EMPTY]                 [ERROR_METAS]        (interactive)

Thread Panel (within READY)

  [NO_SELECTION]
      |
      | select reservationId
      v
  [THREAD_CONNECTING] --watch ok--> [THREAD_READY]
        |  watch err                 | refresh
        v                            v
   [THREAD_ERROR]               [THREAD_REFRESHING]

### UI Loading Rules (Semantic Batching)
Use explicit lanes:

Lane A: Stream channel list
- `channels.isPending` => show 3-5 skeleton rows (no reservation ids).
- `channels.isFetching` with existing data => keep rows, show spinner on refresh button.

Lane B: Reservation metas (`reservationChat.getThreadMetas`)
- When pending and no cached meta => keep rows but render:
  - title skeleton (fixed width)
  - subtitle skeleton
  - status pill placeholder
- When fetching with cached meta => keep meta, show subtle “Updating…” spinner.

Thread messages
- Always render thread shell; show:
  - “Connecting” empty state until channel watch resolves.
  - “Archived read-only” banner if archived.

### Backend Performance Fix (Required)
Replace N+1 `getThreadMetas`:
- Current implementation loops reservationIds and calls `getReservationContext` repeatedly.
- Implement a single Drizzle query (join reservation + profile + court + place + organization) with `inArray(reservation.id, reservationIds)`.
- Filter authorization in SQL or post-filter:
  - player: `profile.userId === ctx.userId`
  - owner: `organization.ownerUserId === ctx.userId`
- Return all meta rows in one round trip.

Files:
- `src/lib/modules/chat/services/reservation-chat.service.ts` (rewrite `getThreadMetas`)
- Optional: new repository `src/lib/modules/chat/repositories/reservation-chat-meta.repository.ts` (clean layering)

### Client Implementation Notes
- Use React Query/TanStack semantics:
  - Prefer `isPending` for initial load, `isFetching` for background refresh.
  - Use `placeholderData` / keep previous data to avoid list clearing on refetch.
- Use `startTransition` when applying meta hydration that changes grouping (Active vs Archive) to reduce perceived jank.
