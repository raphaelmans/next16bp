# Fix clear behavior on location pages

## Goal
When users land on location routes like `/courts/locations/cebu/cebu-city`, make it obvious how to remove the path-based location constraint by adding a dedicated `Reset location` action while preserving existing `Clear filters` behavior for query filters.

## Root cause
- Location pages inject `initialFilters` from pathname segments in:
  - `src/app/(public)/courts/locations/[province]/page.tsx`
  - `src/app/(public)/courts/locations/[province]/[city]/page.tsx`
- `src/app/(public)/courts/courts-page-client.tsx` computes effective filters with fallback:
  - `effectiveProvince = filters.province ?? initialFilters?.province`
  - `effectiveCity = filters.city ?? initialFilters?.city`
- `filters.clearAll()` (nuqs query state) clears URL params only; it cannot clear pathname segments, so on pure location routes Clear Filters can look like a no-op.

## Recommended implementation

### 1) Wire page-level state in `src/app/(public)/courts/courts-page-client.tsx`
- Import `appRoutes`.
- Compute:
  - `hasLocationDefaults` from `initialFilters`.
  - `hasClearableFilters` from actual nuqs state (`filters.q`, `filters.province`, `filters.city`, `filters.sportId`, `filters.verification`, `filters.amenities`).
  - `resetLocationHref = hasLocationDefaults ? appRoutes.courts.base : undefined`.
- Pass new props to filter UIs:
  - `resetLocationHref`
  - `hasClearableFilters`
- For empty state, pass:
  - `onClearFilters={hasClearableFilters ? filters.clearAll : undefined}`
  - This lets existing `EmptyResults` fallback to `Browse all courts` (`/courts`) when there is nothing query-based to clear.

### 2) Add optional reset action to desktop filters in `src/features/discovery/components/court-filters.tsx`
- Extend `PlaceFiltersProps` with optional:
  - `resetLocationHref?: string`
  - `hasClearableFilters?: boolean`
- Keep current clear logic as default, but when `hasClearableFilters` is provided, use it to control `Clear filters` visibility/state.
- Render a secondary `Reset location` button (link to `resetLocationHref`) when provided.

### 3) Add optional reset action to mobile sheet in `src/features/discovery/components/place-filters-sheet.tsx`
- Extend `PlaceFiltersSheetProps` with optional:
  - `resetLocationHref?: string`
  - `hasClearableFilters?: boolean`
- Use `hasClearableFilters` to disable/hide ineffective `Clear all` behavior when only pathname location is active.
- Add `Reset location` button in the sheet footer when `resetLocationHref` is present.

## Why this approach
- Keeps responsibilities clear:
  - `Clear filters` = clears nuqs query filters.
  - `Reset location` = removes pathname location scope by routing to `/courts`.
- Minimal blast radius (focused to courts page + discovery filter UI).
- Preserves current behavior on normal `/courts` pages (no location defaults => no reset action).

## Verification
1. Location-route baseline:
   - From landing chip, open `/courts/locations/cebu/cebu-city`.
   - Confirm `Reset location` is visible (desktop filter bar and mobile filter sheet).
2. Reset behavior:
   - Click `Reset location` and confirm navigation to `/courts`.
3. Clear behavior:
   - On location route with additional query filters (e.g. sport/amenities/search), click `Clear filters` and confirm query filters clear while staying on the location path.
   - On location route with only pathname location active, ensure ineffective clear behavior is no longer presented as an active action.
4. Non-location route regression:
   - On `/courts`, confirm filters and `Clear filters` behave as before; `Reset location` is not shown.
5. Validation:
   - Run `pnpm lint` from repo root.
