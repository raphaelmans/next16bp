# Fix navbar search visual cue from URL query

## Goal
Ensure the public navbar search input reflects the active `q` filter from the URL (e.g. `/courts?q=cebu+city`) so users can immediately see why results are narrowed.

## Root cause
- `src/features/discovery/components/navbar.tsx` controls input value with local `useState("")` only.
- `/courts` filtering already reads `q` via nuqs in `src/features/discovery/hooks.ts` + `src/features/discovery/schemas.ts`, so results are filtered even when navbar appears empty.

## Recommended implementation
Modify only `src/features/discovery/components/navbar.tsx`:

1. Import nuqs query parser/hook for client usage:
   - `useQueryState` and `parseAsString` from `nuqs`.
2. Read `q` from URL in the navbar component:
   - `const [queryParam] = useQueryState("q", parseAsString);`
3. Keep existing local `searchQuery` state (shared by desktop/mobile inputs), but hydrate/sync it from URL:
   - Initialize from `queryParam ?? ""`.
   - Add a `useEffect` that updates `searchQuery` whenever `queryParam` changes.
4. Keep current submit behavior intact (`handleSearch`, telemetry event, `router.push`, mobile sheet close).

## Why this approach
- Minimal blast radius: one-file change.
- Uses nuqs as requested, aligned with existing project conventions.
- Preserves current UX semantics (search runs on submit, not every keystroke).
- Fixes direct loads, landing redirects, and browser back/forward query changes.

## Verification
1. Manual flow checks:
   - From landing page, search `cebu city` and confirm `/courts?q=cebu+city` shows `cebu city` in navbar input.
   - Open `/courts?q=manila` directly and confirm input is prefilled.
   - Open `/courts` with no `q` and confirm input is empty.
   - Use browser back/forward across different `q` URLs and confirm input stays in sync.
   - Confirm both desktop and mobile navbar search inputs show the same synced value.
2. Validation command:
   - Run `pnpm lint` from repo root.
