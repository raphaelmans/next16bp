# Reservation Enablement Diagnostics (Public Venue + Owner Studio)

## Context
- Reported issue: reservation/booking UI still does not show correctly on `http://localhost:3000/venues/kusos-courts-complex` (route re-exports `src/app/(public)/places/[placeId]/page.tsx`).
- Goal: extract a shared util that can (1) detect what is missing for online reservations to be enabled, and (2) drive consistent UI messaging on:
  - Public venue page (`src/app/(public)/places/[placeId]/place-detail-client.tsx`)
  - Owner Availability Studio (`src/app/(owner)/owner/places/[placeId]/courts/[courtId]/availability/page.tsx`)

## Findings (current behavior)
- Public booking is gated by `getPlaceVerificationDisplay()` in `src/features/discovery/helpers.ts`:
  - `showBooking = placeType === "RESERVABLE" && verificationStatus === "VERIFIED" && reservationsEnabled === true`
- Public availability empty state diagnostics are inconsistent:
  - Court day view passes `courtDayAvailabilityQuery.data?.diagnostics` into `AvailabilityEmptyState`.
  - “Any court” day view passes `diagnostics={null}`, which forces an `unknown` reason even when the backend knows `no_schedule` / `no_pricing` / `reservations_disabled`.
- Availability diagnostics DTO schema is missing `reservationsDisabled`:
  - `src/lib/modules/availability/dtos/availability.dto.ts` defines `AvailabilityDiagnosticsSchema` without it.
  - Frontend `src/features/discovery/hooks.ts` duplicates a diagnostics type that also omits it.
- Owner studio already shows banners for unverified + reservations disabled, but does not warn for missing schedule/pricing.
  - Owner has the needed data sources available: `useCourtHours(courtId)` and `trpc.courtRateRule.get` (via `useCourtRateRules(courtId)` in `src/features/owner/hooks.ts`).

## Design

### 1) Shared util: reservation enablement checklist
Add a shared util under `src/common/` (safe to import from both public + owner):

- New file: `src/common/reservation-enablement.ts`
- API (pure function, no React):

```ts
export type ReservationEnablementIssueCode =
  | "PLACE_NOT_RESERVABLE"
  | "VERIFICATION_REQUIRED"
  | "VERIFICATION_PENDING"
  | "VERIFICATION_REJECTED"
  | "RESERVATIONS_DISABLED"
  | "NO_SCHEDULE"
  | "NO_PRICING";

export type ReservationEnablementIssue = {
  code: ReservationEnablementIssueCode;
  severity: "info" | "warning" | "destructive";
};

export type ReservationEnablementInput = {
  placeType?: "CURATED" | "RESERVABLE" | null;
  verificationStatus?: "UNVERIFIED" | "PENDING" | "VERIFIED" | "REJECTED" | null;
  reservationsEnabled?: boolean | null;
  hasHoursWindows?: boolean | null;
  hasRateRules?: boolean | null;
};

export function getReservationEnablement(input: ReservationEnablementInput): {
  canShowPublicBooking: boolean;
  issues: ReservationEnablementIssue[];
};
```

Rules:
- `canShowPublicBooking` matches current behavior: `RESERVABLE + VERIFIED + reservationsEnabled`.
- `issues` enumerates missing prerequisites in a stable order:
  1) Place not reservable (curated or missing type)
  2) Verification (unverified/pending/rejected)
  3) Reservations disabled
  4) Schedule missing (`hasHoursWindows === false`)
  5) Pricing missing (`hasRateRules === false`)

Note: Schedule/pricing issues are optional signals; they only appear if the caller passes `hasHoursWindows/hasRateRules`.

### 2) Keep existing discovery helper, but re-implement via shared util
Update `src/features/discovery/helpers.ts` so `getPlaceVerificationDisplay()` delegates to `getReservationEnablement()` to compute `showBooking` and message/variant.
- Avoid broad refactors by keeping the exported function signature the same.

### 3) Public venue page: fix missing diagnostics + show correct messaging
Update `src/app/(public)/places/[placeId]/place-detail-client.tsx`:

- Fix “Any court day view” empty state:
  - Pass `anyDayAvailabilityQuery.data?.diagnostics ?? null` instead of `null`.
  - Same pattern for the mobile day queries (wherever `AvailabilityEmptyState` is used with `diagnostics={null}` while an availability query exists).

- When `!showBooking`, show the specific reason (pending/rejected/unverified/reservations disabled) via the existing `VerificationStatusBanner` (no owner-only links).

This matches the intent that “what’s missing” should be visible from the public renderer too, while the owner studio gets CTAs.

### 4) Owner Availability Studio: add missing schedule/pricing warnings using the same util
Update `src/app/(owner)/owner/places/[placeId]/courts/[courtId]/availability/page.tsx`:

- Add rate rules query:
  - Use `useCourtRateRules(courtId)`.
  - `hasRateRules = (rateRulesQuery.data?.length ?? 0) > 0`.
- Reuse existing `courtHoursQuery`:
  - `hasHoursWindows = (courtHoursQuery.data?.length ?? 0) > 0`.

- Compute issues via `getReservationEnablement({ placeType: placeData.place.placeType, verificationStatus, reservationsEnabled, hasHoursWindows, hasRateRules })`.
- Render a stacked banner section (non-blocking) above the studio timeline:
  - Keep existing verification/reservations disabled alerts but drive them from `issues` instead of ad-hoc booleans.
  - Add new alerts for `NO_SCHEDULE` and `NO_PRICING` with CTA linking to `scheduleHref`.

This addresses the requirement that schedule + pricing are “key verification” for owners as well (not just public availability empty state).

### 5) Align diagnostics typing with backend behavior
Update availability diagnostics DTO + frontend types to include `reservationsDisabled`:
- `src/lib/modules/availability/dtos/availability.dto.ts`:
  - Add `reservationsDisabled: z.boolean().optional()` to `AvailabilityDiagnosticsSchema`.
- `src/features/discovery/hooks.ts`:
  - Extend `AvailabilityDiagnostics` interface to include `reservationsDisabled?: boolean`.
  - Update `emptyDiagnostics` to include `reservationsDisabled: false`.

This keeps TS aligned and makes it safe to pass diagnostics through without losing the “reservations disabled” reason.
