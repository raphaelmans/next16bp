# Plan: Court Selection Before Bookings Import

## Goal
Add an optional “single court” mode to owner bookings import so an owner can pick a venue, then (optionally) pick a specific court **before upload/normalization**. When a court is selected, all imported bookings are mapped/committed against that court.

This extends the existing flow described in `agent-plans/71-bookings-import-review-commit/71-00-overview.md`.

## Current State (Found In Repo)
- Upload step (Step 1): `src/app/(owner)/owner/import/bookings/page.tsx`
  - Picks venue (`placeId`) + `sourceType`, uploads file via `trpc.bookingsImport.createDraft(FormData)`.
  - No court selection.
- Review/normalize/commit (Steps 2–4): `src/app/(owner)/owner/import/bookings/[jobId]/page.tsx`
  - Normalizes via `trpc.bookingsImport.normalize({ jobId, mode })`.
  - Per-row court edits via an “Edit row” dialog.
- Server:
  - Create draft input schema: `src/modules/bookings-import/dtos/create-bookings-import.dto.ts`.
  - Job schema includes JSON metadata: `src/shared/infra/db/schema/bookings-import-job.ts` (`metadata: jsonb`).
  - Normalize/commit logic: `src/modules/bookings-import/services/bookings-import.service.ts`.

## UX / UI Design (KudosCourts Design System Aligned)
Design intent: minimalist bento; reduce cognitive load; make “court scope” explicit.

### Step 1 (Upload) additions
Location: inside the Step 1 card on `src/app/(owner)/owner/import/bookings/page.tsx`, between “Venue” and “Upload file” (or between “Source type” and “Upload file”).

1) After venue selection, fetch courts for that place (owner-managed list) and show a “Court scope” section:
- Default: `Multiple courts` (no court selection).
- Optional: `Single court` (reveals a court select).

2) Smart defaults:
- If the venue has exactly 1 court: auto-enable single-court mode and preselect that court (no extra user action).
- If `sourceType === "image"`: show a subtle tip that screenshots often don’t include court names and selecting a court avoids mapping errors.

3) Copy:
- Multiple courts: “Use court names in the file (you can remap rows during review).”
- Single court: “Assign every imported booking to one court. Court selection is locked during review.”

### Step 3 (Review) clarity
On `src/app/(owner)/owner/import/bookings/[jobId]/page.tsx`, show the chosen court scope in “Import details”:
- “Court scope: Multiple” OR “Court scope: <Court label> (single-court import)” (badge or secondary text).

Also adjust the table’s Court cell to prefer the canonical mapped court label when `row.courtId` exists (prevents confusing display when `courtLabel` is present but we’ve forced/selected a court).

## Data Model / Contract
### Persist selection on the job
Store `metadata.selectedCourtId?: string` on `bookings_import_job.metadata`.
- Pros: no migration; survives refresh/resume.

### Update createDraft input
Extend `CreateBookingsImportSchema` to accept optional `selectedCourtId` (string) in the FormData payload.

## Server Behavior
### createDraft
File: `src/modules/bookings-import/services/bookings-import.service.ts`
- Accept optional `selectedCourtId`.
- Validate ownership + ensure selected court belongs to the chosen `placeId` (and is active, if that’s a rule elsewhere).
- Persist into `bookings_import_job.metadata.selectedCourtId`.

### normalize
File: `src/modules/bookings-import/services/bookings-import.service.ts`
- Read `selectedCourtId` from job.metadata.
- If present, assign `courtId = selectedCourtId` for every parsed row *before* validation, and clear `courtLabel` (or keep it but ensure UI doesn’t show it over the mapped court).
- This should prevent “Court label could not be mapped…” errors in single-court mode.

### commit
File: `src/modules/bookings-import/services/bookings-import.service.ts`
- If `selectedCourtId` is present on the job, force `courtId = selectedCourtId` for all rows at commit-time (even if a row has a different `courtId`).
- This ensures the requirement “all bookings are mapped against the selected court” holds, even if someone edited a row.

### review/edit restrictions (optional)
On `src/app/(owner)/owner/import/bookings/[jobId]/page.tsx`:
- If single-court mode: disable/hide court selection in the Edit Row dialog (court is derived from job selection). Keep time/reason edits.

## Client Changes
### Upload page
File: `src/app/(owner)/owner/import/bookings/page.tsx`
- Add state for `courtScope: "multi" | "single"` and `selectedCourtId`.
- Fetch courts by place when `selectedPlaceId` is set (likely `trpc.courtManagement.listByPlace` or the existing hook `useOwnerCourtsByPlace`).
- If `courtScope === "single"`, require a court before enabling “Continue” (unless exactly 1 court, auto-selected).
- Append `selectedCourtId` to FormData when present.

### Review page
File: `src/app/(owner)/owner/import/bookings/[jobId]/page.tsx`
- Show selected court scope in Import details.
- Court cell display: prefer `row.courtId` -> mapped label; fall back to `row.courtLabel` only when unmapped.
- If single-court mode, disable court select in the row edit dialog.

## Edge Cases
- Venue has 0 courts: block “single court” mode and show guidance to configure courts first.
- Venue has 1 court: auto-map, but still allow multi-court mode? (likely unnecessary; default to single).
- File includes court labels that don’t match: single-court mode should ignore labels (no errors).
- Idempotent commit: ensure forcing court doesn’t break already-committed rows.

## Verification
- UI:
  - Upload page: selecting “Single court” gates Continue until a court is selected.
  - After upload, review page shows “Court scope” in details.
  - Normalize produces rows without court mapping errors when single-court mode is used.
  - Commit creates `court_block` rows for the selected court.
- Commands:
  - `pnpm lint`
  - `pnpm build`
  - `TZ=UTC pnpm build`

## Decision
- Court selection is **single-court override**: if `metadata.selectedCourtId` is set, normalize + review + commit treat every row as belonging to that court (ignore any file court labels).
