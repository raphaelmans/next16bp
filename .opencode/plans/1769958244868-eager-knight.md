# Admin Notification Test Page (Plan)

Goal: Add an admin-only page with simple text-input forms to trigger notification outbox jobs (and optionally dispatch) without going through reservations/verification/claims flows.

## What We’ll Build

- New admin page: `/admin/tools/notification-test`
  - Enqueue test jobs for these event types:
    - `reservation.created`
    - `place_verification.approved` / `place_verification.rejected`
    - `claim_request.approved` / `claim_request.rejected`
  - Inputs are plain text fields (email/phone + a handful of message fields).
  - Server generates any required IDs (UUIDs) so payload always matches dispatcher validation.
  - Shows results (jobCount, and IDs if useful).

- New admin tRPC router: `trpc.admin.notificationDelivery`
  - `enqueueTest` (or per-event enqueue procedures)
  - `dispatchNow`: triggers `/api/cron/dispatch-notification-delivery` server-to-server and returns the JSON result.

## Why This Approach

- Reuses existing pipeline (outbox + cron dispatcher) so it tests the real integration.
- Admin-only protection is already enforced by `(admin)` layout (`src/app/(admin)/layout.tsx`) and `adminProcedure`.
- No need to create fake reservations/claims/verifications in the DB.

## Implementation Details

### 1) Backend: Admin Router

Add `src/lib/modules/notification-delivery/admin/notification-delivery-admin.router.ts`:

- Procedures (admin-only):
  - `enqueueReservationCreatedTest`
  - `enqueuePlaceVerificationReviewedTest`
  - `enqueueClaimReviewedTest`
  - `dispatchNow` (optional)

- Validation:
  - Zod input schemas for each procedure.
  - Normalize phone via `normalizePhMobile()`.
  - Generate required IDs using `globalThis.crypto.randomUUID()`.

- Enqueue strategy:
  - Create jobs directly via `NotificationDeliveryJobRepository.createMany()` using provided targets.
  - Set `eventType` to the selected event.
  - Provide payload that satisfies the dispatcher’s Zod schemas.
  - Use an idempotency key that allows repeated tests, e.g. `test:<eventType>:<uuid>:email|sms`.

- Dispatch strategy (enabled, including prod):
  - Server-side `fetch(`${ctx.origin}/api/cron/dispatch-notification-delivery`, { headers })`.
  - If `process.env.CRON_SECRET` exists, send `Authorization: Bearer <CRON_SECRET>`.

### 2) Mount Router

Update `src/lib/shared/infra/trpc/root.ts`:

- Import and mount:
  - `notificationDelivery: notificationDeliveryAdminRouter`

This enables `trpc.admin.notificationDelivery.*` on the client.

### 3) Frontend: Admin Page

Add `src/app/(admin)/admin/tools/notification-test/page.tsx` (client component):

- Use the existing admin shell pattern:
  - `AppShell`, `AdminSidebar`, `AdminNavbar`, `useSession`, `useLogout`.

- UI:
  - A select for event type.
  - Text inputs for:
    - Targets: `email` (optional), `phoneNumber` (optional)
    - Common fields: `placeName`, `notes` (optional)
    - Reservation fields: `courtLabel`, `startTimeIso`, `endTimeIso`, `totalPriceCents`, `currency`, `playerName`
    - Reviewed fields: `status` select (`APPROVED|REJECTED`)
  - Submit calls the corresponding tRPC mutation.
  - A separate “Dispatch now” button (optional) calls `dispatchNow`.

- Safety:
  - Dispatch requires a checkbox confirmation like “This sends real email/SMS”.
  - Display the returned dispatch JSON (processed/sent/failed/skipped).

### 4) Optional: Navigation Link

Optionally add a nav item in `src/features/admin/components/admin-sidebar.tsx`:
- “Tools” -> `/admin/tools/notification-test`

# Follow-up: Retry Semaphore Sender Name Failures

Goal: When Semaphore rejects a request due to an unapproved/invalid `sendername`, automatically retry once without `sendername`, and log the retry.

## Approach

- File: `src/lib/shared/infra/sms/semaphore-sms.service.ts`
- In `sendSms()`:
  - Attempt #1: send as-is (includes `sendername` when configured)
  - If HTTP response is non-2xx and body indicates invalid sender name, log a warning and retry once with `sendername` omitted.
  - If retry succeeds, return normally.
  - If retry fails, throw as usual so the outbox job records a failure.

## Detection

- Match error body text conservatively:
  - Must mention `sendername` or `sender name`
  - Must also include `invalid` / `not approved` / `not valid` / similar

## Logging

- Add `logger.warn` with:
  - `event: "sms.semaphore.retry_without_sendername"`
  - `httpStatus` (first attempt)
  - `senderName` (configured value)
  - `toLast4` (do NOT log full phone)
  - `endpoint` (`messages` vs `priority`)
  - `responseBody` (truncated)

## Verification

- Set `SEMAPHORE_SENDER_NAME` to an unapproved value; enqueue SMS via `/admin/tools/notification-test` and dispatch.
- Confirm logs show the retry event and that SMS sends on retry (or fails with a clear error).
