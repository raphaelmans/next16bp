# Owner Bookings Studio: Week + Month Views

## Current State (Repo)

- Studio lives in `src/app/(owner)/owner/bookings/page.tsx`.
- Date state is **single-day only** via `dayKey` query param (`nuqs`) and the UI renders a single-day vertical timeline.
- Drag/drop uses `@dnd-kit/core` with `useDraggable` + `useDroppable` and droppable IDs/data shaped like `{ kind: "timeline-cell", dayKey, startMinute }`.
- Plan reference: `agent-plans/75-owner-bookings-playground-dnd/75-00-overview.md` and `agent-plans/75-owner-bookings-playground-dnd/75-01-playground-route-shell.md` (expects `view` query param).

## Goal

Add **weekly** and **monthly** viewing modes to the owner studio so owners can browse more than a single day while keeping:

- place timezone canonical (`place.timeZone`)
- existing DnD flows (preset create, block move/resize, import draft-row drop)
- URL-driven state (shareable links)

## Recommended UX (Updated)

- `view=day|week|month` (query param)
- **Week (default)**: 7-day time grid (columns) with the same DnD behavior across days (create + move + resize)
- **Month**: 42-cell month grid picker; clicking a day zooms back to Week (DnD disabled)
- **Day**: existing single-day timeline (no behavior change; still supports move + resize)

Note: The reported “resize removed” regression is due to Week view currently rendering blocks as non-interactive indicators; Week must render draggable blocks + resize handles to preserve functionality.

Animation recommendation:

- Use Motion for React (`motion/react`) for **view-level** transitions (Week ↔ Month ↔ Day) because it cleanly supports enter/exit animations during mount/unmount (`AnimatePresence`) and ships `useReducedMotion()`.
- Keep Tailwind/CSS transitions for **micro-interactions** (hover/focus/active states) and simple loading affordances.
- Avoid mixing Tailwind animation utilities and Motion on the same element for the same effect; prefer one source of truth per animation.

## Implementation Plan

### 1) Add URL State: `view`

- In `src/app/(owner)/owner/bookings/page.tsx`, add:
  - `const studioViewSchema = ["day", "week", "month"] as const;`
  - `const [viewParam, setViewParam] = useQueryState("view", parseAsStringLiteral(studioViewSchema).withOptions({ history: "replace" }))`
  - `const view = viewParam ?? "week"` (Week is default once Week view supports full edit interactions)

Reference (Context7): dnd-kit context provider + sensors patterns
- `/websites/dndkit` docs: `DndContext` props + sensors via `useSensors` and `useSensor`.

### 2) Compute Visible Date Window (Day/Week/Month)

- Keep `dayKey` as the *focused day* (used for right-side details and for “Today”).

Add pure helpers (either inline or extracted). Week starts on **Sunday** (`weekStartsOn=0`) by default:

- `getWeekStartDayKey(dayKey, timeZone, weekStartsOn=0)`
  - `dayStart = getZonedDayRangeFromDayKey(dayKey, tz).start` (TZDate in place TZ)
  - `dayOfWeek = dayStart.getDay()` (place-TZ day-of-week)
  - `delta = (dayOfWeek - weekStartsOn + 7) % 7`
  - `weekStart = addDays(dayStart, -delta)` (date-fns works with TZDate in this repo)
  - return `getZonedDayKey(weekStart, tz)`
- `getVisibleDayKeys({ view, dayKey, timeZone })`:
  - day: `[dayKey]`
  - week: 7 keys starting from `weekStartDayKey`
  - month: all dayKeys in the month of `dayKey` (or compute month start/end and derive keys)
- `getBlocksQueryRangeIso({ view, dayKey, timeZone })`:
  - start = start-of-first-visible-day (zoned)
  - end = end-of-last-visible-day (zoned)
  - convert using `toUtcISOString`

### 3) Fetch Blocks For The Visible Range

- Replace the current day-only `dayStartIso/dayEndIso` feeding `trpc.courtBlock.listForCourtRange` with `rangeStartIso/rangeEndIso` derived from `view`.
- Keep `enabled: Boolean(courtId)`.

### 4) Derive View Models

#### Week timeline segments

- Compute `timelineBlocksByDayKey` for rendering in a 7-column grid.
- For correctness when blocks overlap midnight/week boundaries, split into per-day “segments”:
  - For each visible `dayKey`, compute `dayStart = getZonedDayRangeFromDayKey(dayKey, tz).start` and `dayEndExclusive = addDays(dayStart, 1)`.
  - A block contributes a segment to a day if it overlaps `[dayStart, dayEndExclusive)`.
  - For the segment, clamp to the view window:
    - `segmentStart = max(block.startTime, dayStart)`
    - `segmentEnd = min(block.endTime, dayEndExclusive)`
    - `startMinute = getMinuteOfDay(segmentStart, tz)`
    - `endMinute = getMinuteOfDay(segmentEnd, tz)`
    - compute `topOffset/height` from minutes (same formula as current day view)

#### Month agenda grouping

- Group blocks by overlapped dayKey (same overlap approach) and render as a list:
  - `Mon Jan 12` → badges with `h:mm a - h:mm a` + type + optional reason
  - Clicking a day group sets `dayKey` (and optionally switches view to `day`)

### 5) Week View UI (DnD Enabled)

- Add a view toggle (use existing shadcn `ToggleGroup` like `src/app/(public)/courts/[id]/schedule/page.tsx`).
- In the center panel, when `view === "week"`:
  - Render a header row with the 7 day labels (click to set `dayKeyParam`, optionally switch to Day)
  - Render the time grid:
    - left column: time labels
    - right: horizontally scrollable 7 columns
    - each day column renders:
      - a stack of droppable hour rows (same droppable data `{ kind: "timeline-cell", dayKey, startMinute }`)
      - an overlay layer that renders **interactive blocks** (draggable + resize handles) using the per-day segment model

Recommended refactor to restore resize:

- Replace `WeekTimelineCell` (indicator-only) with a `WeekDayColumn` component that:
  - renders `TimelineDropRow` for each hour row
  - overlays `TimelineBlockItem` for each `weekTimelineBlocksByDayKey.get(dayKey)` entry
  - (optional) overlays `DraftTimelineBlock` entries for import overlay
  - keeps droppables as the only droppable targets (blocks remain draggable only)

Resize support requirement:

- Week view must support dragging resize handles to change start/end (60m snap), not just creating new blocks.
- Ensure resize affordances are discoverable in Week view (always-visible handles or `@media (hover: none)` shows handles).

Remove/cancel support requirement:

- Week view must provide a clear way to remove a block (calls existing `courtBlock.cancel`).
- Recommended: add a small "Remove" / trash icon button on the block card itself (inside `TimelineBlockItem`) so it works in both Week + Day when Week uses `TimelineBlockItem` overlays.

Implementation details (to avoid DnD conflicts):

- Prefer a dedicated drag handle region and keep the remove button outside it.
- If the remove button is inside the same element that has `{...listeners}`, ensure it does not start a drag:
  - `onPointerDownCapture={(e) => { e.preventDefault(); e.stopPropagation(); }}`
  - `onClick={(e) => { e.stopPropagation(); onRemove(block.id); }}`
- Reuse existing `handleCancelBlock(block.id)` so behavior remains consistent (confirm + toast).

Keep the existing Day-side "Remove" action in the block list/inspector as a secondary path.

DnD considerations:

- Droppables already encode `{ dayKey, startMinute }` so week drops work naturally.
- Update draft-row drop handling:
  - Change `handleDraftRowDrop(rowId, dropDayKey, startMinute)`
  - In `onDragEnd`, call it with `overData.dayKey` (today it uses the selected `dayKey`, which will break once multiple days exist).

Reference (Context7): useDraggable/useDroppable `data` usage
- `/websites/dndkit` “useDraggable and useDroppable Interaction” example shows `active.data.current` + `over.data.current` type-checking.

### 6) Month View UI (DnD Disabled)

- When `view === "month"`:
  - Show a 42-cell month grid picker (6 rows × 7 columns) with padding days.
  - Highlight Today and the focused week (based on `dayKey`).
  - Click a day → sets `dayKey` and switches to `view=week`.
  - Disable dragging on presets/blocks/resizers (no droppables).

Optional enhancement (if desired): highlight days with blocks in the left `Calendar`.

Reference (Context7): react-day-picker modifiers
- `/gpbl/react-day-picker` “custom modifiers” example (`modifiers` + `modifiersClassNames`).

### 7) Court Hours + Disabled Slots (Week View)

- Week view likely uses a unified hour window (min startHour, max endHour across 7 days).
- For each day column, disable `TimelineDropRow` where `startMinute` is outside that day’s open windows (based on `useCourtHours(courtId)` data).
  - This prevents creating/moving blocks into clearly closed hours.
