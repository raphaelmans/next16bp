# Owner Onboarding Revamp - Changelog (2026-01-26)

Audience: Product + Marketing + Ops

This documents what changed in the platform (as implemented) for the owner onboarding funnel.

## What Changed (High Level)

We reworked the owner onboarding entry points so:

1) Marketing has a single canonical public landing page for owners.
2) Owners can sign up via an owner-specific register page.
3) After login, the app can route users server-side to the correct portal (Player vs Owner).
4) New owners have a single setup hub to create an organization, add/claim a venue, and start importing bookings.

## New / Updated URLs

### New

- Public marketing page: `/owners/get-started`
- Post-auth router (server-side): `/post-login`
- Owner setup hub: `/owner/get-started`
- Owner-specific register page: `/register/owner`

### Changed

- `/list-your-venue` is now a permanent redirect to `/owners/get-started`.

## New Owner Funnel (User Journey)

```text
/owners/get-started
  -> CTA: /register/owner?redirect=/owner/get-started
  -> /owner/get-started
       - Create organization (required)
       - Add new venue (optional)
       - Claim existing curated listing (optional)
       - Import bookings (optional)
       - Continue with verification and go-live steps
```

## Key UX Behaviors

### 1) Public owner landing page (marketing)

- `/owners/get-started` is conversion-focused:
  - explains "3 steps" to get bookable (create owner account, add/claim venue, submit verification)
  - includes FAQ and trust messaging ("you control when bookings go live")
  - primary CTA goes to owner registration

### 2) Owner-specific registration

- `/register/owner` uses owner-specific copy.
- It defaults to sending the user to `/owner/get-started` after registration.

### 3) Role selection on general signup (only when intent is unknown)

- `/register` shows a simple chooser:
  - "I want to book courts" (Player)
  - "I want to list my venue" (Owner)
- The chooser is skipped if the URL already implies owner intent (e.g. `redirect=/owner/...`).

### 4) Server-side post-login routing (no localStorage dependence)

When users log in without an explicit `redirect` parameter, the app routes them via `/post-login`.

Decision logic:

```text
If redirect param exists -> honor it
Else:
  if defaultPortal=owner and user has an organization -> /owner
  if defaultPortal=owner and no organization -> /owner/get-started
  else -> /home
```

This keeps booking deep-links safe while making owner routing reliable.

### 5) Owner setup hub

- `/owner/get-started` provides a guided checklist:
  - Create organization
  - Add venue
  - Claim existing listing
  - Import bookings
- Claim listing uses search over curated listings and submits a claim request (admin review still required).
- Import bookings links into the already-shipped import flow.

### 6) Add venue: special redirect when coming from setup hub

- If the owner creates a venue from the setup hub, success redirects to verification:
  - `/owner/places/new?from=setup` -> on success -> `/owner/verify/:placeId`
- If they create a venue from the standard owner dashboard flow, it keeps the existing behavior (goes to first-court creation).

## SEO / Marketing Notes

- `/owners/get-started` is the canonical, indexable page.
- `/list-your-venue` should not be linked internally and is removed from sitemap.
- Internal links previously pointing at `/list-your-venue` now point to `/owners/get-started`.

## Data / System Changes

### User preference: default portal

We added a persisted user preference so routing can work server-side.

- New table: `user_preferences`
- New enum: `default_portal` = `player | owner`
- Default is `player`
- When an owner creates their first organization, we set `defaultPortal=owner`.

## Limitations / Deferred Items

- Manual smoke testing is still required (product/QA).
- Ensure the `user_preferences` migration is applied in each environment.
- Bookings import: screenshots/images are still not supported for row extraction (ICS/CSV/XLSX supported).
