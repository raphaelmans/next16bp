1. Executive Summary

Project Overview

KudosCourts is a court booking platform that connects pickleball players with court owners through a peer-to-peer payment model. This document outlines 24 user stories covering the complete MVP journey from user onboarding through court booking and owner confirmation.

24
Total Stories
9
Domains

Key Features Covered

Feature Area Stories Status Priority
User Onboarding 7 Complete P0
Organization Registration 1 Complete P0
Court Creation 4 Complete P0
Owner Dashboard 1 Complete P0
Availability Management 2 Needs Implementation P1
Court Reservation 2 Needs Implementation P1
Owner Confirmation 2 Needs Implementation P1
P2P Reservation (Future) 1 Deferred P2

Key Architectural Decisions

  • Simplified MVP Flow: Domains 05-07 implement basic end-to-end booking without TTL timers or payment proof uploads, allowing faster shipping while deferring complexity to Domain 08.
  • Backend-First Approach: Backend implementation for most features is complete. Stories focus on wiring frontend components to existing endpoints.
  • Supersession over Deletion: Original domain 03 stories are marked superseded but retained for historical reference and context.
  • Domain Split: Original US-03-03 (owner confirms payment) was split into US-07-01 (view) and US-07-02 (action) for clearer separation of concerns.

US-00-01 Active
User Authentication Flow
As a user, I want to sign up, sign in, and sign out so that I can access platform features like reservations and profile management.

Acceptance Criteria

Sign Up (Email/Password)
Given I am on /register
When I submit a valid email and password (min 8 characters)
Then my account is created and I see "Check your email for confirmation"

5. Domain 02: Court Creation

Domain Overview

The Court Creation feature domain enables two types of court listings: Curated Courts (created by admins for discovery) and Owner Courts (created by owners for reservations).

Total Stories: 4 Active
Status: Complete
PRD Reference: Section 10, Journeys 3-4

US-02-01 Active
Admin Creates Curated Court
As an admin, I want to create a curated court listing so that players can discover courts before owners onboard the platform.

Key Acceptance Criteria

  • Access form from /admin/courts via "Add Court" button
  • Fill required fields (name, address, city) to create court
  • Court created with type: CURATED, claimStatus: UNCLAIMED
  • Add photos, amenities, and contact socials (Facebook, Viber, Instagram, website)
  • Court appears in discovery with "Contact to Book" label (not reservable)

Form Fields

Required: Name, Address, City

Optional: Description, Photos, Amenities, Facebook URL, Viber, Instagram, Website

US-02-02 Active
Owner Creates Court
As an organization owner, I want to create my own court so that players can discover and book it through the platform.

Key Acceptance Criteria

  • Access form from /owner/courts via "Add Court" button
  • Fill required fields and submit to create court
  • Court created with type: RESERVABLE, linked to organization
  • Configure default pricing (hourly rate and currency)
  • Redirected to /owner/courts/[id]/slots after creation to manage availability
  • Court appears in discovery (shows "No availability" until slots are added)
  • If no organization exists, redirect to /owner/onboarding

Post-Creation Flow

/owner/courts/new → [Create Court] → /owner/courts/[id]/slots → Add time slots
US-02-03 & US-02-04 Active
Additional Court Creation Stories
  • US-02-03: Admin Data Entry Form (enhanced form for bulk court data)

US-05-02 Active
Owner Views and Manages Slots
As an organization owner, I want to view and manage all time slots so that I can block, unblock, or delete slots as needed.

Key Features

  • View all slots across all statuses (AVAILABLE, HELD, BOOKED, BLOCKED)
  • See player information on booked slots (name, phone from reservation snapshot)
  • Block/unblock slots for maintenance or private use
  • Delete available slots
  • Filter slots by date range and status

Slot Status Reference

Status Description Owner Actions
AVAILABLE Open for booking Block, Delete
HELD Temporarily locked (paid reservation pending) View player info
BOOKED Confirmed reservation View player info
BLOCKED Manually blocked by owner Unblock

NEW Backend Requirement: timeSlot.getForCourt

The current timeSlot.getAvailable only returns AVAILABLE slots for player discovery. Owners need to see ALL slots regardless of status.

Input: { courtId: string, startDate: string, endDate: string }
Output: TimeSlotWithPlayerInfo[] - includes player name and phone for HELD/BOOKED slots

8. Domain 06: Court Reservation (Simplified)

Domain Overview

The Court Reservation domain enables players to discover and book pickleball courts. This is the core player-facing booking flow that connects court discovery to confirmed reservations.

Total Stories: 2 Active
Status: Needs Implementation
PRD Reference: Section 7 (Journeys 2-3), Section 8
Supersedes: Domain 03 (US-03-01, US-03-02)

Simplified Scope

This version defers TTL timers, payment proof uploads, and expiration handling to Domain 08 (P2P Reservation Confirmation). Focus is on essential end-to-end booking.

Deferred to Domain 08: 15-minute TTL countdown, payment proof upload (reference number, notes, image), payment instructions display, expiration handling (cron job), T&C acknowledgement UI

Confirm Booking

When I click "Reserve" or "Confirm Booking"
Then a reservation is created via reservation.create({ timeSlotId })
And reservation status is immediately CONFIRMED

10. Domain 08: P2P Reservation Confirmation (Future)

Domain Overview

Domain 08 represents the full-featured P2P reservation confirmation flow with TTL timers, payment proof uploads, and expiration handling. This domain is deferred to post-MVP.

Total Stories: 1 Overview
Status: Deferred
PRD Reference: Section 8 (Complete P2P Flow)

Deferred Items

The following features are deferred from Domains 06 and 07 to this future domain:

  • 15-minute TTL countdown: Requires cron job or edge function
  • Payment proof upload: Reference number, notes, image upload (S3 integration)
  • Payment instructions display: Needs owner payment info setup
  • Expiration handling: Auto-release slots after timer expiration
  • P2P verification flow: Full feature with proof review
  • Email/SMS notifications: Integration complexity

Rationale for Deferral

By deferring these features, the MVP can ship with basic end-to-end booking functionality faster. The simplified flow (Domains 06-07) provides the core value proposition while allowing time to properly architect timer infrastructure and payment proof handling.

Key Flows Summary

Technical Architecture Notes

  • Framework: Next.js 16 with App Router
  • API Layer: tRPC for type-safe API calls
  • Database: PostgreSQL with Drizzle ORM
  • Authentication: Supabase Auth
  • State Management: React Query for server state, Zustand for client state
  • Styling: Tailwind CSS with custom design system
